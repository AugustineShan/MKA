# 核心纪律 - Skill v1

这份文件是核心假设体系的横切纪律单一真源。`/brkd`、`/load`、`/ka`、`/adj`、`/annual-update` 都必须先加载本文件，再加载自己的 operation skill。

本文件是 library/include，不是可独立调用的 operation skill。不要把它当成 `/核心纪律` 命令执行。

它只管“怎么做才不丢、不漂、不偷算”。它不定义输出块格式；输出语法见 `skills/核心假设源语言_skill_v1.md`。

## A0. 操作分流

当分析师只说“公司有新东西/要更新一下”而没有明确 slash 命令时，先分流再执行：

- 新年报、最新实际年度覆盖旧稿历史末年 -> `/annual-update`。
- 一条或少量边际信息、正式稿上局部修判断 -> `/adj incremental`。
- 已有 knob 的纯数值小改 -> `/adj quick` 或 `/frontend-edit`。
- 新 BRKD/LOAD 产物已成型，想重做正式核心假设 -> `/ka`。
- 只有原始财务数据刷新 -> `/init`。
- 只有核心假设源文需要变成 yaml1 -> `/comp`。
- 重资产公司的固定资产/产能/capex 排程要建模 -> `/da`(轻资产或稳态公司别用)。

## A1. 历史基年神圣 + 历史保全

- 历史基年原子神圣不可改。任何情况下都不允许为了配平、好看、接近 headline 而回改历史原子。
- 业务相关历史必须有归宿：正文、收纳区、缺口待补、明确丢弃原因。
- 历史只存骨架声明需要的绝对值原子 + 上挂 headline。
- yoy、毛利率、占比、加总比例等可推导量不作为历史原子存储；它们可以作观察说明。
- 子线合计与 headline 不一致时，显式写差额、口径、来源层级；禁止用残差倒算吃掉差额。

## A2. 接缝铁律

信息在本层要么活，要么死，不能半死不活。

每条有价值的信息必须落到四类之一：

1. 正文假设。
2. 收纳区。
3. 缺口/待补。
4. 明确丢弃原因。

禁止：

- 静默占位。
- 只留下“见 R136-R208”“见某页/某单元格”的死引用。
- 明明捕获不了却装作已经读过。
- 因为下游暂时不算，就把有价值的业务信息丢掉。

若出现来源冲突或候选冲突，必须按核心假设源语言 B7 写“来源与裁决”：候选A、候选B、采用、为什么、未采用方去处。裁掉的一方也必须有归宿；不能因为已经决定不用，就让它静默消失。

## A3. 对账不是算账

唯一合法算术是保真对账：检查自己有没有抄错、量级是否合理、口径差是否需要举旗。

合法输出只有：

- `ok`
- 显式差额 + 口径说明
- 缺口/待查证

禁止：

- 存派生预测序列。
- 残差倒算旋钮。
- 把净利润、营业利润、利润总额当旋钮。
- 手工展开三表或 DCF。
- 为了通过 sanity check 而改历史原子或预测旋钮。

## A4. 押不等于落盘 + 分段停止点

先在对话里押判断，再等分析师拍板，拍板后才落盘。

交互风格是纪律的一部分：押判断要像分析师开会，不像机器审表。默认用“会议 memo”输出自己的理解和预测，再等确认：

- 先讲结论，再讲证据：说明“我读下来这块在押什么 / 我建议怎么装 / 为什么”。
- 控制在一个可读屏幕左右：3-5 条结论、一个紧凑表格、一个风险/待拍板清单。
- 不在确认阶段倾倒完整 markdown、所有历史原子、所有 source range、逐条 JSON/YAML/knobs；这些必须完整落盘，但默认不全量贴进聊天。
- 每次只问一个自然问题：`这段我这样理解/这样装可以吗？确认后我写入底稿，再进下一段。`
- 用户要求看完整底稿、source range 或 knobs 时，可以展开；否则聊天为人读，文件为机器读。

必须按语义区块停，不允许连写到底：

1. overview/时间轴/骨架确认。
2. 收入。
3. 毛利/成本。
4. 费用。
5. below-OP 与税。
6. 中期/terminal。
7. 收口核对。

三种违规态：

- 连写多个未确认区块。
- 先落盘再等拍板。
- 押完一个区块直接进入下一区块。

例外：`/frontend-edit` 和 `/adj quick` 的定点试算回写，用户已在前端或 patch plan 中拍板；它们可以不重跑 `/comp`，但只能做确定性的 scoped yaml1 patch。

这个例外成立的理由：`/comp` 是 LLM 翻译器，不是确定性编译器。对已有 knob 的纯数值微调，重跑整篇 `/comp` 可能改动无关 yaml1；定点 patch 反而更保真。

direct yaml1 patch 的边界必须闭合：

- 白名单不可加宽：只允许已有 knob 的 `values[i]` 或 `terminal.perpetual_growth` 标量；不得改结构、slug、family、horizon、fade、`terminal.fade.target_growth`、历史、来源、stash、新增/删除 path。
- 三处同源：核心假设正文、`knobs`、yaml1 必须一致。
- tie-break 写死：`核心假设.md` 是 canonical，yaml1 是派生缓存；三处不一致时停止，md 赢，回到 `/comp` 重编译。禁止为了“对齐”再手 patch yaml1 去凑 md。
- 若 compiler/yaml1 schema 改动导致 path 定位失效，停止并升级映射或走 `/comp`，不得猜路径。

## A5. 参数化先于数值 + 自洽不过定 + 毛利耦合门

- 先定 compiler family / 参数化族，再押逐年值。
- 一个指标要么是旋钮，要么是派生，不能两头都是。
- 毛利必须先过耦合门：分线派生 vs 整体手拍，要在写收入段前定。
- 若收入 leaf 已挂成本/毛利旋钮，不得同时再写顶层整体 `income.gpm`，除非明确说明覆盖关系。
- 自洽只用来防止互相打架，不等于自动拍板；即使逻辑自洽，也必须等分析师确认。

## A6. 旋钮值精确完整逐年可机读 + knobs 同源回声

- 预测旋钮必须精确、完整、逐年、可机读。
- 禁止写“约”“大概”“往后平”“逐年递减”但不给数组。
- flat 也要写明年份和值。
- 永续增长给一个点值，不给区间。
- 衰减给可解析规则，或明确写“交引擎默认”。
- `knobs` 块是正文旋钮的同源回声，不是 YAML1，不写 yaml path，不做会计路径映射。
- 正文改了，`knobs` 同步改；`knobs` 值必须和正文一字不差。
- 具体语法以 `docs/knobs块契约.md` 为准；其他文档只引用，不另起规则。

## A7. 派生量预测序列不手算，交引擎

- 收入、成本、毛利率、占比、净利率等未来派生量，要么不写，要么标注“派生·引擎算·非翻译输入”。
- 本层零模型计算，不展开未来派生序列。
- 历史实际比率可以作为事实搬运或观察，但要标明它来自实际数据，不是预测判断。

## Operation 引用方式

- `/brkd`：完整继承 A1/A2/A3/A7；A5/A6 只到草稿态；A4 是弱形态，产物整体保持 draft。
- `/load`：完整继承 A1-A7；额外遵守 load-vintage 沙箱纪律。
- `/ka`：完整继承 A1-A7；额外承担冲突裁决和防静默 passthrough。
- `/adj incremental` 或未来 `/modify`：按补丁尺度继承 A1-A7；只动受影响行。
- `/adj quick`：继承 A1/A3/A6/A7 的定点回写形态；不做研究判断。
- `/annual-update`：完整继承 A1-A7；额外遵守旧稿只读、声明式估算、双向核和不拉数据。
