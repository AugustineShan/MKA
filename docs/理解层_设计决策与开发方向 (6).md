## 0. 定位

理解层负责整条 **「一堆投研材料 → 一份能喂给计算引擎的 yaml1」** 的链路。

最终愿景:这一套长成**买方工作台**。市面上 Wind AI / Bloomberg AI / Claude Finance 全是卖方工作台——产出卖方那套标准化、可对外的东西;买方要的是「陪分析师理解每家公司、沉淀私有判断」的东西。这是产品的根本差异点,所有设计都服从它。

## 2. 全局铁律(写进 CLAUDE.md,所有 skill 共享)

1. **LLM 当翻译,Python 当执行器。** 全局工作约束,不可破。
2. **一切可扩展,绝不 hardcode 科目。** 任何科目(成本、销售费用…)未来都可能拆细,设计时就要留口子。
3. **科目对齐查 `数据格式参考.md`(字典/全集),覆盖路径与默认值用 `defaults.yaml`(目标)。** 这两个是不同的东西,曾混为一谈是致命 bug——用 defaults.yaml 的 source 公式做语义匹配是绕的;字典本来就是为「字段名↔中文科目」生成的,直接权威。
4. **能力边界 = 这家公司的真实文件边界,不是任何内置清单。** 保险公司有「已赚保费」,它的 defaults.yaml 里就有 `prem_earned`,现场查就认得到。换结构不同的公司照固定表查字典必崩,按文件现查才泛化。
5. **财务费用不在 yaml1。** 它由 balancing engine 根据现金/负债余额倒算。券商模型里财务费用/营业利润/净利润显示 `#VALUE!` 是**设计如此**,不是模型坏了;归母/净利率往下断链是它的下游传导,同样情理之中。
6. **reconciler 用 GLM/Kimi 补数,是事实层唯一的判断注入口。** 因此审计必须严——是 Agent 审计(拿 skill + Python 执行),不是纯脚本跑。审计链严不严,决定 facts 层干不干净。
7. **引擎是公式求值器,不是封闭的"族目录";简单族是默认、复杂算法走开放公式层(默认关、按需开)。** 量×价这些族不是特权内置,只是最常见的声明;装不下的线(如茅台收入←产能←基酒滞后链)用自然语言把算法讲清→翻成可执行公式→引擎跑→回测复现历史才算数。**绝不预先列死"允许哪些运算"(那是变相 hardcode)。** 详见 6.5 + 第 9 节开发计划。
8. **"绝不丢信息"是逐层铁律,不止 compiler 那一层。** 每道接缝(源材料→.md、.md→yaml1)都一样:**忠实搬运;捕获不了就举旗(标待办);绝不静默降级成"看着像捕获了"的占位**(如收纳区只留单元格引用)。上游丢了的,下游再诚实也补不回。(新乳"子公司副拆分被挑着誊"就是漏在 .md 这层;已在生成器 v10 焊死。)
9. **修复在源头焊死,别让下游每家公司都踩。** 符号、`unit_factor`、forex 路径这类系统性问题,都在它该被解决的那一层一次性解决,不靠每条线/每家公司手工绕。
10. **符号靠利润表配平等式动态验证,不假设、不手洗。** 减值类符号曾横跨 clean_annual/yaml2/calc.py 静默注水(减值反增厚利润);根治:减值从 `total_cogs` 剥离、按真符号并入营业利润,clean.py 的 IS 校验从"假设符号"升级成"验证符号"。TuShare 符号 2019 报表格式变更后逐年翻 → 定**预测期靠 yaml1 现代口径、历史期靠动态验证**。
11. pip 装包一律走清华镜像 `-i https://pypi.tuna.tsinghua.edu.cn/simple`。

## 3. 核心假设生成器(规格)

> 完整规格见 `核心假设生成修改器_skill_v10.md`。产出文件名沿用 `核心假设.md`(名字只指向"预测"那层,实际内容是一个完整盈利模型;别被名字误导成"只记假设")。

**产出永远是 markdown 核心假设.md,不是 yaml。** 翻译成 yaml1 是下游 compiler 的活;这个 skill 绝不输出 yaml、不"导出 driver 结构为 yaml"。(教训:v5 把"第一读者是 yaml 翻译器"放顶部强调,反而诱发 agent 产 yaml 的幻觉、还退回"转写器"身份;v6 改成**先立"只产 markdown"防火墙,再讲为什么要精确**。)

**本质——它是 DCF「判断覆盖层」的人类可读、带出处的源头。** 拿到 skill 的 LLM 不需要知道 yaml1 的存在,只看一堆投研材料,产出一份语义化的盈利模型底稿。yaml2 是机器自动铺的底,这份底稿(经 compiler 变成 yaml1)是人压在上面的判断。它捕获 clean_annual 里没有、机器铺不出来的东西:**非标 + 判断 + 出处。**

**产出 = 一个盈利模型的三层,捕获方式不同(认 / 搬 / 议):**
- **骨架——认。** 怎么拆 + 每条线的计算规则(旋钮/派生/公式族/上挂科目)。会计常识,读模型时认出来。
- **历史——搬。只存绝对值原子 + 收入 headline,砍掉 yoy/毛利率/占比(可推导的不存,见 v7)。** 存什么由这条线的骨架决定(量×价线 = 收入/销量/吨价/吨成本;增速族 = 收入/成本;门店模型 = 门店数/单店收入……),**绝不 hardcode 行清单**。收入照抄(它是 headline 锚 + 加总校验 + "销量×吨价≈收入"的探针)。**非标拆分的历史(那套量价、副拆分)clean_annual 里根本没有、丢了永远找不回——而且它正是非标分部喂给引擎的 carry-forward 基数(见第 6 节),所以必须忠实接住。**
- **预测——议。** 把旋钮拨到哪 + 三件套(谁定/为什么/来源)。开会形成:汇报理解 → 一条条过 → 先押再问 → 老板拍板。

口诀:**骨架认清、历史照搬、预测开会。** 别只做"开会"那层、把骨架历史当填表扔了——结构是图纸、历史是地基、判断是楼,三个都要。

**计算骨架——记规则,不只记数:**
- 每条线选一个**参数化族**:量×价 / 量×价×毛利率 / 增速 / 绝对值 / 费用率。定下哪些指标是**旋钮**(预测直接拨)、其余怎么**派生**。
- **自洽铁律:一个指标要么旋钮要么派生,不能既当旋钮又当派生**(设了吨成本就不能再设毛利率,俩通过吨价锁死)。落字前逐条线查不过定。
- **买方自由:参数化是老板选的,可和券商原模型不一样。** 老板想直接拨毛利率而不是吨成本,就走 (量,毛利率) 族——这正是"把判断从券商牢笼解放出来"。记清用哪套,不同则 ⚠️。
- **拆分可嵌套,封顶两级**(收入→产品线→产品号),**只拆到老板拨旋钮的那一级**(树的深度=判断的深度,不是数据的深度);**超过两级就举手,按种类分流**——纯加总→归并到产品号 + 收纳区;藏着真关系(mix 规则/共享/滞后)→ 那是算法穿了拆分的外衣 → 转算法提升(v9)。基年原子存在驱动的那一级。

**预测怎么写(v5 收紧,因为下游 compiler 要照翻这份 markdown):**
- **只写旋钮,且旋钮精确、完整、逐年、可机读。** flat 也写明值+年份(`0.5%(2025-2031 全程)`),逐年变化逐年给值;**禁止 ~/约/平推/往后平/递减(不带值)**。永续给一个点不给区间,衰减给可解析规则或写"交引擎默认"。标杆:`有效税率 全程 14.61%(2025-2030)/2031 起 16%`。
- **派生量(收入/毛利率/占比…)的预测序列不手算、不填,交引擎。** 手算既越界(算账是引擎的活)又易引入没人发现的算术错。要给就明标"派生·引擎算·非翻译输入"。**历史同理:派生比率(yoy/毛利率/占比)也不存,只存绝对值原子 + 收入**(v7)。

**职责边界(它不干什么):**
- **会计化对齐**(段收入正式对到标准科目)→ compiler 的活。底稿只记「上挂哪个科目」的意图。
- **算账**(收入=量×价、财务费用倒算、三表配平)→ 引擎的活。底稿只点名族和旋钮。
- **平推**(没碰的行)→ yaml2 的活。底稿标一句「沿用默认」+历史即可。

这条边界决定它**重在非标 + 判断 + 出处,轻在标准平推**(机器替你铺)。

**通用 IS 核心骨架(写进契约作默认基准,可扩展):**
营业收入 / 营业成本 / 税金及附加 / 销售 / 管理 / 研发费用 / 财务费用(跳过) / below-OP 调节项 / 所得税。逐行点名的锚 = **「全表分类规则 ∩ 本公司 clean_annual 实际非空字段」**,不拿全行业字段表去套(否则给消费股冒出一堆「已赚保费:无」的废话)。

**中期 · 三段式:** 显式期(老板真正有看法的 2-3 年)→ 衰减期(fade 到永续)→ 永续。远端不能默默平推、也不能默默交给系统,先押再问、老板明确点头。

**收纳区——收就收数(全序列,不是单年比例)。** 副拆分、算了不用的口径、定性情报、口径说明,连多年数据一起收。只记「西南 35.9%(2024)」等于没收。

**不写会议状态 ✅ 清单**(过程痕迹);该盯的默认假设用 ⚠️ 就地标在线块上。每条预测带三件套;历史不需逐条三件套(块头标「照搬自 XX」)。

**I/O:**
- input:**init**(从零)/ **modify**(改已有底稿,按边际信息改)。
- output:聊透数齐 → `核心假设.md`;有分歧/数据不齐 → `核心假设参考.md`(**骨架和历史照样落全**,只是预测带待办)。
- **素材门槛:** init 至少「最近年报 + 一份非标材料」(券商模型是其中一种,非唯一);modify 至少「之前的核心假设.md」。料不够有权说"做不了"。

## 5. compiler(规格)✅ 已定稿 = `yaml1compiler.md`

> **状态:已定稿并实战跑通新乳。** 第一原则"能翻就翻、翻不了举旗、绝不猜";产出 **A 类稀疏旋钮 + B 类完整 stash**;四种 `kind`(见 6.5);结构化 `unit_factor`(量价 100 / 增速 1,按族给);fade 的 hold/fade 分组焊进去;第三层分流镜像(不判断,只归并/转算法)。

**三份输入,各司其职:**
- `核心假设.md` —— 要翻译的语义判断。
- `数据格式参考.md` —— **字典**,查字段名↔中文科目的语义匹配(对齐用这个)。
- `defaults.yaml` —— **覆盖目标**,告诉 compiler 这家公司有哪些路径、默认值、放在哪个层级。

**职责:** 读核心假设.md 一条判断 → 去字典语义匹配认领科目名 → 求值出数组 → 覆盖 defaults.yaml 对应路径。**常见 IS 科目清单**写进 skill 作快速参照(降低每次摸索成本),但明确是「常用页」不是封闭边界,清单外回字典(全集)查。

**unaligned 的正确含义:** 把整份 defaults.yaml/字典扫一遍,确实找不到任何语义对得上的路径(比如分析师对一个根本没参数化的东西有判断),才标 unaligned。**绝不许因「没背过」就标。**

**compiler 的两条岔路(逐线判断,默认走简单):**
- 这条线核心假设.md 声明的是**已知族**(量×价/增速/费用率/绝对值)→ **简单模式**:对齐标准路径 → 求值出旋钮数组 → 覆盖 defaults.yaml。(现有行为)
- 这条线声明的是**自定义算法**(骨架里写的是公式、不是族名)→ **进入算法模式**(详见 6.5、第 9 节)。
- 这条线是**嵌套拆分**(收入→产品线→产品号)→ **递归对齐**:mint `…segments.<产品线>.segments.<产品号>`,逐级 rollup=sum;**带深度守卫 ≤2**,撞到第三层按种类分流(纯加总自动归并到产品号 + 报一句;真关系转算法模式)。
- ⚠️ **compiler 不自己决定一条线复不复杂——核心假设.md 已经由 skill + 人共同定了**(skill 默认关、按需"举手共探"才开)。compiler 只 honor 那个声明:见族名走简单、见公式进算法模式。
- **但 compiler 必带回测:** 不管哪条路,翻完都在历史上跑一遍,复现 source 已知值才算翻对;不符就报错/标记,**绝不静默发一个翻错的东西**。回测不过 = .md 没说清 或 compiler 翻歪 → 回人。

## 6. yaml1 / yaml2 怎么配合(引擎的双层参数体系)

> **一条架构事实,解释下面一切:** `yaml1` 是**理解层的 raw_tushare**——结构化了但极非标,背着这家公司全部个性(几条线、什么族、哪些旋钮、再参数化、嵌套拆分、将来的算法节点),**忠实、绝不丢信息,但引擎不能直接吃**。把它清洗成"标准科目逐年参数"的是 `清洗yaml1.py`(**理解层的 clean.py**:折叠 decomposition→收入、展开 flat/fade→逐年、resolve⊕yaml2、回测硬闸)。所以 **`calc.py`(纯 DCF 算账核)永远看不到 yaml1**——它只吃清洗后的标准参数。你写的 decomposition 子树、formula 节点、stash,calc.py 一概不认:它们是给 `清洗yaml1.py` 和前端的。

整个估值的参数来自**两层叠加**。看懂这两层怎么配合,就看懂了为什么核心假设.md 要那样设计(为什么只存原子、为什么旋钮要逐年写满、为什么非标结构必须保留)。

### 6.1 两层各是什么

**yaml2 = 自动铺的底(完整但"笨")。**
- 谁生成:`defaults_gen.py` 读 clean_annual(清洗后的历史),给这家公司三张表每一个标准科目都铺一个 **carry-forward 平推**值——默认"未来沿历史规律走"。
- 特点:**完整**——每条标准线都有值,引擎永远不缺料、不会留洞;但**笨**——它不懂这家公司的故事,只会平推。
- ⚠️ **它就是各公司目录里的 `defaults.yaml`**(由 defaults_gen 生成)。"yaml2"是它在引擎里的角色名(平推 base);"defaults.yaml"是它作为文件、作为 compiler 覆盖目标的名字——**同一个东西的两个视角**(类似第 5 节"字典 vs 目标"那种同物多名,记清免得混)。

**yaml1 = 人压上去的判断(稀疏但"聪明")。**
- 谁生成:compiler 读 核心假设.md 翻出来。
- 特点:**稀疏**——只覆盖老板真正动过的线(拆开的非标分部 + 设了旋钮的科目),没碰的一概不写;但**聪明**——带着非标拆分结构和逐年旋钮,是这家公司的"看法"。

一句话:**yaml2 是"什么都不变会怎样"的机器基线,yaml1 是"我的看法",DCF = 基线 + 看法。** 这正是买方工作台的内核——也是为什么市面那些卖方工具给不了:它们没有"我的看法"这一层。

### 6.2 怎么叠(resolver 逐路径覆盖)

resolver 合并两层,规则是**逐参数路径覆盖:yaml1 有值就用 yaml1,没有就落回 yaml2 平推。**

- **标量 vs 数组。** yaml2 的平推常是**单值**(如 `销售费用率: 0.156`,引擎广播到预测期每一年);yaml1 覆盖时常是**数组**(逐年旋钮,如 `[0.154,0.154,…]`)。引擎见单值就全程广播,见数组就逐年用。**这就是为什么 yaml1 统一用数组、旋钮必须逐年写满**(第 3 节)——它要逐年顶掉 yaml2 的广播值,缺一年就顶不严。
- **结构替换(非标的关键)。** yaml2 只有标准科目——它只知道一条"营业收入平推 yoy",**没有那套四线量×价拆分**。yaml1 不是改一个数,而是**用一整块拆分结构顶替**那条平推线:四条分部各带销量/吨价/吨成本旋钮,引擎逐线 量×价 算出收入再加总,覆盖掉 yaml2 那条笨的"营业收入平推"。所以 yaml1 的 `income.revenue` 结构比 yaml2 丰富得多(第 3 节"非标必须保留"就是为这个)。
- **没碰的线静默落底。** 老板没表态的(某个小的 BS 周转、某项杂费),yaml1 不写,自动落回 yaml2 平推。**这就是为什么核心假设.md 不用铺满每一行**——平推那半 yaml2 全包了,底稿只管"我有看法"的那部分。

### 6.3 非标分部的"基数"从哪来——这正是 v7 历史只存原子的原因

引擎要把"低温鲜奶"往前推,需要两样:**起点(基年实际值)** + **旋钮(逐年 yoy)**。
- 旋钮:核心假设.md 预测段给(销量 yoy 等)。
- 基年:鲜奶 2024 的 销量=19.82、吨价=13820——这套**非标分部的历史,clean_annual 里没有**(它只有标准营业收入总额,没有四线量价拆)。所以这套分部的 carry-forward 基数,**只能来自核心假设.md 的历史原子**。

这就把 v7 那个决定接上了:核心假设.md 历史**只存绝对值原子 + 收入**,不是为了好看——**那些原子就是非标分部喂给引擎的 carry-forward 基数**;而 yoy/毛利率/占比不存,是因为引擎自己从原子算(派生)。归纳成两句:
- **标准线的基数** → yaml2 从 clean_annual 平推铺好。
- **非标分部的基数 + 结构** → 核心假设.md 历史原子 → 经 compiler 进 yaml1。

两层的"历史地基"来源不同(一个来自 clean_annual,一个来自核心假设.md),合起来才完整。这也解释了职责分工:clean_annual 管得了标准总额、管不了非标拆分,后者天然是核心假设.md 的活。

### 6.4 合起来的流(以新乳为例)

1. `defaults_gen` 读 clean_annual → **yaml2**:每条标准线一个平推值(营业收入平推、销售费用率=2024 值、税率平推…),完整铺满。
2. `compiler` 读 核心假设.md → **yaml1**:营业收入那条被四线量×价结构顶替(各线旋钮数组 + 基年原子);销售费用率覆盖成 `0.154`;资产处置覆盖成 `[-70,-30,…]`;税率覆盖成 `0.1461`;**营业外支出覆盖成 `8.0`(顶掉模型平推的 47.4——那是助理查年报附注后的主动修正)**……老板没碰的线留空。
3. **`清洗yaml1.py`**(理解层的 clean.py):把四线量×价的 decomposition **折叠成营业收入/收入 yoy**、flat/fade **展开成逐年**、resolve ⊕ yaml2(yaml1 赢的地方用 yaml1、其余落 yaml2)、跑**历史回测硬闸** → 输出引擎能吃的逐年标准参数表。
4. **`calc.py`(纯算账核,看不到 yaml1)**:吃清洗后的标准参数 → 逐年算收入/成本/毛利率(派生)、三表配平、财务费用按现金负债余额倒算 → 三表 → DCF。

> 注意:**折叠非标 = `清洗yaml1.py` 的活,不是 calc.py 的。** calc.py 拿到的是折叠后的标准 `收入 yoy`,根本不认 decomposition 子树——这就是"calc.py 永远看不到 yaml1"。也注意**没有一处是核心假设.md 手算的**:收入、毛利率、财务费用、净利全是机器从旋钮 + 基数算的;核心假设.md 只给了旋钮、基年原子、结构、和那条 47.4→8.0 的判断修正。这就是"LLM 翻译 / Python 执行"在这条链上的完整落地。

### 6.5 复杂算法:yaml1 的"公式节点"(开放公式层,默认关)

6.1–6.4 讲的是简单族:yaml1 用**数组**覆盖 yaml2 的平推标量。但有的线(茅台收入←产能←基酒滞后链)没有哪个族装得下。这时 **yaml1 要能装的不只是数组,而是一个"公式节点"**——这就是 yaml1 在算法模式下要新具备的能力。

**yaml1 路径的值,有四种形态(带 `kind` 标签分派):**
- **数组(`knob`,默认):** `path: [v1,v2,…]` ——旋钮,清洗后引擎按已知族算。
- **拆分子树(`decomposition`):** 非标拆分,顶替 yaml2 那条平推;`{rollup: sum, segments: {…}}`,每个 segment 要么是 leaf(family+base+knobs)、**要么自己又是一个 decomposition(嵌套第二级)**。**递归但带深度守卫 ≤2**(收入→产品线→产品号);基年原子存在驱动的那一级。**由 `清洗yaml1.py` 折叠成收入,calc.py 看不到它。**
- **公式节点(`formula`,算法模式,按需):** 一个结构,装五样——
  1. **可执行的公式表达式**(NL 翻成的、引擎能求值的式子);
  2. **输入声明**:哪些是旋钮数组、哪些是对**别的线/路径的引用**、哪些是对**自己前几年的滞后引用**;
  3. **历史种子**:滞后/移动平均要的前 N 年原子(从核心假设.md 历史原子来);
  4. **回测参照**:source 的历史已知值 + 容差;
  5. **上挂**的标准科目。
- **收纳区(`stash`,B 类):** 不进 DCF,**但必须进 yaml1 且落全**——它是 yaml1"绝不丢信息"灵魂的一部分,是**一等公民**,不是可省的注释(曾误把"不进 DCF"当成"不进 yaml1",被纠正)。前端读它;calc.py 不碰。对应生成器那条"收纳区一视同仁誊全"。

> A 类(knob/decomposition/formula:进 DCF 的驱动)走稀疏覆盖;B 类(stash:不进 DCF 的参考)走完整保全。两类都在 yaml1 里,但去向不同。

**公式节点之间组成一张依赖图(DAG)。** 产量节点引用产能节点、产能节点引用基酒节点……引擎按**拓扑序**、且对滞后按**年份顺序**求值。这正是为什么核心假设.md 必须存历史原子、且滞后规则要存前 N 年——**它们就是公式节点的种子和回测参照**(6.3 那条"非标基数来自核心假设.md"在算法模式下更刚需)。

**叠加在 `清洗yaml1.py` 里完成(不是某个泛化的 resolver/引擎):** 它 fold/expand/resolve——decomposition/formula 顶替 yaml2 那条平推线,没被碰的线照旧落 yaml2 平推。两层叠加的机制不变,只是 yaml1 这层的"积木"从纯数组扩展到 **`knob | decomposition | formula | stash` 四种,按 `kind` 分派**;calc.py 只在最后吃折叠展开后的标准参数。
