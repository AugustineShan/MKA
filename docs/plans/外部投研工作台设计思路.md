# 外部投研工作台设计思路

> 讨论稿。本文侧重需求、分层和工作流，不讨论技术实现。

## 1. 为什么需要外部投研工作台

MKA 的核心能力是把正式假设翻译成可运行模型。它适合处理：

- 官方历史事实。
- 分析师已经确认的判断。
- 核心假设的裁决。
- 从核心假设到 yaml1/forecast 的翻译和审计。

但真实投研工作远早于建模。大量信息来自研报、纪要、公告、电话会、行业段子、专家交流、竞品动态和产业链变化。这些信息不一定马上进入模型，但它们会改变我们对行业、公司和业务驱动的理解。

如果这些信息直接进入 `/ka`，KA 会变得过重：既要读材料、理解行业、认识公司、整理历史，还要裁决未来假设。结果就是每次建模都像重新认识公司。

外部投研工作台的目标是把“认识世界”这件事从 MKA 前面拆出来：

```text
外部投研工作台：吸收行业和公司信息
-> 业务与业绩底稿：形成公司人话理解中枢
-> /ka：裁决正式模型假设
-> /comp：翻译运行
```

## 2. 一句话定位

外部投研工作台是 MKA 的上游投研信息吸收系统。

它不直接生成模型，不直接写 official 核心假设，不直接跑 DCF。它负责把各种非标信息整理成长期可复用的行业判断、主题判断、公司理解和边际变化记录，再汇总成“业务与业绩底稿”供 MKA 使用。

一句话：

> 工作台看世界，业务底稿讲清公司，KA 裁决未来，COMP 翻译运行。

## 3. 三层式工作台

外部工作台建议分为三层：行业、主题、公司。

### 3.1 行业层

行业层回答“这个行业怎么运转”。

它关注：

- 行业空间和周期位置。
- 供需关系。
- 价格机制。
- 成本曲线。
- 竞争格局。
- 渠道结构。
- 政策和监管。
- 核心跟踪指标。
- 行业内不同商业模式的差异。

行业层的产物不是公司模型，而是行业判断底稿。

例子：

```text
MLCC 行业研究
- 行业需求来自消费电子、汽车电子、工业控制等。
- 价格受库存周期、产能利用率、产品结构影响。
- 高端车规和工业品与普通消费品周期不同。
- 日系、台系、大陆厂商在产品等级和客户结构上有明显差异。
```

### 3.2 主题层

主题层回答“这个变化如何影响一组公司”。

主题可以是：

- AI 服务器产业链。
- 消费复苏。
- 国产替代。
- 渠道改革。
- 原材料涨价。
- 出海。
- 价格战。
- 库存周期。
- 政策变化。

主题层连接行业与公司。它不是泛泛讲行业，也不是单家公司模型，而是跟踪某个变量如何传导到相关公司。

例子：

```text
MLCC 库存周期主题
- 行业库存从高位去化到补库，会先影响价格，再影响稼动率和利润率。
- 高端产品公司受价格压力较小，普通品公司弹性更大但波动更强。
- 对公司 A，主要体现为出货量恢复。
- 对公司 B，主要体现为产品 mix 改善。
- 对公司 C，主要体现为价格和毛利率修复。
```

### 3.3 公司层

公司层回答“这家公司在行业里处于什么位置，它的业绩由什么驱动”。

它关注：

- 公司生意模式。
- 核心产品和客户。
- 业务拆分。
- 管理层战略。
- 竞争优势和短板。
- 历史业绩归因。
- 关键经营指标。
- 行业变量如何影响公司。
- 最新边际变化是否改变公司理解。

公司层最终沉淀到业务与业绩底稿。

例子：

```text
某 MLCC 公司
- 在行业里偏高端还是偏普通品。
- 收入增长来自行业补库，还是来自客户份额提升。
- 毛利率变化来自价格、mix、稼动率，还是原材料。
- 与同行相比，弹性来自产品结构还是产能利用率。
```

## 4. 业务与业绩底稿：连接工作台和 MKA 的中枢文件

业务与业绩底稿是外部工作台和 MKA 之间的中枢文件。

它不是分析师手写观点，也不是 official 核心假设。

三者关系：

| 文件 | 谁写 | 身份 | 权威性 |
|---|---|---|---|
| 公司判断和最新观点.md | 分析师 | 手写宪法 | 最高 |
| 业务与业绩底稿.md | 工作台 Agent | 公司理解中枢 | 强参考 |
| 核心假设.md | KA | official 模型裁决稿 | 模型源头 |

业务底稿负责吸收一切投研信息，并用固定投研框架整理成可复盘的人话理解。

它回答：

- 这家公司是什么生意。
- 它所在行业怎么影响它。
- 业绩应该怎么拆。
- 历史变化来自哪里。
- 核心驱动是什么。
- 哪些信息已经被采纳。
- 哪些信息只是线索。
- 哪些问题需要交给 KA 裁决。

业务底稿不写：

- knobs。
- yaml1 path。
- compiler family。
- official 预测。
- terminal。
- DCF 结论。

## 5. 业务底稿建议框架

业务底稿应当是一个长期维护的公司理解文件，而不是一次性摘要。

建议结构：（日志可以单拆出来）

```markdown
# {公司} 业务与业绩底稿

## 0. 当前结论快照
- 一句话生意:
- 所属行业与核心主题:
- 当前核心 thesis:
- 最大争议:
- 最近边际变化:
- 对 KA 的影响:

## 1. 行业判断
- 行业周期:
- 供需和价格:
- 竞争格局:
- 政策/技术/渠道变化:
- 核心跟踪指标:

## 2. 公司与行业的联系
| 行业变量 | 对公司的影响机制 | 历史证据 | 当前判断 | 重要性 |
|---|---|---|---|---|

## 3. 公司信息
- 产品和服务:
- 客户和渠道:
- 产能和供应链:
- 成本结构:
- 管理层战略:
- 竞争优势:
- 风险和短板:

## 4. 业绩怎么拆
| 模块 | 历史证据 | 解释了什么 | 可信度 | 备注 |
|---|---|---|---|---|

## 5. 核心驱动
| 驱动 | 作用机制 | 历史证据 | 当前判断 | 对模型的启发 |
|---|---|---|---|---|

## 6. 历史归因
- 收入变化来自哪里:
- 毛利变化来自哪里:
- 费用率变化来自哪里:
- below-OP / 税 / 少数股东是否扭曲净利:

## 7. 边际变化日志
| 日期 | 来源 | 新信息 | 改变了什么理解 | 处理 |
|---|---|---|---|---|

## 8. 证据地图
| 信息 | 来源 | 可信度 | 当前状态 | 去处 |
|---|---|---|---|---|

## 9. 收纳区
重要但暂不入模、不足以裁决、或只作为背景的信息。

## 10. 给 KA 的问题
| 问题 | 背景 | 可选判断 | 需要谁拍板 |
|---|---|---|---|
```

## 6. 信息如何进入工作台

所有非标信息先进入工作台，不直接冲击 KA。

信息来源包括：

- 研报。
- 纪要。
- 公告。
- 电话会。
- 管理层访谈。
- 行业数据。
- 竞品动态。
- 专家交流。
- 市场段子。
- Alphapai 输出。
- 评级报告素材。

每条新信息先变成一张“信息卡”：

```markdown
### 信息卡
来源:
日期:
类型: 研报 / 纪要 / 公告 / 段子 / 行业数据 / 管理层
核心内容:
影响对象: 行业 / 主题 / 公司 / 业务线 / 成本 / 费用 / 风险
可信度:
是否改变当前业务底稿:
建议处理: 采纳 / 收纳 / 待核 / 丢弃
```

处理原则：

- 重要信息必须有去处。
- 段子可以收纳，但必须标可信度。
- 不确定信息进入待核，不得包装成判断。
- 新材料不一定改变模型假设。
- 只有改变公司理解或核心驱动的信息，才建议交给 KA。

## 7. 工作台如何统摄行业研究到公司

工作台的关键不是存材料，而是建立“行业变量 -> 公司影响 -> 模型问题”的链路。

典型链路：

```text
行业判断
-> 主题变化
-> 公司受影响的业务线
-> 历史证据
-> 当前理解
-> 是否交给 KA 裁决
```

例子：MLCC

```text
行业层:
MLCC 行业进入补库阶段，高端和普通品周期差异明显。

主题层:
库存周期改善可能带来价格修复和稼动率提升。

公司层:
公司 A 普通品占比高，利润弹性更大。
公司 B 高端车规占比高，价格弹性较小但稳定性更强。

业务底稿:
记录各公司产品结构、客户结构、历史毛利率弹性和库存周期敏感度。

KA:
只裁决是否把这些理解转成收入、毛利或费用假设。
```

## 8. 与 MKA 的分工

外部工作台不替代 MKA。

分工如下：

| 层 | 角色 | 主要产物 |
|---|---|---|
| 外部工作台 | 吸收行业、主题、公司和边际变化 | 行业底稿、主题底稿、公司信息、信息卡 |
| 业务底稿 | 汇总为公司人话理解中枢 | 业务与业绩底稿.md |
| KA | 官方核对和模型裁决 | 核心假设.md |
| COMP | 翻译和审计 | yaml1 + forecast |

KA 的未来职责应更轻：

- 读取业务底稿。
- 读取分析师手写宪法。
- 读取 /init 官方事实。
- 核对业务底稿与官方事实是否冲突。
- 裁决哪些理解进入 official 核心假设。
- 记录哪些信息收纳、缺口或未采纳。

KA 不应重新翻材料堆。

## 9. 前端假设修改如何回流业务底稿

这是保持系统自洽的关键。

前端改假设后，不应该只更新核心假设和 forecast。它还应该形成一条“判断回流记录”，回到业务底稿。

回流记录建议格式：

```markdown
### 假设回流记录 - YYYYMMDD
来源: frontend-edit / adj quick / ka / annual-update
影响项:
原假设:
新假设:
修改原因:
属于哪类:
- 业务理解变化
- 官方事实更新
- 情景试算
- 纯数值微调
是否改变业务底稿正文:
回写位置:
后续待核:
```

回流处理原则：

| 修改类型 | 业务底稿处理 |
|---|---|
| 业务理解变化 | 必须更新相关正文和边际变化日志 |
| 官方事实更新 | 更新历史归因、证据地图和事实状态 |
| 情景试算 | 只记录为试算，不改变当前主判断 |
| 纯数值微调 | 记录原因；若没有业务解释，不改正文 |
| KA 正式裁决变化 | 标注已被 official 采纳，并记录核心假设文件 |

这样业务底稿不会被每次前端试算污染，但正式判断变化不会丢。

## 10. 业务底稿的状态管理

为了保持自洽，业务底稿里的重要判断应有状态。

建议使用：

| 状态 | 含义 |
|---|---|
| current | 当前采用 |
| contested | 有争议 |
| stashed | 收纳，不进入主判断 |
| pending | 待核 |
| superseded | 已被新材料替代 |

每条重要判断都应能回答：

- 它来自哪里。
- 是否仍有效。
- 是否和官方事实冲突。
- 是否已被 KA 采纳。
- 如果模型假设改了，业务底稿是否已有回流记录。

## 11. 第一阶段目标

第一阶段不要做成大而全系统。

建议先做到：

1. 定义行业、主题、公司三层工作台的文档框架。
2. 定义业务与业绩底稿框架。
3. 让非标信息统一先进业务底稿。
4. 让 KA 读取业务底稿，而不是直接重读所有材料。
5. 让前端正式假设修改生成回流记录，进入业务底稿。

第一阶段不要求：

- 自动判断所有材料真假。
- 自动生成模型假设。
- 自动替代分析师判断。
- 自动维护完整行业资料库。

## 12. 会议讨论问题

需要继续讨论的问题：

1. 工作台第一版先服务哪些行业和公司。
2. 业务底稿是每家公司一份 living file，还是每次运行生成一份日期稿。
3. 行业层和主题层是否也需要 living file。
4. 前端修改假设后，哪些情况必须回写业务底稿正文。
5. KA 是否只允许读取业务底稿、分析师宪法和 /init 事实，还是保留按需读原材料的例外。
6. 外部工作台如何处理低可信度段子和市场传闻。

## 13. 最终目标

最终希望形成的工作方式：

```text
行业研究和非标信息
-> 外部投研工作台吸收
-> 业务底稿沉淀公司理解
-> KA 裁决 official 假设
-> COMP 翻译运行
-> 前端/年度更新/边际变化回流业务底稿
```

外部工作台让 MKA 不再承担整个投研世界。

业务底稿让公司理解长期自洽。

KA 只做首席裁决。

MKA 的模型质量来自：

```text
分析框架 + 业务底稿 + 官方事实 + 分析师拍板 + 可审计翻译
```
