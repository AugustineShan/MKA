# 季度跟踪层 Implementation Plan

> **For agentic workers:** Implement this plan task-by-task from the checkboxes. If your runtime has an `executing-plans` skill/tooling, use it to keep progress and verification explicit.

**Goal:** 给已做好的年度预测加一个季度粒度监控/展示层——把年度预测按历史季节相位摊开、用季报实际对轨、前端可调首个预测年未发布季的驱动参数，四态自动判定，年度/季度勾稽成立，不进 DCF、不回写年度、不产新文件。

**Architecture:** 现算视图 + 仅存③覆盖。新增 `src/quarterly_tracker.py` 计算引擎：每次打开从 `forecast_is.csv`（年度预测）+ `clean_quarterly`（单季实际）+ 历史季节形状 + 新表 `quarterly_overrides`（③人工参数）现算四态季度 IS。④ Q4 在数值层面倒挤 = 年度 − Σ(Q1:Q3)，年度勾稽自动成立，任意 ①②③ 混搭均成立。`workbench.py` 加 3 个端点；前端加「季度展示」tab。

**Tech Stack:** Python 3.11（系统全局，禁 venv）、SQLite、pandas、FastAPI（workbench）、React+Vite（前端）。会计科目/排序/标签复用 `src/field_registry.yaml`（唯一真源）；subtotal 公式复用 `src/clean.py` 的 `is_bucket_sum` + `check_is`（公式留代码，不重造）。

## §0 决策摘要（brainstorming 已确认）

1. **季度分布总原则** = 年度金额锚 + 历史季度相位 + 状态锁定 + Q4 残差。年度预测给全年金额，历史季度给形状，①实际/③人工先锁住，②只填未锁 Q1–Q3，④ Q4 永远倒挤。
2. **③ 可编辑参数集** = 营业收入 yoy + 毛利额/毛利率 + 销售费用额/率 + 管理费用额/率 + 研发费用额/率 + 税附加额/率 + 财务费用额 + 所得税额/率（按季可拨；前端可用率展示，后端统一落成该季金额锁定）。
3. **存储/计算模型** = 现算视图 + 仅存③覆盖（新表 `quarterly_overrides`，状态派生不落库）。
4. **毛利率/费用率不直接继承年度率**：先锚定年度毛利额/费用额，再按历史季度金额相位分配，最后反推季度率。这样收入季节性变化时，季度毛利率/费用率仍有真实形状。
5. **年度勾稽** = 交给 ④ 残差兜底；② 分配不强行解约束方程组，但金额锚保证 Q1–Q3 的算法值有季节解释，④ 只吸收已发布/人工/算法路径与年度目标之间的剩余差。
6. **Q4 验证带** = 近3年 Q4 同比率 [min,max]，落外软提示不拦；N=3 可调。
7. **前端** = 新增第 6 个 tab「季度展示」。
8. **财务费用和所得税也按季节相位处理**：财务费用受年底结汇、利息结算、一次性财务收益影响，所得税受季度结税/递延税项影响，不能简单套年度率。两者先按历史金额相位分配，Q4 倒挤并做隐含率软提示。
9. **数据时效** = 开发默认使用“时光机”测试状态：新乳业 DB 最新实际回到 2024Q4，2025 是首个预测年且全季未发布。不得在默认验证里重跑 `data_fetcher` 拉回 2025/2026；actual 混态用 fixture 或单独可选恢复测试。
10. **范围** = 后端计算引擎 + workbench 端点 + 前端可编辑 tab 一次性做完，分 4 个 Phase 提交。

**边界（不做）：** 不进 DCF、不回写年度、不产新文件（仅 db 内新表）、不做业务拆分（只 IS 标准科目）、季度默认只覆盖首个预测年、不改 `clean.py`/`field_registry.yaml`/`init.py`。

## §2 核心算法契约（实现者必读）

### 2.1 IS 科目分类（method）

每条 IS leaf 按下表归类。subtotal/derived 不独立分布，由 leaves 加总（复用 `clean.is_bucket_sum` + `check_is` 公式）。

| method | 含义 | ② 季度公式 | ④ 公式 | 字段 |
|---|---|---|---|---|
| `revenue_amount` | 收入金额锚 | `rev_Y × phase_q`, `phase_q = priorQ_rev / prior_annual_rev`；③ 若输入 yoy，先换成 `rev_q = priorQ_rev × (1 + yoy_override)` | `rev_Y − ΣQ1:Q3` | `revenue` |
| `gross_profit_amount` | 毛利额锚，反推毛利率 | `gp_Y × phase_q`, `phase_q = priorQ_gp / prior_annual_gp`; `oper_cost_q = revenue_q − gp_q`；③ 若输入 gpm，先换成 `gp_q = revenue_q × gpm_override` | `oper_cost_Y − ΣQ1:Q3 oper_cost`（等价于 `gp_Y − ΣQ1:Q3 gp` 后反推） | `oper_cost` |
| `expense_amount` | 费用金额锚，反推费用率 | `item_Y × phase_q`, `phase_q = priorQ_item / prior_annual_item`；③ 若输入费用率，先换成 `item_q = revenue_q × rate_override` | `item_Y − ΣQ1:Q3` | `biz_tax_surchg`, `sell_exp`, `admin_exp`, `rd_exp`, `other_bus_cost` |
| `seasonal_amount` | 损益/税费/财务费用/少数股东等金额项 | `item_Y × phase_q`, `phase_q = priorQ_item / prior_annual_item`；可按绝对值权重兜底，见 §2.3 | `item_Y − ΣQ1:Q3` | 见下表 |

`seasonal_amount` 类字段（phase 分摊，去年优先+回退）：
`assets_impair_loss`, `credit_impair_loss`→`credit_impa_loss`, `oth_impair_loss_assets`, `oth_income`, `invest_income`, `fv_value_chg_gain`, `asset_disp_income`, `net_expo_hedging_benefits`, `forex_gain`, `non_oper_income`, `non_oper_exp`, `fin_exp`, `income_tax`, `minority_gain`, 以及金融专用项 `int_income`/`comm_income`/`n_oth_income`/`n_oth_b_income`/`prem_*`/`compens_*`/`reins_*`/`int_exp`/`comm_exp`/`oper_exp`/`insurance_exp`/`out_prem`/`une_prem_reser`/`div_payt`/`reser_insur_liab`（工业公司多为 0，phase 分摊天然回退均摊）。

> **为什么税和财务费用也用金额相位**：所得税不是稳定季度税率，可能受季度结税、递延所得税、一次性税务调整影响；财务费用也可能受年底结汇、利息结算、贴息确认、汇兑损益集中影响。它们在季度层按金额相位继承，Q4 倒挤后用隐含税率/财务费用率做软提示，不用全年率硬摊。

> **`sub_item` 子项不独立分布**：`fin_exp_int_exp`/`fin_exp_int_inc`（`fin_exp` 的 `resolve_children`）、`ass_invest_income`（`invest_income` 子项）等是展示用子项，`is_bucket_sum` 已跳过它们不重复计入 subtotal。季度层只对**父项**做 phase 分布；子项季度值 = 父项季度值 ×（去年同季子项/父项占比）作展示，不参与勾稽。

subtotal/derived（由 leaves 加总，复用 `clean.is_bucket_sum` + `check_is` 公式，clean.py:871-988）：
```
total_revenue_q = is_bucket_sum("revenue_item", q_leaves)
total_cogs_q    = is_bucket_sum("cost_item", q_leaves)
operate_profit_q = total_revenue_q − total_cogs_q + is_bucket_sum("operating_adjustment", q_leaves)
total_profit_q  = operate_profit_q + non_oper_income_q − non_oper_exp_q
n_income_q      = total_profit_q − income_tax_q
n_income_attr_p_q = n_income_q − minority_gain_q
total_opcost_q  = total_cogs_q − fin_exp_q   # clean.py:906
```

> **不展示/不季度推导类**：`comprehensive`（其他综合收益等 4 项）、EPS/分红/转增等不参与利润表勾稽的 `derived` 展示项，本版不进入季度表（或只在后端 `excluded_rows` 标注原因）。不要把年度值塞进 Q1–Q4，避免误导成“四个季度各有一个全年数”。`sub_item` 只作父项拆分展示，不参与 subtotal。

### 2.2 状态机（§3 优先级，派生不落库）

对预测年 Y 的季 q ∈ {1,2,3,4}：
1. `period("YQq") ∈ clean_quarterly` → **① 已发布**（值 = clean_quarterly 实际，只读）
2. `q == 4` → **④ Q4验证**（值 = 年度 − Σ(Q1:Q3)，逐 leaf 倒挤；subtotal 由 ④ leaves 加总）
3. `quarterly_overrides` 有该 period 行 → **③ 人工**（比率参数用 override，绝对值项仍走②相位）
4. 否则 → **② 继承**（§2.1 公式）

一年内四列可混态。③ 覆盖允许前端以 `revenue_yoy / gpm / sell_exp_rate / admin_exp_rate / rd_exp_rate / biz_tax_surchg_rate / fin_exp_abs / income_tax_rate / income_tax_abs` 等形式输入；后端保存时统一落成该季锁定金额（同时保留 `input_param`/`input_value` 供展示）。③ 季的被覆盖金额锁住；未覆盖项目仍走②相位。清回② = 删该 period 的 override 行。

同一 `(ticker, period, locked_field)` 只能有一个有效覆盖：例如 `gpm` 与 `gross_profit_abs` 都锁 `gross_profit`，`income_tax_rate` 与 `income_tax_abs` 都锁 `income_tax`。`set_override` 在写入新覆盖前，必须事务内先删除同 period、同 locked_field 的旧行，再 upsert 新的 `param` 行；这样 UI 可以自由在“率/金额”输入之间切换，而计算层只看到一个锁定金额。

### 2.3 季节参考（去年优先 + 回退，全 method 统一）

- `priorQ_*` = 被预测年 Y 的前一年(Y−1)同季值。Y−1 同季若为 ① 实际，用实际；若 Y−1 也是预测年，用其②计算值（跨年链式，按年升序计算并缓存）。
- Y−1 同季缺数据/为 0（如 2017Q1/Q2 缺）→ 回退到最近可用年同季值。
- 全无可用年同季 → 收入/毛利/费用/税费/财务费用等金额退均摊(÷4)，并在 view 标 `degraded=True`。graceful，不崩。
- 对正负混杂或 prior_annual 接近 0 的项目：优先用最近 3 个完整年度同季**绝对值权重**分配年度金额；如果三年权重仍不可用，再均摊。这样避免 `phase_q = priorQ / prior_annual` 在年度净额很小但季度波动很大时爆炸。
- `prior_annual_*` = Y−1 年度值（① 年用 `clean_annual`，预测年用 `forecast_is.csv`）。

### 2.4 年度勾稽（核心不变量）

④ 在**数值层面**倒挤每个 leaf：`leaf_Q4 = leaf_annual − Σ(Q1:Q3 leaf)`。因此：
- 每个 leaf 的 Σ(Q1:Q4) = 年度 ✓
- subtotal 由 leaves 加总 → Σ(Q1:Q4) = 年度 ✓
- `income_tax`、`minority_gain`、`fin_exp` 都是 leaf 金额，Q4 同样残差倒挤，因此 Σ income_tax / minority_gain / fin_exp = 年度 ✓；隐含税率、少数股东占比、财务费用率只用于提示，不用于硬摊。
- **任意 ①②③ 混搭均成立**（①/③ 是固定值，② 是估计，④ 吸收全部残差）

### 2.5 Q4 验证带（§5）

④ 算出后，核隐含比率：`yoy_Q4 = rev_Q4 / priorQ_rev − 1`、`gpm_Q4 = 1 − opcost_Q4/rev_Q4`、`expense_rate_Q4 = expense_Q4/rev_Q4`、`tax_rate_Q4 = income_tax_Q4/total_profit_Q4`、`fin_exp_rate_Q4 = fin_exp_Q4/revenue_Q4`、`minority_ratio_Q4 = minority_gain_Q4/n_income_Q4`。
带 = 近 3 年（可用）Q4 同比率 [min, max]，并给 20% 宽容垫（或至少 ±2pct，取更宽者）。落外 → 软提示对象 `{ratio, implied, band_min, band_max, msg}`，不拦。金额项本身不核，只核它们反推出来的率/占比是否离谱。

### 2.6 年份范围

- 默认展示年 = `forecast_is.csv` 首个预测期，即 `min(period)`。新乳业时光机状态下 `base_period=2024`、首个预测年 `2025`，所以默认季度跟踪年是 2025。
- 历史 ① 全实际（只读）；默认预测年中已发布的季①、未发布的②/③、Q4 为④（Q4 实际发布后翻①）。
- 未来预测年默认不展示；允许 URL `?year=` 手动切换到 forecast horizon 内年份用于调试，但 UI 首屏只给首个预测年。
- "在轨差异" = ① 实际 − ② 继承值（都用当前年度预测现算，无需冻结快照；活链接意味着年度预测重跑后②会随之变化）。

## 附录 A：`quarterly_overrides` 表 schema

```sql
CREATE TABLE IF NOT EXISTS quarterly_overrides (
  ticker     TEXT NOT NULL,
  period     TEXT NOT NULL,      -- "2025Q2"
  param      TEXT NOT NULL,      -- revenue_yoy|revenue_abs|gpm|gross_profit_abs|sell_exp_rate|sell_exp_abs|admin_exp_rate|admin_exp_abs|rd_exp_rate|rd_exp_abs|biz_tax_surchg_rate|biz_tax_surchg_abs|fin_exp_abs|income_tax_rate|income_tax_abs...
  value      REAL NOT NULL,
  locked_field TEXT,             -- 后端换算后的锁定字段，如 revenue/gross_profit/sell_exp/income_tax
  locked_value REAL,             -- 该字段锁定金额（百万元）；计算优先读它
  updated_at TEXT NOT NULL,      -- 调用方传入 ISO8601（引擎不调 date.now()）
  note       TEXT,
  PRIMARY KEY (ticker, period, param)
);

CREATE INDEX IF NOT EXISTS idx_quarterly_overrides_locked
ON quarterly_overrides (ticker, period, locked_field);
```

业务规则：同一 `(ticker, period, locked_field)` 只保留一条有效覆盖，由 `set_override` 在事务内先删旧锁定字段再写新 `param` 实现；不要让 `gpm`/`gross_profit_abs` 或 `income_tax_rate`/`income_tax_abs` 同时锁同一个字段。
