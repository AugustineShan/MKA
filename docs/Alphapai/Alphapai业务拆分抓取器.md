# Alphapai业务拆分抓取器

把下面整段复制给 Alphapai 网页端。替换 `{公司}`、`{ticker}`、`{用户指定主拆分或定向取数表}`、`{关注点}` 即可。

---

# 任务：为后续建模抓取 Alphapai 业务拆分历史 factpack

请基于 Alphapai 自有金融数据库、公告、年报、季报、业绩会纪要、研报图表和可检索资料，为 `{公司}`（`{ticker}`）生成一份 **业务拆分历史事实底稿**。

这不是核心假设稿，不是预测稿，不是估值报告。你的唯一任务是尽可能抓全 **非标准业务拆分历史数值**，供后续 `Alphapai-load 核心假设参考稿` 和 `/ka` 使用。

## 0. 产物定位

严格遵守：

- 模式：`alphapai-business-split`
- 状态：`factpack/reference`
- 用途：只供后续 Alphapai-load / `/ka` 建模裁决，不可直接编译。
- 只抓历史事实、口径桥和缺口；**禁止写预测、目标价、评级、DCF、terminal、knobs**。
- 用户指定主拆分或定向取数表必须锁定为事实底稿主轴，不得自行软降级为更容易的披露口径。
- 必须包含 `## 待 /ka 裁决清单`，只列口径、缺口、可信度和下一步取证/裁决事项；不写未来预测候选。

链路是：

```text
Alphapai业务拆分抓取器 -> 业务拆分历史 factpack -> Alphapai-load 核心假设参考稿 -> /ka 裁决 -> official 核心假设
```

不是：

```text
Alphapai业务拆分抓取器 -> 预测候选 / knobs / DCF / 正式核心假设
```

## 1. 输入模式与主拆分锁定

用户指定主拆分或定向取数表：

```text
{用户指定主拆分或定向取数表}
```

例如用户说“青岛啤酒主拆分方式是高端 / 中端 / 低端三个产品线”，你必须把 **高端 / 中端 / 低端** 作为主建模拆分历史表的行轴。

### 1.1 默认模式：主拆分锁定

当用户只给出拆分方向，例如“高端 / 中端 / 低端”、“产品线 A/B/C”、“地区/渠道/价格带”，你按该主拆分设计事实底稿。

如果公司公告没有直接披露用户主拆分：

- 不得把主拆分改成官方披露口径。
- 必须写“官方披露 -> 用户主拆分”的桥表、公式、可回勾检查和缺口。
- 可以把官方口径作为副拆分或 sanity check，但不能替代用户指定主拆分。
- 推断项必须标 `推断口径`、`待校准`、来源可信度，不得伪装成公司披露。

### 1.2 定向取数模式：按用户 leaf 表逐项找历史

当用户直接给出一张拆分表，尤其是包含 `部`、`leaf`、`2025-2031 yoy`、`abs` 或类似列的表格时，自动进入 **定向取数模式**。

定向取数模式下：

- 严格按用户表里的 `部` / `leaf` 层级和顺序取数，不重新设计拆分，不合并、不改名、不删 leaf。
- 未来 yoy、abs、flat 等列只用于识别用户想建模的 leaf，不得评价、修正或续写这些预测。
- 对每个 leaf 查历史收入、销量/件数、ASP/价格、yoy、占比、口径、来源、可信度。
- 如果官方没有 leaf 级披露，必须写清可用的父项、近似项、品牌/产品映射、研报估算和缺口。
- 输出的主表行轴必须是用户给的 leaf，而不是 Alphapai 自己认为更稳的官方披露口径。
- `其他`、`水饮等`、`植物基` 这类兜底 leaf 也要查历史；找不到时写 `未披露`，不要直接删掉。

## 2. 检索优先级

按以下顺序检索和摘数：

1. 公司公告、年报、季报、经营数据公告、交易所文件。
2. 业绩会、投资者关系活动记录、管理层纪要。
3. 高质量深度研报、行业报告、图表、专家纪要。
4. 产业链与经营原子：产品、品牌、价格带、地区、渠道、客户、门店、用户、订单、产能、销量、吨价/客单价/ARPU。
5. 历史口径变化、管理层对拆分口径的定性说明。

如用户给 `{关注点}`，优先围绕它检索，但不能牺牲用户指定主拆分历史表的完整性。

可信度分层：

- A：公司公告、财报、交易所文件。
- B：业绩会、投资者关系、管理层纪要。
- C：高质量研报、行业报告。
- D：弱来源、市场传闻、你的推断。

状态标签、裁决结果、来源层级、unit 与候选稿标准块语法遵守 MKA 的 `docs/核心假设源语言语法规范.md`：不得自造 status、decision、unit 或 family 同义词。

## 3. 硬边界：只抓历史，不做预测

禁止输出：

- 未来年份预测候选。
- 收入、销量、ASP、毛利率、费用率的未来增速。
- `knobs` fenced block。
- `horizon`、`terminal`、`perpetual_growth`。
- 毛利率、销售/管理/研发费用率的长篇历史复盘。
- BS/CF/DCF 驱动预测，如 capex、折旧、营运资本、WACC、股本。

允许输出：

- 历史收入、销量/件数、ASP/价格、yoy、占比。
- 用户主拆分的口径桥、公式和可回勾校验。
- 对主拆分有直接解释力的辅助拆分历史表；没有增量价值时可以不写。
- 缺口清单和下一步应补哪些年份/口径/来源。
- 与业务拆分直接相关的成本或毛利率历史片段，但只作为解释，不展开预测。

## 4. 输出格式

只输出 Markdown，标题为：

```markdown
# {公司}-{日期}-Alphapai业务拆分事实底稿.md
```

抬头必须包含：

```text
模式: alphapai-business-split
状态: factpack/reference
公司: {公司}
ticker: {ticker}
取数模式: {主拆分锁定 / 定向 leaf 表}
用户指定主拆分: {用户指定主拆分或定向取数表中的 leaf}
事实截至: {最新已披露年度/季度}
资料截至: {检索到的最新资料日期}
用途: 供 Alphapai-load / ka 建模，不可直接编译
禁止: 预测、knobs、DCF、目标价、评级
```

## 4.1 一页结论

用 5-8 条 bullet 说明：

- 用户指定主拆分是什么。
- 如果是定向取数模式，用户给出的 `部` / `leaf` 表是什么，哪些 leaf 有历史证据，哪些完全未披露。
- 官方直接披露的最稳定口径是什么。
- 用户主拆分最近 5 年有多少可得历史，哪些线是官方、哪些是推断。
- 哪些年份/指标可以回勾总量，哪些不能。
- 是否存在对主拆分有直接解释力的辅助拆分；没有就说明不强行展开。
- 最大缺口是什么，后续 Alphapai-load / `/ka` 必须怎么裁决。

## 4.2 资料清单

| 材料 | 日期 | 类型 | 可用于哪张表 | 可信度 | 时效性 | 备注 |
|---|---:|---|---|---|---|---|

每条材料必须说明它支持哪个拆分表或桥表。不要只列“年报/研报”而不说明用途。

## 4.3 官方披露事实库存

先列官方或高可信资料直接给出的事实，不做建模映射。

建议格式：

```markdown
#### 官方披露事实库存
| 年份/期间 | 披露口径 | 收入 | 收入yoy | 销量/件数 | 销量yoy | ASP/价格 | 占比 | 单位 | 来源 | 可信度 | 备注 |
|---|---|---:|---:|---:|---:|---:|---:|---|---|---|---|
```

说明：

- 最近 5 年是完整性重点，必须尽力逐年抓取。
- 只拿到最近 1-2 年时，仍保留表格并标 `最近 5 年历史不足，待补`。
- 早于最近 5 年的数据不要求完整；有意义、有参考价值的年份可以继续抓取，放入主表或事实收纳区。
- 某列未披露就写 `未披露`，不要删列。

## 4.4 官方披露到用户主拆分的桥

这是本 factpack 最关键的部分。若用户主拆分不是公司直接披露口径，必须逐年写桥表。

建议格式：

```markdown
#### 官方披露 -> 用户主拆分桥表
| 年份/期间 | 用户主拆分线 | 官方披露输入A | 官方披露输入B | 其他输入/估算 | 计算公式 | 结果 | 是否可回勾总量 | 来源 | 可信度 | 缺口/待校准 |
|---|---|---:|---:|---:|---|---:|---|---|---|---|
```

规则：

- 最近 5 年内，用户主拆分有几条线，每年就至少有几行。
- 公式必须能读懂，不能只写“推断”。
- 能回勾总量时写清回勾公式，例如三条线销量之和 = 总销量。
- 不能回勾时写清原因，例如“中端/高端拆分依赖经典单品研报估算，无法直接回勾收入”。
- 多源冲突时并列候选，不取平均。

青岛啤酒示例口径，不得机械照抄数值：

```text
低端 = 其他品牌 + 主品牌非中高端
中端 = 经典 / 1903 等 6-10 元价格带
高端 = 中高端以上 - 中端，或纯生 / 白啤 / 超高端合计
```

## 4.5 用户主拆分历史总表

这一节必须把用户指定主拆分做成完整矩阵。不得只写最近一年。

建议格式：

```markdown
#### 用户主拆分历史总表
| 年份/期间 | 主拆分线 | 官方/推断口径 | 收入 | 收入yoy | 销量/件数 | 销量yoy | ASP/价格 | ASPyoy | 占比 | 单位 | 来源 | 可信度 | 备注 |
|---|---|---|---:|---:|---:|---:|---:|---:|---:|---|---|---|---|
```

强制要求：

- 行轴必须使用用户指定主拆分，不得换成官方口径。
- 最近 5 年完整性最高优先：用户主拆分 × 最近 5 年在资料支持时应尽量完整。例如最近 5 年 × 3 条线就优先尝试 15 行；拿不到就标缺口，不为了补表扩大到无关材料。
- 早于最近 5 年的数据不要求齐全；只抓有意义、有参考价值、能解释周期或结构变化的年份。
- 拿不到的单元格填 `未披露`、`待补`、`推断待校准`，不要删行。
- 任何 D 类推断必须在备注写清来源和计算逻辑。
- 如果最近 5 年历史不足，必须写“缺哪一年、缺哪条线、缺收入/销量/ASP 哪一项”。

## 4.6 高价值辅助拆分历史表（可选）

一个深度、完整、可回勾的主拆分方式，强于很多种泛泛的辅助拆分。除用户主拆分外，只有当辅助拆分能直接解释主拆分、校验口径或补足缺口时才写；没有高价值辅助拆分时，可以不写，并说明“不强行展开泛泛副拆分”。

可选方向：

1. 产品/价格带/品牌。
2. 地区。
3. 渠道/客户/门店/用户/订单。
4. 海外/内销。
5. 产能/利用率。

辅助拆分表格式：

```markdown
#### 副拆分A：{维度}历史
| 年份/期间 | 维度项 | 口径 | 数值 | yoy | 占比 | 单位 | 来源 | 可信度 | 备注 |
|---|---|---|---:|---:|---:|---|---|---|---|
```

拿不到完整历史时写具体缺口，不能只写一句“资料未披露”。辅助拆分不得挤占主拆分最近 5 年完整矩阵的篇幅。

## 4.7 口径冲突与裁决备忘

| 冲突/缺口 | 候选A | 候选B | 哪个更稳 | 为什么 | 后续交给谁裁决 |
|---|---|---|---|---|---|

这里只处理历史口径和拆分口径冲突，不处理未来预测。

## 4.8 待 /ka 裁决清单

本节只交历史 factpack 的晋升议程，不写预测。每项必须指向具体年份、业务线、指标、口径或来源缺口。

| 事项 | 候选值/方向 | 证据 | 分歧/缺口 | 建议处理 |
|---|---|---|---|---|
| 用户主拆分口径 | | | | /ka 骨架门裁决 |
| 缺失历史原子 | | | | 补资料或进入缺口区 |
| 推断口径可信度 | | | | /ka 只可收纳或待人工确认 |

## 4.9 事实收纳区

放不进入主表但可能对后续建模有用的历史事实：

- 价格带定义。
- 产品矩阵。
- 管理层对结构升级/渠道/区域的表述。
- 罐化率、即饮/非即饮、渠道结构等历史片段。
- 与业务拆分相关但无法入表的图表摘录。

仍然要求年份、单位、来源和可信度。

## 5. 最终自检

输出前逐项检查：

- 抬头是 `模式: alphapai-business-split`，`状态: factpack/reference`。
- 明确写了 `禁止: 预测、knobs、DCF、目标价、评级`。
- 没有 `knobs` fenced block。
- 没有 `horizon`、`terminal`、`perpetual_growth`。
- 没有未来年份预测候选。
- 用户指定主拆分没有被官方口径替代。
- 定向取数模式下，主表行轴严格等于用户给出的 `部` / `leaf`，未来 yoy 只当 leaf 上下文，没有被评价或续写。
- 有官方披露事实库存。
- 有“官方披露 -> 用户主拆分桥表”。
- 有“用户主拆分历史总表”，且行轴是用户指定主拆分。
- 主拆分表最近 5 年尽力完整；早于最近 5 年只抓有意义、有参考价值的数据。
- 没有强行凑辅助拆分；若写辅助拆分，它直接服务主拆分解释、校验或缺口。
- 缺口写到具体年份、线、指标、来源，不用空泛描述。
- 包含 `## 待 /ka 裁决清单`，且没有把任何未来预测写进该 factpack。
