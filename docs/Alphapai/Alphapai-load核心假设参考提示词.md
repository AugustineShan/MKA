# Alphapai-load 核心假设参考稿提示词

把下面整段复制给 Alphapai 网页端。替换 `{公司}`、`{ticker}`、`{关注点}` 即可。

如果已经先跑过 `docs/Alphapai/Alphapai业务拆分抓取器.md`，请把它产出的 `{公司}-{日期}-Alphapai业务拆分事实底稿.md` 一并贴给 Alphapai。本提示词必须优先承接该 factpack 的主拆分、桥表、副拆分和缺口，不得重新发明业务口径。

# 任务：把 Alphapai 金融数据库理解装载成核心假设参考.md

请基于 Alphapai 自有金融数据库和可检索资料，为 `{公司}`（`{ticker}`）生成一份 **`核心假设参考alphapai_yymmdd.md`**。

这不是研报摘要、不是估值报告、不是投资评级。你的任务是把数据库里的公司理解、业务线索、最新进展、预测候选和分歧，整理成一份后续建模 Agent 可以裁决的“人话核心假设参考稿”。

## 0. 产物定位

严格遵守：

- 模式：`alphapai-load`
- 状态：`reference`
- 用途：供后续核心假设裁决，不可直接编译，不可直接作为正式模型。
- 禁止输出 YAML1、JSON schema、DCF、目标价、买卖建议或正式估值结论。
- 你可以给候选值和建议旋钮，但必须标 `reference / 待裁决`。

链路是：

```text
Alphapai 金融数据库理解 -> 核心假设参考alphapai_yymmdd.md -> 后续裁决 -> 正式核心假设 -> 编译/预测
```

不是：

```text
Alphapai -> 直接 YAML1 / 直接 DCF / 直接正式核心假设
```

## 0.1 交互方式：先会议 memo，后完整参考稿

不要第一轮直接输出完整 `核心假设参考alphapai_yymmdd.md`。第一轮必须先像分析师开会一样，输出一页 **会议 memo**，讲清你对公司的理解、预测候选和待裁决问题，然后停止等待用户确认。

会议 memo 必须包含：

```text
我读完 Alphapai 材料后，先给结论：
1. 这家公司当前核心 thesis 我读成...
2. 利润表主链路最该裁决的是...
3. 我建议的业务骨架是...
4. 可用于 /ka 建模的业务拆分历史，我找到了哪些主拆分和副拆分...

时间轴候选：
| 项 | 候选 | 理由 |
|---|---|---|
| 历史事实至 | ... | ... |
| 显式期 | ... | ... |
| 衰减期 | ... | ... |
| 永续增长 | ... | ... |

关键预测候选：
| 区块 | 我的理解 | 候选旋钮 | 主要证据/分歧 |
|---|---|---|---|

待后续裁决：
- ...

这个理解方向可以吗？
```


## 0.2 与 Alphapai业务拆分抓取器的分工

`Alphapai业务拆分抓取器` 是上游 factpack，只抓历史业务拆分，不写预测。本 Alphapai-load 是下游 reference 稿，负责在 factpack 事实底座上补充利润表预测候选、时间轴候选、待裁决清单和 `knobs` 自报。

如果用户贴入了 `模式: alphapai-business-split`、`状态: factpack/reference` 的业务拆分事实底稿：

- 必须把 factpack 的 `用户指定主拆分` 或定向 leaf 表作为 4.4 收入拆分候选的主轴。
- 必须保留 factpack 的“官方披露 -> 用户主拆分桥表”和“用户主拆分历史总表”，不得降级成官方更容易披露的口径。
- factpack 中的缺口、推断口径、可信度和待校准标识必须原样继承。
- 可以把更稳健的官方两分法或其他披露口径放入“来源与裁决”或 sanity check，但不能替代用户指定主拆分。
- 如果 factpack 是定向取数模式，本稿必须按用户给出的 `部` / `leaf` 层级写收入拆分；未来 yoy 列只作为用户建模意图上下文，不得当作历史事实。
- 一个深度、完整、可回勾的主拆分优先于很多种泛泛副拆分；factpack 没有高价值辅助拆分时，本稿不得强行补凑。
- 本稿只在这些历史事实之上给预测候选；不要重新从零抓业务历史导致口径漂移。

## 1. 检索范围

请围绕用户指定的 `{公司}`、`{关注点}` 和已贴入 factpack，定向检索 Alphapai 数据库中的高权重材料；不要为扩大材料面而泛化检索。优先级如下：

1. 最新年报、季报、业绩预告、经营数据公告、重大事项公告。
2. 公司官网、交易所问询回复、投资者关系活动记录、业绩会纪要。
3. 高质量研报、深度报告、行业报告、一致预期和盈利预测。
4. 产业链、价格、销量、产能、渠道、门店、用户、订单、招投标等经营线索。
5. 历史业务变化、竞争格局、管理层表述、政策和行业周期。

如用户给出 `{关注点}`，优先围绕该关注点检索；没有关注点时，围绕利润表主链路补足候选，不追求材料覆盖面最大化。

输出中必须列出资料清单：材料名称、日期、类型、用途、可信度、时效性。

可信度分层：

- A：公司公告、财报、交易所文件。
- B：业绩会、投资者关系、管理层纪要。
- C：高质量研报、行业报告、一致预期。
- D：弱来源、市场传闻、你的推断。

资料冲突时必须写“来源与裁决”：候选 A、候选 B、暂采用哪个、为什么、未采用方放哪里。不能静默丢掉冲突信息。

## 2. 写作纪律

按“事实、推断、建议”三层写：

- 事实：来自公告、财报、纪要、研报或数据库的明确数据和表述。
- 推断：你基于多源资料归纳出的业务判断。
- 建议：可供后续裁决的候选旋钮或参数化方式。

关键数字必须带年份和来源。拿不到就写“未检索到 / 待核”，不要编。

不要倒算派生利润来反推旋钮。收入、毛利、费用、税率等要尽量写成业务驱动或明确比率/金额候选。

状态标签、裁决结果、来源层级、unit 与候选稿标准块语法遵守 MKA 的 `docs/核心假设源语言语法规范.md`：不得自造 status、decision、unit 或 family 同义词。

## 2.1 本稿最稀缺资产：业务拆分历史

财务 headline、费用率、税率、少数股东等标准利润表历史，后续 `/ka` 可以从 `/init` 的 `Agent/core_metrics_overview` 和 `Agent/defaults.yaml` 快速取得。本稿不要把主要篇幅花在重复财务历史表上。

你的主要增量价值是：围绕用户筛选的主题和本轮建模主线，从 Alphapai 数据库、年报、研报、图表和纪要里找出 **非标准业务拆分的关键历史数值**，尤其是 `/init` 官方财务表里没有的产品、品牌、价格带、地区、渠道、客户、门店、用户、订单、产能、销量、吨价/客单价/ARPU 等经营原子。

若已有 `Alphapai业务拆分抓取器` factpack，本稿的增量价值不是重新抓历史，而是 **保真承接 factpack 的用户主拆分历史、桥表、高价值辅助拆分和缺口**，并在其上形成预测候选。

强制要求：

- 主建模拆分必须有可追踪历史数值。最近 5 年是完整性重点，但只在资料支持时逐年写完整；早于最近 5 年的数据不要求齐全，有意义、有参考价值的年份可以抓取并收纳。
- 每个历史表优先包含：年份/期间、收入、销量或件数、价格或 ASP、同比、占比、单位、口径、来源、可信度。某列拿不到就写 `未披露`，不要为了补列扩大检索范围。
- 如果建模拆分是非标口径或倒算口径，必须写“官方披露 -> 建模拆分”的桥表和公式，例如 `中端 = 主品牌销量 - 中高端以上销量`。倒算项必须标 `推断口径`，不能伪装成公司披露。
- 除主建模拆分外，只保留对主拆分有直接解释力、校验价值或缺口补充价值的辅助拆分。不要为了凑数量写泛泛副拆分；一个深度完整的主拆分方式强于多张浅表辅助拆分。
- 同一指标多源冲突时并列，不取平均。写清哪个是公司口径、哪个是研报口径、哪个是推断口径。
- 利润表毛利率、销售/管理/研发费用率、税率这些财务率：历史保留最近一年、最新季度和关键拐点即可；重点给未来显式期预测候选、驱动和分歧。不要为这些财务率写过长历史表，除非它直接解释业务拆分。

默认范围是 **利润表 + 业务层盈利模型**：

- 可以处理：收入、业务线拆分、毛利/成本、销售费用率、管理费用率、研发费用率、税金及附加、below-OP 项目、所得税率、少数股东。
- 默认不主动预测：`financial expense`、`EBIT`、`DA`、`CAPEX`、`CWC`、`shares`、`WACC` 等 BS/CF/DCF 驱动。
- 财务费用例外：只有资料明确给出利润表外生“其他财务费用”时，才可作为 `other_fin_exp_abs` 候选；由现金、债务、利率、BS 推导的 financial expense 不写成预测旋钮。
- BS/CF 硬边界：所有库存、应收、合同负债、资本开支、折旧、营运资本、WACC、股本等线索一律放入收纳区，标注“可能的 BS/CF thesis，待人工判断是否走 /da 或专门流程”。你不得自行判断它们是核心 thesis，不得给逐年预测候选，不得写进 `knobs`。

## 3. 输出抬头

用户确认会议 memo 后，请只输出一份 Markdown，标题为：

```markdown
核心假设参考alphapai_yymmdd.md
```

抬头必须包含：

```text
模式: alphapai-load
状态: reference
公司: {公司}
ticker: {ticker}
资料截至: {你检索到的最新资料日期}
历史事实至: {最新已披露年度/季度}
建议显式预测期: [{YYYY}, {YYYY}, ...]
建议衰减期至: {YYYY / 待裁决}
永续增长候选: {x% / 待裁决}
用途: 供后续裁决，不可直接编译
```

时间轴四数是参考候选，不是拍板结果。你必须给出：

1. 历史事实至哪一年/季度。
2. 建议显式期从哪年到哪年。
3. 衰减期建议到哪年或写“待裁决”。
4. 永续增长候选或写“待裁决”。

显式期不能机械默认 5 年；必须覆盖资料中已经披露或高置信的拐点年份，例如产能投产、合同到期、税收优惠到期、补贴退坡、价格周期拐点、门店/用户增长拐点。无法判断时写“待裁决”，不要硬平推。

## 4. 正文结构

用户确认后输出完整参考稿时，请严格按以下顺序输出。

### 4.1 一页总览

总览只是导读，不能替代后面 4.4-4.9 的结构化候选。即使总览已经说清楚，也必须继续按后文模板逐段输出。

用 5-10 条 bullet 写清楚：

- 公司怎么赚钱。
- 核心业务线是什么。
- 最近 1-2 年最大变化是什么。
- 未来 3-5 年最关键收入驱动是什么。
- 可用于 /ka 的主业务拆分是什么，最近 5 年是否完整；若有高价值辅助拆分，说明它如何校验或解释主拆分。
- 毛利率/费用率核心变量是什么。
- 数据库材料里的最大分歧是什么。
- 哪些判断最适合交给后续裁决。
- 时间轴四数候选是什么，以及覆盖了哪些已知拐点。

| 材料 | 日期 | 类型 | 用途 | 可信度 | 时效性 |
|---|---:|---|---|---|---|

### 4.3 业务理解

写清主营业务、商业模式、产品/服务、客户、渠道、地区、价格、销量、产能、订单、行业位置、竞争格局、近年结构变化、管理层关键表述。

### 4.4 收入拆分候选

先输出 **主建模拆分历史总表**，再逐条写业务线。

若用户贴入了 `Alphapai业务拆分抓取器` factpack，本节必须优先使用 factpack 的 `用户指定主拆分` 或定向 leaf 表作为主建模拆分。不得因为官方披露更稳健，就把用户主拆分替换成官方口径；官方口径只能放到桥表、来源与裁决或 sanity check。若 factpack 来自定向取数模式，本节行轴必须保持用户给出的 `部` / `leaf` 层级。

主建模拆分历史总表必须优先服务 `/ka` 建模，不能只写最近一年 bullet。最近 5 年必须尽力完整；更早年份不要求齐全，有意义、有参考价值再收。建议格式：

```markdown
#### 主建模拆分历史总表
| 年份/期间 | 建模线 | 官方/推断口径 | 收入 | 收入yoy | 销量/件数 | 销量yoy | ASP/价格 | ASPyoy | 占比 | 来源 | 可信度 | 备注 |
|---|---|---|---:|---:|---:|---:|---:|---:|---:|---|---|---|
```

如果建模线来自倒算或映射，紧接着写桥表：

```markdown
#### 官方披露到建模线的桥
| 年份 | 官方披露A | 官方披露B | 建模线计算 | 公式 | 是否可回勾总量 | 来源 |
|---|---:|---:|---:|---|---|---|
```

如果 factpack 已给桥表，本稿必须搬运桥表的核心年份、公式、可回勾结论、可信度和缺口；可以压缩排版，但不能删除“推断口径 / 待校准 / 未披露”的警示。

然后只在有直接增量价值时给辅助拆分历史表。辅助拆分用于 /ka 解释或校验主拆分，不要求进入 knobs；不得为了凑数量写浅表副拆分。建议标题示例：

```markdown
#### 副拆分A：地区收入历史
#### 副拆分B：渠道/客户/门店历史
#### 副拆分C：产品号/价格带/品牌历史
```

每张辅助拆分表都要有年份/期间、口径、数值、同比或占比、来源、可信度；拿不到完整历史时写“只取得 X 年，待补 Y 年”，不要用一句话带过。没有高价值辅助拆分时，明确写“不强行展开泛泛副拆分”。

每条业务线按模板写：

```markdown
### {业务线} [上挂: 营业收入; compiler: factor_product/growth/abs/formula; status: reference]
- 这条线是什么:
- 历史事实:
  - headline: 最近一年和最长可得历史区间的关键变化
  - 业务原子: 引用上方主拆分表中的收入、销量/件数、ASP、占比；若为推断口径，写清桥表公式
  - 单位:
  - 来源:
- 参数化候选:
  - 推荐 family:
  - 旋钮:
  - 派生:
- 预测候选:
  - {YYYY}: ...
  - {YYYY}: ...
- 三件套:
  - 谁定:
  - 为什么:
  - 来源:
- 来源与裁决:
- 风险/缺口:
```

可用 compiler family：

- `factor_product`：量 x 价、门店 x 单店、用户 x ARPU、产能 x 利用率 x 价格等。
- `growth`：收入增速。
- `abs`：收入绝对值。
- `formula`：只有资料明确存在跨期、滞后或复杂公式，且普通 family 装不下时才建议；必须标 `待回测验证`。

优先使用最简单可解释的 family。不要为了显得精细而强行 formula。

### 4.5 毛利/成本候选

写历史趋势、产品结构、价格、原材料、产能利用率、汇率、规模效应。说明建议整体毛利率还是分业务线毛利率。

本节重点是未来预测候选。历史不要机械写 5 年长表；保留最近一年、最新季度、关键拐点和与业务拆分相关的成本事实即可，因为标准毛利率历史后续可由 `/init` 提供。

```markdown
### 毛利率 [上挂: 营业成本; compiler: gpm / leaf_margin; status: reference]
- 历史:
- 驱动因素:
- 预测候选:
- 三件套:
- 风险/缺口:
```

分线毛利率可以写在正文作为候选，但末尾 `knobs` 块默认先不要写 `leaf_margin`，除非用户明确说后续系统支持该项。为了兼容后续工具，`knobs` 优先写整体 `gpm` 或收入/费用/税等明确旋钮。

### 4.6 费用率候选

分别写：

```markdown
### 销售费用率 [上挂: 销售费用; compiler: cost_rate; status: reference]
### 管理费用率 [上挂: 管理费用; compiler: cost_rate; status: reference]
### 研发费用率 [上挂: 研发费用; compiler: cost_rate; status: reference]
### 税金及附加率 [上挂: 税金及附加; compiler: cost_rate; status: reference]
```

每项写历史费用率、变化原因、经营杠杆或投入周期、预测候选和证据。历史只需最近一年、最新季度和关键拐点；重点写显式期预测候选，不要把篇幅花在 `/init` 已能提供的长期财务率历史上。

### 4.7 below-OP、税率、少数股东候选

覆盖以下项目；没材料就写“未检索到明确判断，建议后续沿用默认/年报口径”：

- 资产减值损失
- 信用减值损失
- 其他收益
- 投资净收益
- 公允价值变动净收益
- 资产处置收益
- 营业外收入
- 营业外支出
- 有效税率
- 少数股东损益率
- 其他财务费用外生项，仅限明确利润表外生项

绝对值单位用“百万元人民币”。收益为正，损失/支出为负，符号口径必须说明。

### 4.8 BS/CF/DCF 线索收纳

本节只收纳，不预测，不给逐年候选，不进 `knobs`。

凡是库存、应收、应付、合同负债、资本开支、折旧、营运资本、FCFF、WACC、股本、回购、分红、DA/CAPEX 排程等信息，都写在这里：

```markdown
### {线索}
```

## 5. knobs 机器自报清单

文末必须放一个 fenced block，语言标签必须是 `knobs`。它是正文预测旋钮的机器可读回声，不是 YAML1。

只写正文里明确给出逐年候选值的“预测输入旋钮”。没有足够逐年数值时，写 `knobs: []` 并说明原因。

### 5.1 顶层结构

下面只是格式示例，不得照抄年份和值。实际 `horizon` 必须使用第 3 节给出的建议显式预测期；如果无法形成逐年候选，就写 `knobs: []`。

```knobs
horizon: [YYYY1, YYYY2, YYYY3]
terminal:
  explicit_end: YYYY3
  fade:
    to_year: null
    kind: 待裁决
  perpetual_growth: null
reason: "Alphapai-load reference 稿，terminal 待裁决"
knobs:
  - anchor: "#整体毛利率"
    family: gpm
    unit: pct
    values: [29.2, 29.9, 30.5]
    status: reference
    source: "正文同源: Alphapai 数据库参考"
```

### 5.2 单条 knob 字段

每条必须包含：

- `anchor`：正文小节锚点，必须用引号包住，例如 `"#低温鲜奶"`。
- `sub`：同一小节多个旋钮时必须写，例如 `销量`、`吨价`、`收入`。
- `family`：语义族名。
- `unit`：只允许 `pct`、`ratio`、`abs_mn`。
- `values`：逐年数组，长度必须等于 `horizon`。
- `status`：本稿统一写 `reference`，估算项可写 `estimated·待校准`。
- `source`：简短说明来源。

### 5.3 单位

- `unit: pct`：写百分数显示值。例如 15.4% 写 `15.4`，不要写 `0.154`。
- `unit: ratio`：写小数或倍率原值。例如 30 天写 `30`，2.5% 若作为 ratio 写 `0.025`。
- `unit: abs_mn`：写百万元人民币金额。例如 -70 百万元写 `-70`。

平推或“维持不变”也必须展开逐年：

```yaml
values: [15.4, 15.4, 15.4, 15.4, 15.4]
```

不能写：

```yaml
values: "全程 15.4%"
```

### 5.4 推荐 family

`family` 在这里是语义标签，不是 YAML1 path，也不是让你扩展模型能力的许可。为降低误写，只允许以下保守 family。其他情况写正文或收纳区，不进 `knobs`。

| family | 适用 | unit |
|---|---|---|
| `factor_yoy` | 销量、价格、门店数、ARPU 等因子增速 | `pct` |
| `growth` | 收入增速 | `pct` |
| `abs` | 收入或 below-OP 科目绝对值 | `abs_mn` |
| `gpm` | 整体毛利率 | `pct` |
| `cost_rate` | 销售/管理/研发/税金及附加费用率 | `pct` |
| `tax_rate` | 有效税率 | `pct` |
| `minor_rate` | 少数股东损益率 | `pct` |
| `other_fin_exp_abs` | 其他财务费用外生项 | `abs_mn` |

不要在 `knobs` 块里写：

- 历史值。
- 派生预测值，例如收入金额、毛利额、净利润、净利率、EBIT。
- YAML1 path、defaults path、base、history、完整拆分树。
- WACC、股本、目标价。
- BS/CF/DCF 线索，包括 capex、折旧、营运资本、库存、应收、合同负债、FCFF。
- `leaf_margin` 或复杂 formula input，除非用户明确说后续系统支持。
- terminal fade 期逐年序列。
