# 核心假设源语言与编译器技能指南

> 面向后来者:本指南帮助你理解 `核心假设源语言`、`核心假设编辑器`、`核心假设调整器` 和 `yaml1compiler` 的定位、分工、关键纪律,以及它们在整个管道中的位置。旧时代的"核心假设生成修改器 init/modify 一体化"已经拆分：`/load` 读 Excel，`/brkd` 读研报纪要，`/ka` 只做全量裁决，`/adj` 和 `/frontend-edit` 负责已有正式稿调整，`/comp` 负责编译。

---

## 一、你在哪条管道上

整条链由四个阶段组成,每个阶段只干一件事:

```
投研材料(Excel/年报/研报/增量信息)
    │
    ├──[/load]──> {原Excel文件名}_核心假设.md        (旧模型假设源文·load-vintage)
    ├──[/brkd]──> Agent业务讨论.md                  (业务预讨论)
    ├──[/ka]────> 核心假设.md                       (正式人话判断源头)
    ├──[/adj / /frontend-edit]──> 新版核心假设.md    (已有正式稿调整)
    │                           │
    │              [/comp / yaml1compiler·LLM]──> yaml1  (机器话·按路径组织·忠实覆盖)
    │                           │
    │                   [yaml1_cleaner.py·Python]──> 标准参数表 ⊕ defaults.yaml
    │                           │
    │                           [forecast.py + calc.py·Python]──> 三表 + DCF
    │
    └──[前端] 读 yaml1 中不进 DCF 的部分(decomposition、history、stash)
```

**你在这条链上负责语义层或编译层时，要先分清角色。** `/ka`、`/load`、`/brkd`、`/adj` 都不产 YAML；`/comp` 不做业务判断、不重算人话假设。各自的分工边界是死的。

## 二、核心技能分工(谁负责什么)

| 角色 | 输入 | 产出 | 核心动作 | 不做什么 |
|---|---|---|---|---|
| `/load` | 唯一 Excel 模型 + load 时间沙箱 | `{原Excel文件名}_核心假设.md` | 还原旧模型当时的预测假设 | 不生成当前正式判断 |
| `/brkd` | 研报/纪要/年报 + /init 历史事实 | `Agent业务讨论.md` | 整理业务线索和利润表讨论点 | 不拍最终旋钮 |
| `/ka` | 最高权重材料 + BRKD + LOAD + /init 校验层 | 正式 `核心假设.md` | 全量裁决、时间轴和骨架门、数值门 | 不读原始 Excel/研报/纪要，不做 modify |
| `/adj` / `/frontend-edit` | 已有正式核心假设 + yaml1 + 小改或增量材料 | 新版 `核心假设.md` / 新 forecast | 已有正式稿调整 | 不从零生成、不绕过核心假设源头 |
| `/comp` | `核心假设.md` + defaults.yaml + 数据格式参考.md + yaml1算法模板契约.md | `yaml1_公司名_YYYYMMDD.yaml` + forecast | 忠实翻译、举旗、运行 DCF | 不算新判断、不替原文改数 |

**最关键的认知:这是一次翻译。** compiler 不是录入表,不是参数配置器,而是把两种完全不同的语言(人话→机器话)逐条翻译。理解"两种语言为什么长得不一样",比记任何具体规则都重要。

## 三、核心假设源语言:三条灵魂纪律

无论是 `/ka`、`/load`、`/adj incremental` 还是 `/annual-update`，只要产出或修改核心假设源文，都必须遵守下面三句话:

### 1. 历史保全

- **只搬绝对值原子 + 收入 headline**,不搬可推导的比率(yoy/毛利率/占比)。
- **行清单随线随公司变**,绝不 hardcode。骨架声明什么原子就存什么。
- 历史是证据,不是装饰。哪怕 cleaner 暂时不吃,也要给它归宿(进 leaf `history` 或 `stash`)。
- 加总校验只有两个合法结局:ok 或显式差额说明。**禁止倒算残差配平,禁止为配平改历史基年原子。**

### 2. 旋钮识别

- 预测层不是把未来格子全搬出来,而是识别**哪些是人拍的输入、哪些是公式派生**。
- 拍了销量和吨价,就不要再把收入当输入;拍了分线 margin,就不要再拍整体 gpm。
- **自洽铁律**:一个指标要么是旋钮、要么是派生,不能既当旋钮又当派生。

### 3. 算法抽象

- Excel 的灵魂是公式关系,不是数字。识别它是 sum、n 因子连乘、增速、绝对值、还是模板装不下的滞后链/DAG。
- 能落到有限模板就清楚写出模板和原子;落不进去的举手,不得发明族名。
- 当前 compiler 支持的可执行模板:`factor_product` / `growth` / `abs` / `vol_price`(兼容) / leaf margin → `income.gpm`。

### 关键产出物

- `公司名-日期-核心假设.md`(正式底稿)或 `公司名-日期-核心假设参考.md`(未聊透)。
- `/load` 的 `{原Excel文件名}_核心假设.md`(状态为 `model-extracted` 的旧模型假设源文,供 `/ka` 裁决或沙箱 DCF 使用)。
- 底稿末尾的 **```knobs 机器自报清单**。详见下文§六。

## 四、编译器:一次翻译要弥合的四个语法差异

| # | 差异 | 源(`.md`) | 目标(`yaml1`) | 性质 |
|---|------|-----------|---------------|------|
| 1 | **组织维度** | 按业务线分段 | 按路径命名空间 | 形变·照翻 |
| 2 | **深度语义** | 拆到"判断的深度" | 镜像成 ≤2 拆分树 | 形变·照翻 |
| 3 | **符号口径** | 按"对利润的正负贡献"显示 | 按引擎公式做加项/减项存值 | 形变·照翻 |
| 4 | **歧义容忍** | 宽容(省略、口径注、措辞矛盾) | 零容忍(摊满、明确、自洽) | **信息缺口·举旗** |

**第一守则:形变照翻,歧义举旗,绝不猜。**

能机械转换的(组织/深度/符号)直接翻;遇到歧义(信息缺口)停下举旗——标 `# 路径待核`、`# 语义待核`、列入 `unaligned` 清单、在收尾报告写明歧义+你取的自洽解,交人裁定。**举旗不是失败,是这条接缝上唯一诚实的动作。**

## 五、编译器的三种 `kind` 积木与 leaf 三件套

### 5.1 三种积木

| kind | 用途 | 典型路径 |
|---|---|---|
| `knob` | 旋钮(费用率/绝对值/毛利率/税率/少数股东率) | `income.cost_rates.sell_exp`、`income.gpm` |
| `decomposition` | 非标拆分(收入按业务线拆) | `income.revenue` 下挂 `segments` |
| `formula` | 受限算法(跨期递推/DAG/分段),只接长尾 | `formulas.nodes` + `formula_ref` |

**`formula` 不是默认选择。** 绝大多数收入线先试模板(`factor_product`/`growth`/`abs`),装不下才进 formula。不要为能模板表达的线升级 formula。

### 5.2 leaf 三件套(每条 decomposition leaf 必须齐)

```yaml
base:      # 起点:单年锚(base_year + 基年原子 + unit_factor_to_million_cny)
knobs:     # 未来:逐年满数组(旋钮)
history:   # 过去:完整历史序列(逐年照搬,占位/异常年标 note)
```

- `base` 只取折叠用的单年锚;**完整多年历史落在同一个 leaf 的 `history.series`** 下。
- `unit_factor_to_million_cny` 必须结构化输出,按族给,不全局拍一个。
- 历史与 base/knobs **共用同一个 segment slug**,这是后续 `/adj`、`/frontend-edit` 和前端取数不漂移的前提。

### 5.3 符号铁律(附录A速查表是唯一权威源)

| .md 科目 | 存值符号 | 为什么 |
|---|---|---|
| `cost_abs.*`(资产减值/信用减值) | **存负值**,与 .md 同号 | 引擎按同号做减项 |
| `operating_adjustments_abs.*`(资产处置/其他收益) | 亏损存**负**,与 .md 同号 | 引擎做加项 |
| `below_line_abs.non_oper_exp`(营业外支出) | 存**正值**(常与 .md 负号相反) | 引擎公式里已是减项 |
| `cost_rates.*` / `gpm` / 税率 | 正值比率 | |

**减值符号已固化,不要再标"符号待核"。** 其余未固化项,逐路径核对引擎怎么用这个值再定符号。

## 六、knobs 块:上游无损化

### 6.1 为什么存在

编译器 Gate C 需要把 yaml1 的旋钮值与生成器的意图做对值双射。老办法是用正则读人话 `.md`,但 .md 是**自然语言**——同一个数字有无数种写法(加粗、全角负号、措辞漂移),正则脆性高(误报/漏报)。

**根治:让生成器在写人话的同时,顺手再吐一份机器可读清单。** 人话不动,多一个 fenced 块。

### 6.2 设计原则(通用性)

- **不用 yaml1 路径做键**:生成器看不到 defaults、不产 yaml1 路径。键用生成器自己的**上挂科目锚点**(=`#销售费用`、`#整体毛利率` 等),而 yaml1 的 `src` 字段就是这个锚点。
- **完全由块内容驱动**:不含任何公司特定假设。`anchor`/`sub`/`family`/`unit`/`values` 都是声明式,换任何公司照填。
- **值用生成器的显示惯例**:% 写百分数、金额写百万。验证器做单位归一后比对。

### 6.3 格式

````markdown
```knobs
horizon: [2025, 2026, 2027, 2028, 2029]
knobs:
  - {anchor: "#整体毛利率", family: gpm,       unit: pct,    values: [29.2,29.9,30.5,31.1,31.6], override: true}
  - {anchor: "#销售费用",   family: cost_rate, unit: pct,    values: [15.4,15.4,15.4,15.4,15.4]}
  - {anchor: "#资产处置收益", family: op_adj_abs, unit: abs_mn, values: [-70,-30,-30,-30,-30]}
  - {anchor: "#低温鲜奶", sub: 销量, family: factor_yoy, unit: pct, values: [7,6,6,6,6]}
  - {anchor: "#边缘业务", sub: 收入, family: growth,     unit: pct, values: [0,-10,-10,-10,-10]}
```
````

- `anchor`:同正文 `###` 小节标题的核心词,归一后须等于 yaml1 的 `src`。
- `sub`:区分一节多旋钮(销量/吨价/收入)。
- `family`:同生成器已标的 `compiler:` 提示,让验证器多一层家族核对。
- `unit`: `pct`(验证器 /100) / `ratio` / `abs_mn`。
- `override: true`:主动覆盖项(值故意偏离外部模型),验证器据此不去模型溯源。
- **值用满数组**:生成器把"全程/平推"自己展开成逐年。

**这不是 YAML1。** 不写路径、不导出拆分结构/base/history、不做会计化对齐——那些仍是 compiler 的活。

## 七、忠实度校验器:三道闸 + 双模式

`src/yaml1_fidelity_check.py` 是编译后的必跑闸(已接进 `/comp` skill)。

### 7.1 它补的洞

下游 `yaml1_cleaner.py` 把守 **yaml1↔历史现实**(fold 回测、历史符号对账)和 **yaml1↔自洽**,但**从不读 `核心假设.md`**。所以预测旋钮抄错、漏一条旋钮、符号反、路径映射错——全部静默折进 DCF。

校验器只填这条缝(`yaml1↔.md 预测意图`),不重造下游 fold/回测。

### 7.2 三道闸

| 闸 | 比对 | 抓什么 | 杠杆 |
|---|---|---|---|
| **A 结构** | yaml1 单独 | 数组长度≠horizon、深度>2、rollup/leaf 混用、family 非法、margin 二选一违反 | 算法契约代码化 |
| **B 路径+符号** | yaml1 vs `defaults.yaml` | 路径不存在(发明/嵌套错,下游静默丢弃)、符号与基年相反、费率越界 | defaults 当符号神谕 |
| **C 值双射** | yaml1 vs `.md` / knobs 块 | 抄错值、小数点滑位、漏译、符号反 | `src` 锚点 / knobs 块 |

### 7.3 双模式(自动路由)

- **block-diff(无损,优先)**:`.md` 含 ```knobs 块时,校验器抓出块、与 yaml1 按 `src[+sub]` 逐条结构 diff。**真双射**:yaml1 在块里找不到=幻觉;块条目在 yaml1 里找不到=漏译。报错精确到**年索引+两边值**。零人话解析、零脆性。
- **regex(回退)**:无块时,按 src 定位 .md 小节 + 符号量级集合核对。能工作,但脆性高(格式漂移会误报/漏报)。

报告里的 `gate_c_mode` 字段标明走了哪条。

### 7.4 明确不做的

- **不做语义级判断**:该不该有这条旋钮、判断是否成立、主动覆盖理由是否合理——Python 解决不了,是 compiler §9 人话回读和分析师的活。
- regex 模式下是集合核对,不是逐年顺序核对(抓不了同小节纯顺序交换)。

## 八、实战:从 `/ka` 到 `/comp` 的完整链路

```
# 1. 生成核心假设.md(分析师判断层)
/ka 新乳业
  → 核心假设源语言 + 核心假设编辑器加载
  → 读最高权重材料 + BRKD 产物 + LOAD 产物 + /init 校验层
  → 逐块开会(收入→毛利→费用→below-OP→三段式)
  → 产出: 新乳业-20260616-核心假设.md (含末尾 ```knobs 块)

# 2. 编译 yaml1(翻译层)
/comp 新乳业
  → compiler skill 加载
  → 读四份输入(.md + defaults.yaml + 数据格式参考.md + 算法契约.md)
  → 按翻译规则编译
  → 产出: yaml1_新乳业_20260616.yaml

# 3. 忠实度校验(翻译质量闸)
  → yaml1_fidelity_check.py 自动运行
  → 有 knobs 块 → block-diff
  → verdict PASS / BLOCK
  → BLOCK 则修正 yaml1 后重跑校验,直到 PASS

# 4. 下游清洗+计算(已在你视线之外)
  → yaml1_cleaner.py 折叠分解→收入总额、展开 fade→逐年
  → resolve ⊕ yaml2 defaults
  → calc.py 三表 + DCF
```

## 九、常见陷阱速查

| 陷阱 | 在哪层 | 为什么错 | 正确做法 |
|---|---|---|---|
| 生成器产了 YAML | 生成器 | 越界,它不产机器话 | 只写 markdown;YAML 归 compiler |
| 编译器重算 .md 的数 | 编译器 | 翻译器不算账 | 觉得错 → 举旗,不替它改 |
| 利息净额写了 knob | 编译器 | 引擎倒算,写了会 `#VALUE!` | 利息净额不写;其他财务费用外生项必写且嵌 `income.financial_expense` |
| 财务费用提到顶层 | 编译器 | `cleaner` 用 `income.get("financial_expense")` 取,顶层会被静默丢弃 | 必须嵌在 `income.financial_expense` 下 |
| 减值存了正 | 编译器 | 与附录A固化结论相反 | `cost_abs.*` 存**负值**,与 .md 同号 |
| 分线历史塞进 stash 平行块 | 编译器 | slug 脱钩,后续 `/adj` / 前端编辑漂移 | 历史挂该 leaf 的 `history` |
| 收纳项塞进 A 类 note | 编译器 | 前端溯源捞不到,B 类完整性失败 | 收纳区独立成块进 `stash` |
| margin 二选一违反 | 编译器 | 部分 leaf margin + 顶层 gpm = over-determined | 有任一 leaf margin → 禁止顶层 gpm;全无 → 才允许顶层 |
| 发明路径/族名 | 编译器 | 下游不认,静默错 | 路径以 defaults.yaml 为准;族名以算法契约为准;装不下就举旗 |
| 生成器用 ~ 写旋钮 | 生成器 | 不可机读,翻译器续不了 | 精确逐年值,flat 也要写值+年份 |
| 生成器把派生量当旋钮填 | 生成器 | over-determined | 旋钮内部不冲突,派生量能唯一算出 |

## 十、一句话

**生成器把公司盈利模型从 Excel 牢笼里解放成人话底稿;编译器把人话底稿忠实地翻译成机器话 yaml1。生成器负责"判断对、历史全、算法清";编译器负责"形变照翻、歧义举旗、绝不猜"。校验器守的是两者之间最窄的缝——预测意图有没有被忠实搬运。所有设计都服从一个更高原则:兼容任意公司,不写死任何一家的形状。**
