# MKA 数据流水线

这份文档是给接手本项目的大模型看的。请把它当成一张地图：你不需要先记住每个函数的细节，但必须先知道数据从哪里来、在哪里被校验、哪些 YAML 是人维护的、哪些是机器生成的、`financial_expense.yaml` 为什么不能遗漏，以及最终 DCF 和完整三表是怎样落盘的。

## 一句话总览

MKA 的主线不是“直接填假设跑 DCF”，而是：

```text
TuShare 原始三表
-> SQLite raw_tushare
-> clean.py 生成可信历史 clean_annual / clean_quarterly
-> core_metrics_overview.py 从 clean_annual 生成 Agent/core_metrics_overview.*
-> financial_expense_analyzer.py 从年报附注生成 financial_expense.yaml
-> defaults_gen.py 从 clean_annual + meta + financial_expense.yaml 生成 defaults.yaml
-> yaml1_cleaner.py 合并 defaults.yaml + yaml1*.yaml，生成 Agent/.modelking/forecast_params.yaml
-> calc.py 生成预测三表 + DCF
-> forecast.py 拼接历史与预测，生成 full_is / full_bs / full_cf
```

最重要的分工是：

- `clean.py` 负责历史数据可信。
- `core_metrics_overview.*` 是 clean 后的年度事实速览，给后续 Agent/LLM 读，不进入 forecast。
- `Agent/OfficialBreakdowns/` 是 `business_breakdown_extractor.py` 从年报"收入和成本分析"抽的官方拆分素材（行业/产品/地区/销售模式四维营收+成本+毛利率，外加产销量、成本构成两张表），给 `/ka`、`/brkd` 当素材，不进入 forecast。
- `financial_expense.yaml` 负责把财务费用从一个净额拆成利息支出、利息收入、其他财务费用。
- `defaults.yaml` 是机器平推底座。
- `yaml1*.yaml` 是人工判断覆盖层。
- `forecast_params.yaml` 是编译后的逐年标准参数表，只给计算引擎吃。
- `calc.py` 只做会计推演和估值，不直接理解 yaml1。
- `forecast.py` 是正式用户入口，负责把整个过程串起来。

## 文件性质：哪些能改，哪些不能手改

后续大模型接手时，先按文件性质判断动作，不要看到 YAML 或 CSV 就直接改。

```text
人工输入：
  companies/{公司名}_{代码}/Agent/yaml1*.yaml

证据输入：
  companies/{公司名}_{代码}/Agent/data.db
  companies/{公司名}_{代码}/公告/年报/
  companies/{公司名}_{代码}/Agent/recon/
  companies/{公司名}_{代码}/Agent/financial_expense.yaml

机器事实速览：
  companies/{公司名}_{代码}/Agent/core_metrics_overview.md
  companies/{公司名}_{代码}/Agent/core_metrics_overview.json
  companies/{公司名}_{代码}/Agent/core_metrics_overview.csv

机器底座：
  companies/{公司名}_{代码}/Agent/defaults.yaml

内部编译产物：
  companies/{公司名}_{代码}/Agent/.modelking/forecast_params.yaml
  companies/{公司名}_{代码}/Agent/.modelking/yaml1_clean_report.json
  companies/{公司名}_{代码}/Agent/.modelking/forecast_build.json

正式输出：
  companies/{公司名}_{代码}/Agent/forecast/
```

操作纪律：

- `yaml1*.yaml` 是人工判断覆盖层，主要由 `/comp` 从核心假设 compiler 生成；只有 `/adj quick` 和 `/frontend-edit` 的 A4 白名单小改可以在同步更新 `核心假设.md` 与 `knobs` 后定点 patch。
- `financial_expense.yaml` 是证据输入，不是预测输出；改财务费用逻辑时必须保证它、`defaults_gen.py`、`calc.py` 三者一致。
- `core_metrics_overview.*` 由 `/init` 在年度 clean 成功后覆盖重生成；它只读 `clean_annual`，不手改、不做预测、不依赖 yaml1/defaults/forecast。
- `defaults.yaml` 是机器生成的 YAML2 底座，通常不手改；如果确实改了生成逻辑，应重跑生成而不是只改单家公司样本。
- `Agent/.modelking/` 下的文件是内部编译产物，只能由程序写，不能手工维护。
- `Agent/forecast/` 是正式结果目录，会被正式运行清空重写；不要把人工底稿放进去。

## 建模前置站：业务预理解与装载（/brkd、/load → /ka）

在取数流水线之上，建模管线有三站：

```text
研报/纪要 → /brkd → Agent业务讨论.md ──┐
外部Excel → /load → KA 参考稿区 核心假设参考load_*.md ─┤→ /ka → 核心假设.md → /comp → yaml1
Alphapai  →        KA 参考稿区 核心假设参考alphapai_*.md ─┘   译准/翻译
```

**主导方向**：`/brkd`、`/load` 的首要任务是忠实抓业务拆分+详细历史+业务数据（即便不参与计算、只进收纳区），预测旋钮是次要 draft 候选；`/ka` 拿这些产物和分析师裁决预测，同时忠实收集与预测有关的历史防丢数。预测是末，业务结构与历史是本。

`/brkd`（业务预理解器）读 `Skills素材包/BRKD业务理解器（研报和纪要放在这里）/` 下的研报纪要，产出 `Agent业务讨论.md`（公司根目录）。`/load`（模型装载器）读 `Skills素材包/LOAD外部EXCEL模型理解器（一次最多一个）/` 的唯一 Excel，产出 KA 参考稿区 `核心假设参考load_{YYYYMMDD}.md`。两者都只交候选理解，不拍正式判断。brkd/load/alphapai 参考稿统一落 `Skills素材包/KA（ALPHAPAI拆出来的东西放在这里）/`，命名 `核心假设参考{来源}_{YYYYMMDD}.md`；`/ka` 到该目录找参考稿，根目录只剩 official `核心假设.md`。`/webka` 可把 `/ka` 所需规则与材料预打包到 `WEBCLAUDE/webka(...)/` 供网页端执行。

## 第一层：取数与原始镜像

入口模块：

```text
src/data_fetcher.py
```

输出位置：

```text
companies/{公司名}_{代码}/Agent/data.db
```

`data_fetcher.py` 从 TuShare 拉取三张表：

- `income`
- `balancesheet`
- `cashflow`

写入 SQLite 的 `raw_tushare` 表。`raw_tushare` 是 EAV 形态：

```text
ticker, endpoint, report_type, end_date, field, value, ...
```

原则是：`raw_tushare` 是原始镜像，后续任何补数、重分类、年报 evidence 都不能回写或污染它。项目里很多设计都是围绕这个原则展开的。

> **2010 闸门的两处口径**：`clean.RECONCILE_MIN_YEAR=2010` 同时约束了「校验」和「下载」两侧，但**口径不同，不要混**。
> - **校验侧（clean.py / reconciler）**：2010 前的年度硬校验失败降级为 warning 直接入库、不触发 reconciler；`raw_tushare` 仍含 2010 前全历史（`data_fetcher.py` 拉全历史），`clean_annual` 仍写入 2010 前年度（仅未核对）。
> - **下载侧（report_downloader.py / init.py stage_reports）**：cninfo 年报/季报 PDF+Markdown **只下载 2010 及以后**（`--min-year` 默认 `DEFAULT_MIN_REPORT_YEAR=2010`，与 `RECONCILE_MIN_YEAR` 对齐）。2010 前的 `公告/年报/` Markdown 不会存在——reconciler / financial_expense 也只消费 2010+ 年报，两侧一致。
> - 即：2010 前年度有 TuShare 数据（clean 入库但不核对），无 cninfo 年报 PDF/Markdown。

> **pre-IPO 闸门（2026-06-25）**：除 2010 闸门外，还有一条按公司动态判定的 IPO 边界——`clean.earliest_annual_md_year(db_path)` 扫描本地 `公告/年报/*_年度报告.md` 取最小年份（= IPO 后首份年报）。上市公司上市前年份的 TuShare 数据来自招股说明书、cninfo 无该年年度报告 MD，reconciler 无 MD 可核对。`clean.validate_wide` 把**早于该年的年度硬校验失败降级为 warning 直接入库**（不阻塞、不触发 reconciler）；`annual_report_reconciler.collect_failures` 也据此跳过 pre-IPO 年度。`init.py stage_clean` 在年报下载完成后重跑一次 clean 让闸门生效，pre-IPO 年由此通过、不再空跑 reconciler。无年报 MD 时闸门关闭（返回 None）。与 2010 闸门同性质：有据可审计，非静默改判。

同一个 `data.db` 还会有 `meta` 表，保存 ticker、公司名、股本、市值、收盘价等元信息。后续 `defaults_gen.py` 会用它生成股本、净债务、市值等估值参数。

## 第二层：clean.py 生成可信历史宽表

入口模块：

```text
src/clean.py
```

输入：

```text
data.db/raw_tushare
data.db/meta
companies/{公司名}_{代码}/Agent/recon/annual_report_overrides.json
```

输出：

```text
data.db/clean_annual
data.db/clean_quarterly
data.db/clean_adjustments
data.db/clean_warnings
```

`clean.py` 做的是“历史三表可信化”，不是预测。

> **会计科目与排序的唯一真源(2026-06-22)**:三表 325 字段的分类/会计序/标签/resolve/sign 都声明在 `src/field_registry.yaml`。`clean.py` 的 `IS/BS/CF_FIELD_CATEGORIES`·`SUB_RESOLVE`·`COMBO_RESOLVE`·`SIGN_QUESTIONABLE` 都 `from .field_registry import`(clean 命名空间仍重导出,下游 `clean.IS_FIELD_CATEGORIES` 等访问不变);`workbench.py` 三表渲染排序/标签、`docs/数据格式参考.md`(`scripts/gen_field_reference.py` 派生)同源。改字段分类/排序只编辑 `field_registry.yaml` 一处。check_* subtotal 公式仍留代码。

它会把 `raw_tushare` 的 EAV 数据 pivot 成宽表。宽表字段保持统一：`period` 加 325 个 TuShare 官方字段，再加 6 个 QA plug 字段。跨端点同名字段会加前缀消歧，比如：

```text
income.credit_impa_loss
cashflow.credit_impa_loss
```

年度表和季度表的处理边界不同。

年度 `clean_annual` 必须通过 hard check：利润表、资产负债表、现金流量表、跨表关系都要配平。若年度 hard check 失败，尤其是合计大于明细和这种 TuShare 明细缺口，正确动作是调用 `annual_report_reconciler.py` 去本地年报 Markdown 里找缺失科目，生成 approved override。若目标年度缺少单独年报，可使用后续年度年报的比较报表列作为证据来源。override 应用后只改变 clean 宽表，不改变 raw。

reconciler 的通用性约束（不为任何单一公司写死）：LLM 确认按单个 hard-check 失败 (period, code) 分片调用，复杂公司（含金融子公司、逐年漏同一明细）失败数可达 30+，单次打包所有候选会 ReadTimeout 丢掉全部年报证据；确认用 `temperature=0` + 重试 + 截断检测，使同一证据的判断可复现；年报 snippet 窗口须覆盖整张合并报表（长报表的应收/融资行常在标记 250+ 行之后）；当某年报科目名精确等于残差时优先于"多字段凑和"的投机 group；`target_lt_calc`（明细和 > 合计）对应 TuShare 漏录的负值项（如终止经营净亏损），按 −residual 匹配；短别名是更具体科目名子串时（"应收款项"⊂"应收款项融资"）丢弃以免干扰 LLM。自动核对的失败上限默认须覆盖完整年报历史（≥60），否则复杂公司会"靠前几年配平、后几年悄悄漏"。已确认的 TuShare 字段缺口（`lending_funds` 发放贷款和垫款、`estimated_liab` 预计负债、`receiv_financing` 应收款项融资、`end_net_profit` 终止经营净利润等）以触发条件+字段记录在 `knowledge/known_tushare_defects.json`，只作 LLM 检索提示，不作自动补数依据。这些分片彼此独立，因此通过 `annual_report_utils.parallel_map`（有界、保序、异常透传，`LLM_MAX_WORKERS` 默认 6）并发确认：墙钟从"逐次相加"压成"最慢一次"，但每次调用仍各自保留超时/重试/`chunk_errors` 分片审计，合并结果按分片原序确定性输出——并发只改速度，不改判定与审计。同一并发工具也用于 `financial_expense_analyzer` 的逐年附注分析（各年互不依赖）。

**`/init` 阶段编排（2026-06-25）**：`init.py` 的 5 阶段不再全串行。② 年报/季报下载丢后台线程，与 ③ 首轮 clean **并行**——首轮 clean 只读 `raw_tushare`（不读年报），纯 TuShare 配平的公司秒级通过 ③ 不等 ② 下载完；仅当首轮年度硬失败（reconciler 需要年报 Markdown）时才 `join` ② 再显式调 `auto_reconcile_annual_failure` 触发核对。`_run_clean` 由 `python -m src.clean` subprocess 改同进程 `clean_all()` 直调，两轮 backfill 的多次 clean 重跑省掉每次冷启动 + 重复 import；reconciler 仍 subprocess（stderr 流式回显保留）。`clean.py` 配套把 `all_errors` 附加到 `CheckError.errors`、`clean_all` wrapper 传播，供 init 结构化取硬失败（CLI `python -m src.clean` 行为不变）。④ 核心速览 / ⑤ 财务费用仍在 ② join 后按原序跑（⑤ 读年报 Markdown）。数据契约（clean_annual/quartely schema、输出路径、override/plug/exemption 语义）均未变。

**年度核对是混合两层（2026-06-18）**：reconciler 是**无人值守地板**——工作台重算/cron/批量 `init` 都只靠它（rule + GLM fallback 两轮），关简单/精确案例，关不掉就 exit 3。但 reconciler 的 GLM 对某些案例结构性赢不了（喂 jumbled snippet、拿 compound 残差单字段打分闭合不了、发字段名会飘如 `lease_ncl`/null 被守卫挡）。这类残差由 **init skill 的 subagent 升级通道**处理（仅 agent 在线、`init` exit 3 时触发）：`src/recon_subagent_bridge.py` 两个 CLI——`context` 重跑 `clean --mode annual --no-auto-reconcile` 解析 `HARD CHECK FAIL` 取**净残差**（已扣已批准 override，绕开 reconciler 内部 collect_failures 重分类未反映的 bug），为每个失败算 candidate 字段/年报 section/已批准 override 上下文写 `Agent/recon/subagent_context.json`；agent 派并发 subagent 各读年报干净 Markdown 找证据、返回结构化提案合并到 `Agent/recon/subagent_proposals.json`；`apply` **服务端按提案集合净影响验闭合**（`evaluate_proposals`：字段必须在 candidate 集、add_override/reclass 的 calc 有符号影响、`|净影响−所需|<TOLERANCE` 才整组批准），写 `source=claude` 的 approved override 合并进 `annual_report_overrides.json`（`clean.py` `APPROVED_OVERRIDE_SOURCES` 已含 `claude`），重跑 clean 验证。纪律：subagent 只读只提案不写文件不批准自己，防脏配平闸门在代码不信 subagent 自报，raw_tushare 不动，找不到证据诚实留 exit 3。详细 7 步流程见 `.claude/skills/init/SKILL.md`「退出码 3 的 subagent 升级通道」。reconciler 地板侧只做便宜加固：`COMMON_ANNUAL_ALIASES` 补 lease_liab/estimated_liab/lt_borr 等常见漏报科目别名；`resolve_candidate_field` 用科目名反向确定性映射 candidate 字段（挡 LLM 字段名飘）。

季度 `clean_quarterly` 因为披露不完整，允许显式 QA plug 吸收部分残差。QA plug 是审计字段，不是静默补数：

```text
qa_bs_current_asset_plug
qa_bs_noncurrent_asset_plug
qa_bs_current_liab_plug
qa_bs_noncurrent_liab_plug
qa_bs_equity_plug
qa_cf_cash_reconcile_plug
```

plug 使用必须写入 `clean_warnings`。

**跨表 7.4 重述豁免（2026-06-18，与 BS override 通道并行，不补数）**：跨表 7.4（上期 CF 期末 = 本期 CF 期初）失败的根因是**年报重述**——公司在新一年年报比较列里追溯重述上年期末现金，TuShare 存的却是各年原始披露值，边界不衔接。override 闭合对重述结构性不可行：override 一侧会破坏该侧 CF 5.5（期末=期初+净增加），改净增加又级联到 CF 5.4/5.1-5.3；多年连续重述（每年报都在再追溯）要彻底闭合得整体重载被重述年份的整张现金流量表，fragile 且破坏 TuShare 口径。重述是公司披露的会计事件、**非数据错误**，故走第三条路——**证据化豁免 → clean 降级软 warning**（与 2010 闸门同性质：有据、可审计，非静默改判）。流程：`recon_subagent_bridge context` 把 `跨表 7.4` 失败路由到 `build_restatement_context`（解析 prev_end/cur_beg/残差/方向 + 定位本期年报合并现金流量表期初/期末现金行号）；agent 派 subagent 读本期年报确认重述（抽本期期初现金 + 上年比较列期末现金 + 证据行号）合并到 `Agent/recon/subagent_restatement_proposals.json`；`apply-restatements` 用 `evaluate_restatement_proposal` 过 **6 道确定性闸门**（confirmed / 引用行号真实出现元金额反幻觉 / 年报本期期初==上年比较列期末自洽 / ==TuShare 本期期初 / ≠TuShare 上年期末确属重述 / 残差吻合），写 `Agent/recon/restatement_exemptions.json`（`source=claude`）。`clean.py` `load_restatement_exemptions` 加载后，`validate_wide` 的 7.4 检查对豁免边界降级为 `clean_warnings`（带审计，残差需与豁免记录吻合——TuShare 值变动后旧豁免自动失效防脏豁免），`--no-restatement-exemptions` 可关。**关键边界**：豁免只降级 7.4 这类经年报确认的重述边界，绝不掩盖 BS/IS/CF 内部配平失败——那些仍走 override 通道或诚实留 exit 3。潍柴动力 000338 实测：2021/2022 两条 7.4 经此通道豁免，年度 10 期+季度 48 期全过。详细流程见 `.claude/skills/init/SKILL.md`「跨表 7.4 重述豁免通道」。

## 第三层：financial_expense.yaml 的位置与作用

这是最容易被遗漏的一层，必须单独记住。

入口模块：

```text
src/financial_expense_analyzer.py
```

输入：

```text
companies/{公司名}_{代码}/公告/年报/{年份}_年度报告.md
data.db/clean_annual
```

输出：

```text
companies/{公司名}_{代码}/Agent/financial_expense.yaml
```

这一步发生在 `clean.py` 之后、`defaults_gen.py` 之前。它不是 DCF 输出，也不是 yaml1 覆盖层。它是从年报“财务费用”附注中拆出来的多年档案。

为什么必须有它：

`clean_annual.fin_exp` 通常只是利润表里的财务费用净额。DCF 预测时，财务费用不能只靠这个净额平推，因为模型需要知道：

- 利息支出是多少。
- 利息收入是多少。
- 其他财务费用是多少。
- 利息支出应当跟平均有息债务走。
- 利息收入应当跟平均现金走。

`financial_expense.yaml` 会保存按年份归档的分析结果，典型结构包含：

```text
periods:
  "2024":
    status: approved
    confidence: high
    components:
      interest_expense: ...
      capitalized_interest: ...
      interest_income: ...
      subsidy: ...
      other: ...
    derived:
      interest_expense_rate: ...
      cash_interest_rate: ...
      other_fin_exp_abs: ...
      interest_expense: ...
      interest_income: ...
    checks:
      total_check:
        ok: true
      basis_check:
        ok: true
```

`defaults_gen.py` 只会采用 latest、`status=approved`、`confidence=high`、检查通过且年份匹配 `base_period` 的记录。采用后，`defaults.yaml` 里的 `income.financial_expense` 会把 source 改成：

```text
annual_report.fin_exp_note
```

这说明财务费用参数来自年报附注证据，而不是单纯机械平推。

如果你修改财务费用相关逻辑，必须检查三处是否一致：

- `src/financial_expense_analyzer.py` 如何生成 `financial_expense.yaml`。
- `src/defaults_gen.py` 如何读取并覆盖 `income.financial_expense`。
- `src/calc.py` 如何用这些参数做循环财务费用计算。

## 第五层：yaml1*.yaml 是人的判断覆盖层

入口文件：

```text
companies/{公司名}_{代码}/Agent/yaml1*.yaml
```

处理模块：

```text
src/yaml1_cleaner.py
```

`yaml1` 不是完整模型，而是稀疏覆盖。它只写人真正想改的东西。

收入拆分和算法模板的详细接口见 `docs/yaml1算法模板契约.md`。这里先讲它在流水线中的位置。

`yaml1` 主要支持两类内容。

第一类是收入分解。收入分解分两层：结构层是 `decomposition_sum` 递归求和，叶子层才是具体算法模板。

```yaml
income.revenue:
  kind: decomposition
  rollup: sum
  segments:
    segment_a:                         # 可以继续嵌一层 decomposition
      revenue_family: factor_product
      base:
        base_year: 2024
        unit_factor_to_million_cny: ...
      factors:
        - key: stores
          label: 门店数
          base: ...
          projection: { kind: yoy, values: [...] }
        - key: sales_per_store
          label: 单店收入
          base: ...
          projection: { kind: abs, values: [...] }
```

当前可执行的收入 leaf 模板只有这些：

- `factor_product`：n 因子连乘。适合门店数 × 单店收入、装机量 × 利用小时 × 电价、生息资产 × 净息差等。
- `growth`：基年收入按 `revenue_yoy` 逐年递推。
- `abs`：用 `revenue_abs` 直接给逐年收入金额。
- `kind: formula` + `formula_ref`：引用 `formulas.nodes` 中已经求值的受限公式节点，适合跨期递推、分段函数、中间变量复用等模板装不下的长尾。
- `vol_price` / `vol_price_margin`：旧样本兼容名，本质是二因子 `factor_product`。

如果 yaml1 含有 `formulas.nodes`，`yaml1_cleaner.py` 会先执行受限 formula/DAG 求值，再递归折叠 decomposition 树，把所有 leaf 收入加总，最后换算成 `model.revenue_yoy`。注意：预测收入增长率锚定的是 `clean_annual` 的基年收入，而不是分部手工加总值。分部加总和 clean 基年收入的残差会进入 backtest。

单位纪律：

`defaults.yaml`、`Agent/.modelking/forecast_params.yaml`、`calc.py` 和 `Agent/forecast/` 主体金额单位都是百万元人民币。业务 driver 可以保留自己的自然单位，例如吨、元/吨、门店、万元/店、兆瓦、小时、电价，但 revenue leaf 最终必须通过 `unit_factor_to_million_cny` 折算到百万元收入。

典型例子：

```text
收入（百万元）= 销量（吨） * 吨价（元/吨） / 1,000,000
收入（百万元）= 门店数（家） * 单店收入（百万元/家） / 1
收入（百万元）= 生息资产（百万元） * 净息差 / 1
```

如果单位拿不准，不能靠调 `unit_factor_to_million_cny` 让 backtest 看起来通过；必须回到历史原子和来源材料核对。

如果每个 revenue leaf 都带 `knobs.margin`，cleaner 会执行 margin fold：

```text
income.gpm_t = sum(leaf_revenue_t * leaf_margin_t) / sum(leaf_revenue_t)
```

这条路径只折成整体 `income.gpm`，`calc.py` 不看分线 margin。

over-determined 守则：

- 任一 leaf 带 `margin` 时，所有 revenue leaf 都必须带 `margin`。
- 任一 leaf 带 `margin` 时，禁止同时写顶层 `income.gpm` knob。
- 没有任何 leaf 带 `margin` 时，才使用顶层 `income.gpm` 手拍。
- `formula` / DAG 只能通过 `formulas.nodes` + `formula_ref` 进入 cleaner；不得自创 `revenue_family`。`mix_allocation` 仍未实现，遇到父定子占比分配仍需举旗。

第二类是普通 knob：

```yaml
income.gpm:
  kind: knob
  values: [...]
  src: "..."

income.cost_rates.sell_exp:
  kind: knob
  values: [...]
  src: "..."
```

`kind=knob` 的路径必须能映射到 `defaults.yaml` 中的合法路径。否则清洗失败。

普通标准路径也可以使用 `kind: formula`：

```yaml
formulas:
  nodes:
    sell_exp_rate:
      kind: formula
      unit: ratio
      expr: "fixed_sell_exp / revenue + variable_sell_exp_rate"
      inputs: [fixed_sell_exp, revenue, variable_sell_exp_rate]

income.cost_rates.sell_exp:
  kind: formula
  formula_ref: sell_exp_rate
  src: "..."
```

这条路径和 `kind=knob` 一样进入 overlay，区别只是逐年值由 cleaner 的 formula/DAG 求值器算出。formula 输出路径仍必须存在于 `defaults.yaml`，不能拿它创造 calc 不认识的新成本/毛利入口。

`terminal.fade` 用来把显式期扩展到完整预测期：

```yaml
terminal:
  explicit_end: 2031
  fade:
    kind: linear
    to_year: 2036
    target_growth: 0.055
    fade_paths: [model.revenue_yoy]
    hold_paths: []
  perpetual_growth: 0.025
```

如果某个路径在 `fade_paths` 中，它会从显式期末值线性收敛到 `terminal.fade.target_growth`；缺少 `target_growth` 时兼容旧语义，收敛到 `perpetual_growth`。其他逐年路径默认 hold 最后一年值。`target_growth` 是中期经营交接增速，`perpetual_growth` 仍是 `calc.py` Gordon 终值使用的长期经济锚。

这条契约现在由 `tests/test_yaml1_cleaner.py` 和 `tests/test_yaml1_formula.py` 锁住：最小 fixture 必须能覆盖 `factor_product`、`growth`、`abs`、受限 formula leaf、标准路径 formula overlay、嵌套 decomposition、顶层 `income.gpm`、leaf margin 折 `income.gpm`、`income.financial_expense.other_fin_exp_abs`，并从 `clean_yaml1()` 一路跑到 `calc.build_forecast_statements()`。以后新增模板或改路径时，先补测试再改 skill。

### 前端假设沙盘的边界

工作台的「核心假设展示」tab 现在承担两件事：只读时解释 yaml1，进入 edit 模式后临时改所有可识别 knobs 并预览重算结果。这里的改动是**沙盘 patch**，不是正式写文件。

当分析师认可沙盘结果，前端调用 `POST /api/companies/{id}/assumption-brief` 生成给 `/frontend-edit` 的指令。正式链路仍然回到语义源头：`/frontend-edit` 定点修改 `核心假设.md` 正文与末尾 `knobs` 块，并在 A4 白名单内定点 patch 当前 yaml1 → `forecast.py` 正式重算。普通自然语言小改走 `/adj quick`；读新增材料或结构性改动走 `/adj incremental -> /comp`。不要让前端直接把 patch 写回 yaml1 当源头；yaml1 只是派生缓存，这里的 patch 只适用于已有 knob 纯数值小改，语义源头仍是 `核心假设.md`。

`forecast_params.yaml` 是逐年标准参数表。它是 `calc.py` 唯一接受的参数输入。

`yaml1_clean_report.json` 很重要，里面记录：

- yaml1 路径

- clean_annual 来源
- base year
- explicit horizon
- revenue fold 结果
- unit factors
- yearly_paths
- backtest 状态
- warnings
- errors

如果清洗报告里有 errors，预测不能继续。

结果判据：

- `errors` 非空：停止，不生成正式预测。
- `backtest.status` 不是 `passed`：不能当正式结果，必须先解释或修正。
- `warnings` 非空：不一定阻断，但必须逐条读；它们通常是口径差、增长诊断、历史残差或需要复核的边界。
- `fold.revenue_yoy`、`fold.segment_base_revenue`、`fold.gpm_values` 是排查收入折叠和 margin fold 的第一现场。
- `forecast_build.json.review_flags` 或 `dcf_summary.json.review_flags` 非空：三表和估值可以生成，但需要人工复核 flag 指向的年份和科目。

## 第七层：calc.py 生成预测三表与 DCF

入口模块：

```text
src/calc.py
```

输入：

```text
Agent/.modelking/forecast_params.yaml
```

输出数据对象：

```text
income_statement
balance_sheet
cash_flow
dcf
summary
```

`calc.py` 的原则是“会计优先”。它按年循环，每年顺序是：

```text
收入
-> 利润表
-> 资产负债表
-> 现金流量表
-> FCFF
```

但是财务费用和 BS plug 有循环依赖：

```text
平均债务/现金 -> 财务费用 -> 净利润 -> 留存收益 -> BS plug -> 现金/债务 -> 平均债务/现金
```

所以 `solve_forecast_year()` 会迭代：

```text
build_income_statement
-> build_balance_sheet
-> financial_expense_from_balances
-> repeat until converged
```

利润表核心逻辑：

```text
revenue_t = revenue_{t-1} * (1 + revenue_yoy_t)
oper_cost = revenue * (1 - gpm)
cost_rates.* = revenue * rate
cost_abs.* = absolute value
fin_exp = interest_expense - interest_income + other_fin_exp
operate_profit = total_revenue - total_cogs + operating_adjustments
total_profit = operate_profit + non_oper_income - non_oper_exp
n_income = total_profit - income_tax
n_income_attr_p = n_income - minority_gain
```

资产负债表核心逻辑：

```text
revenue_pct fields = revenue * pct
cogs_days fields = oper_cost * days / 365
capex = revenue * capex_pct
depreciation = previous fixed assets * depr_rate
fix_assets = previous fix_assets + capex - depreciation
undistr_porfit = previous undistr_porfit + n_income_attr_p - dividends
minority_int = previous minority_int + minority_gain
plug cash or st_borr to balance BS
```

现金流量表不是单独预测，而是从 IS 和 BS 推导：

```text
D&A = depreciation + amort_intang_assets + lt_amort_deferred_exp + use_right_asset_dep
delta_nwc = current operating_working_capital - previous operating_working_capital
CFO = n_income + D&A - delta_nwc
CFI = -capex
cash_change = ending money_cap - beginning money_cap
CFF = cash_change - CFO - CFI
```

每年 hard check：

```text
BS: total_assets == total_liab_hldr_eqy
CF: beginning cash + cash increase + qa plug == ending cash
```

FCFF：

```text
EBIT = operate_profit + fin_exp
NOPAT = EBIT * (1 - tax_rate)
FCFF = NOPAT + D&A - capex - delta_nwc
```

DCF：

```text
PV_FCFF = sum(FCFF_t / (1 + WACC)^t)
terminal_fcff = last_nopat + last_D&A * (1 - terminal_capex_da_ratio)
terminal_value = terminal_fcff * (1 + terminal_growth) / (WACC - terminal_growth)
enterprise_value = PV_FCFF + PV_terminal_value
equity_value = enterprise_value - net_debt
per_share_value = equity_value / total_shares
```

WACC、terminal growth、terminal capex/D&A ratio 属于 DCF 层，可以在不重建三表的情况下重新估值。

> **`terminal_growth` 单源化（2026-06-25）**：`calc.py` 的 Gordon 终值读 `model.terminal_growth`（defaults 纪律默认 0.025），而 `yaml1_cleaner.py` 的 fade 末端用 yaml1 `terminal.perpetual_growth`。两者分裂时，分析师拍板的永续增速（如会稽山 0.02）只作用于 fade，终值仍用 defaults 0.025，DCF 系统性偏高。修复：`_apply_terminal_growth_override` 在 yearly 参数解析后，把 yaml1 `terminal.perpetual_growth` 灌进 `model.terminal_growth`（source 标 `yaml1.terminal.perpetual_growth`），让 fade 终点与 Gordon g 同源；仅当 yaml1 显式声明 `perpetual_growth` 时覆盖，defaults-only 运行不变（其 terminal 由 `_empty_terminal_from_yaml2` 构造，pg 已 = `model.terminal_growth`，自洽）。会稽山 601579 实测：per_share 16.00（g=0.025）→ 15.16（g=0.02）。新增 `test_terminal_perpetual_growth_overrides_model_terminal_growth`、`test_defaults_only_run_leaves_terminal_growth_at_defaults`。

## 第九层：历史与预测如何拼成完整三表

拼接函数：

```text
src/forecast.py::_write_full_tables()
```

输入：

```text
data.db/clean_annual
Agent/forecast/forecast_is.csv
Agent/forecast/forecast_bs.csv
Agent/forecast/forecast_cf.csv
```

输出：

```text
Agent/forecast/full_is.csv
Agent/forecast/full_bs.csv
Agent/forecast/full_cf.csv
```

拼接逻辑：

```text
1. 从 clean_annual 读取历史年度宽表。
2. 只保留 period <= base_period 的历史行。
3. 若历史 period 与预测 period 重叠，丢掉历史行。
4. 按预测三表的列名投影历史数据。
5. 将历史行与预测行 concat。
6. 按 period 排序。
7. 写 full_*.csv。
```

这里有两个口径转换：

```text
income.credit_impa_loss -> credit_impa_loss
total_opcost = total_cogs
```

第一个转换是因为 clean 历史表要区分 income 和 cashflow 的同名字段，而预测引擎内部利润表只用 `credit_impa_loss`。

第二个转换是为了让历史和预测里的 `total_opcost` 含义一致，避免历史 TuShare 口径和预测定义不一致。

注意：`full_is.csv`、`full_bs.csv`、`full_cf.csv` 是展示和分析用的完整时间序列，不是 DCF 输入。DCF 已经在预测三表、FCFF 和 terminal value 上完成。不要把 full 三表再喂回 `calc.py`，也不要用 full 三表反向覆盖 `Agent/.modelking/forecast_params.yaml`。

## 第十层：季度跟踪层

季度跟踪层是年度预测之后的监控/展示层，不参与 DCF，不回写年度预测。

输入：

```text
Agent/forecast/forecast_is.csv      # 年度预测利润表
Agent/data.db clean_quarterly        # 已发布单季实际
Agent/data.db quarterly_overrides    # 前端写入的③人工覆盖
```

核心口径：

```text
1. 默认展示年 = forecast_is.csv 首个预测年。
2. Q1-Q3 按四态判定：实际 clean_quarterly 优先，其次人工覆盖，其次算法继承。
3. 算法继承采用年度金额锚 + 历史同季金额相位，不直接平摊年度率。
4. 毛利率/费用率/所得税率在展示层反推；后端锁定的是季度金额。
5. Q4 永远残差倒挤：年度预测 - Σ(Q1:Q3)，保证季度合计等于年度。
```

存储边界：只在公司 `data.db` 内新增/使用 `quarterly_overrides` 表；不生成新的季度预测文件，不改 `forecast_is.csv`，不改 `clean.py` 的数据清洗结果。

## 常见误区

误区一：把 `defaults.yaml` 当成最终预测参数。

不对。`defaults.yaml` 是 YAML2 底座。真正给 `calc.py` 的是 `Agent/.modelking/forecast_params.yaml`。

误区二：让 `calc.py` 直接读 yaml1。

不对。`calc.py` 永远不看 yaml1。yaml1 必须先经过 `yaml1_cleaner.py`。

误区三：忘记 `financial_expense.yaml`。

这是高风险误区。财务费用预测依赖利息支出率、现金利率和其他财务费用拆分。`financial_expense.yaml` 是这些参数的年报附注证据来源，位置在公司目录顶层。

误区四：年度 clean 失败时手工 plug。

不对。年度 hard check 失败通常表示 TuShare 漏明细，应该走 `annual_report_reconciler.py` 找年报证据并生成 approved override。季度才允许 QA plug。

误区五：在公司目录顶层写中间产物。

不对。`Agent/.modelking/` 是内部产物目录；`Agent/forecast/` 是正式结果目录；公司目录顶层只放人工/证据输入和主数据库。

误区六：单位不清时靠 unit_factor 调平。

不对。`unit_factor_to_million_cny` 是单位换算，不是配平旋钮。收入 leaf 的历史原子必须先忠实，单位换算只能表达真实单位关系。

误区七：把 full 三表当成模型输入。

不对。full 三表只是历史和预测的拼接结果，方便看长序列；预测和估值的输入是 `Agent/.modelking/forecast_params.yaml`。
