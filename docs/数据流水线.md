# MKA 数据流水线

这份文档是给接手本项目的大模型看的。请把它当成一张地图：你不需要先记住每个函数的细节，但必须先知道数据从哪里来、在哪里被校验、哪些 YAML 是人维护的、哪些是机器生成的、`financial_expense.yaml` 为什么不能遗漏，以及最终 DCF 和完整三表是怎样落盘的。

## 一句话总览

MKA 的主线不是“直接填假设跑 DCF”，而是：

```text
TuShare 原始三表
-> SQLite raw_tushare
-> clean.py 生成可信历史 clean_annual / clean_quarterly
-> financial_expense_analyzer.py 从年报附注生成 financial_expense.yaml
-> defaults_gen.py 从 clean_annual + meta + financial_expense.yaml 生成 defaults.yaml
-> yaml1_cleaner.py 合并 defaults.yaml + yaml1*.yaml，生成 .modelking/forecast_params.yaml
-> calc.py 生成预测三表 + DCF
-> forecast.py 拼接历史与预测，生成 full_is / full_bs / full_cf
```

最重要的分工是：

- `clean.py` 负责历史数据可信。
- `financial_expense.yaml` 负责把财务费用从一个净额拆成利息支出、利息收入、其他财务费用。
- `defaults.yaml` 是机器平推底座。
- `yaml1*.yaml` 是人工判断覆盖层。
- `forecast_params.yaml` 是编译后的逐年标准参数表，只给计算引擎吃。
- `calc.py` 只做会计推演和估值，不直接理解 yaml1。
- `forecast.py` 是正式用户入口，负责把整个过程串起来。

## 目录与文件位置

每家公司对应一个目录：

```text
companies/{公司名}_{代码}/
  data.db
  defaults.yaml
  financial_expense.yaml
  yaml1*.yaml
  annuals/
  recon/
  .modelking/
    forecast_params.yaml
    yaml1_clean_report.json
    forecast_build.json
  forecast/
    forecast_is.csv
    forecast_bs.csv
    forecast_cf.csv
    dcf_detail.csv
    dcf_summary.json
    dcf_summary.csv
    full_is.csv
    full_bs.csv
    full_cf.csv
    run_manifest.json
```

> 注：项目根目录下还有独立子包 `tushareclean/`，封装“取数 → 年报下载 → clean → 自动年报 reconciler → 干净宽表”的能力。它复用 `src/data_fetcher.py`、`src/clean.py`、`src/report_downloader.py`、`src/annual_report_reconciler.py` 的逻辑，但提供独立的 CLI / Python API 与 `companies/` 输出目录，供不需要 DCF/yaml1 的下游直接使用。其入口为 `python -m tushareclean {ticker}`。

其中 `financial_expense.yaml` 的位置必须牢记：

```text
companies/{公司名}_{代码}/financial_expense.yaml
```

它不在 `.modelking/`，也不在 `forecast/`，而是在公司目录顶层。它是年报附注分析得到的业务证据档案，供 `defaults_gen.py` 读取。遗漏它会导致 `defaults.yaml` 退回机械口径，只能从 `clean_annual.fin_exp` 直接推财务费用，无法正确拆分利息支出、利息收入和其他财务费用。

## 文件性质：哪些能改，哪些不能手改

后续大模型接手时，先按文件性质判断动作，不要看到 YAML 或 CSV 就直接改。

```text
人工输入：
  companies/{公司名}_{代码}/yaml1*.yaml

证据输入：
  companies/{公司名}_{代码}/data.db
  companies/{公司名}_{代码}/annuals/
  companies/{公司名}_{代码}/recon/
  companies/{公司名}_{代码}/financial_expense.yaml

机器底座：
  companies/{公司名}_{代码}/defaults.yaml

内部编译产物：
  companies/{公司名}_{代码}/.modelking/forecast_params.yaml
  companies/{公司名}_{代码}/.modelking/yaml1_clean_report.json
  companies/{公司名}_{代码}/.modelking/forecast_build.json

正式输出：
  companies/{公司名}_{代码}/forecast/
```

操作纪律：

- `yaml1*.yaml` 是人工判断覆盖层，可以由核心假设 compiler 生成或由人审后修改。
- `financial_expense.yaml` 是证据输入，不是预测输出；改财务费用逻辑时必须保证它、`defaults_gen.py`、`calc.py` 三者一致。
- `defaults.yaml` 是机器生成的 YAML2 底座，通常不手改；如果确实改了生成逻辑，应重跑生成而不是只改单家公司样本。
- `.modelking/` 下的文件是内部编译产物，只能由程序写，不能手工维护。
- `forecast/` 是正式结果目录，会被正式运行清空重写；不要把人工底稿放进去。

## 第一层：取数与原始镜像

入口模块：

```text
src/data_fetcher.py
```

输出位置：

```text
companies/{公司名}_{代码}/data.db
```

`data_fetcher.py` 从 TuShare 拉取三张表：

- `income`
- `balancesheet`
- `cashflow`

写入 SQLite 的 `raw_tushare` 表。`raw_tushare` 是 EAV 形态：

```text
ticker, endpoint, report_type, end_date, field, value, ...
```

原则是：`raw_tushare` 是原始镜像，后续任何补数、重分类、年报 evidence 都不能回写或污染它。项目里很多设计都是围绕这个原则展开的。

同一个 `data.db` 还会有 `meta` 表，保存 ticker、公司名、股本、市值、收盘价等元信息。后续 `defaults_gen.py` 会用它生成股本、净债务、市值等估值参数。

## 第二层：clean.py 生成可信历史宽表

入口模块：

```text
src/clean.py
```

输入：

```text
data.db/raw_tushare
data.db/meta
companies/{公司名}_{代码}/recon/annual_report_overrides.json
```

输出：

```text
data.db/clean_annual
data.db/clean_quarterly
data.db/clean_adjustments
data.db/clean_warnings
```

`clean.py` 做的是“历史三表可信化”，不是预测。

它会把 `raw_tushare` 的 EAV 数据 pivot 成宽表。宽表字段保持统一：`period` 加 325 个 TuShare 官方字段，再加 6 个 QA plug 字段。跨端点同名字段会加前缀消歧，比如：

```text
income.credit_impa_loss
cashflow.credit_impa_loss
```

年度表和季度表的处理边界不同。

年度 `clean_annual` 必须通过 hard check：利润表、资产负债表、现金流量表、跨表关系都要配平。若年度 hard check 失败，尤其是合计大于明细和这种 TuShare 明细缺口，正确动作是调用 `annual_report_reconciler.py` 去本地年报 Markdown 里找缺失科目，生成 approved override。若目标年度缺少单独年报，可使用后续年度年报的比较报表列作为证据来源。override 应用后只改变 clean 宽表，不改变 raw。

reconciler 的通用性约束（不为任何单一公司写死）：LLM 确认按单个 hard-check 失败 (period, code) 分片调用，复杂公司（含金融子公司、逐年漏同一明细）失败数可达 30+，单次打包所有候选会 ReadTimeout 丢掉全部年报证据；确认用 `temperature=0` + 重试 + 截断检测，使同一证据的判断可复现；年报 snippet 窗口须覆盖整张合并报表（长报表的应收/融资行常在标记 250+ 行之后）；当某年报科目名精确等于残差时优先于"多字段凑和"的投机 group；`target_lt_calc`（明细和 > 合计）对应 TuShare 漏录的负值项（如终止经营净亏损），按 −residual 匹配；短别名是更具体科目名子串时（"应收款项"⊂"应收款项融资"）丢弃以免干扰 LLM。自动核对的失败上限默认须覆盖完整年报历史（≥60），否则复杂公司会"靠前几年配平、后几年悄悄漏"。已确认的 TuShare 字段缺口（`lending_funds` 发放贷款和垫款、`estimated_liab` 预计负债、`receiv_financing` 应收款项融资、`end_net_profit` 终止经营净利润等）以触发条件+字段记录在 `knowledge/known_tushare_defects.json`，只作 LLM 检索提示，不作自动补数依据。这些分片彼此独立，因此通过 `annual_report_utils.parallel_map`（有界、保序、异常透传，`LLM_MAX_WORKERS` 默认 6）并发确认：墙钟从"逐次相加"压成"最慢一次"，但每次调用仍各自保留超时/重试/`chunk_errors` 分片审计，合并结果按分片原序确定性输出——并发只改速度，不改判定与审计。同一并发工具也用于 `financial_expense_analyzer` 的逐年附注分析（各年互不依赖）。

季度 `clean_quarterly` 因为披露不完整，允许显式 QA plug 吸收部分残差。QA plug 是审计字段，不是静默补数：

```text
qa_bs_current_asset_plug
qa_bs_noncurrent_asset_plug
qa_bs_current_liab_plug
qa_bs_noncurrent_liab_plug
qa_bs_equity_plug
qa_cf_cash_reconcile_plug
```

plug 使用必须写入 `clean_warnings`。

## 第三层：financial_expense.yaml 的位置与作用

这是最容易被遗漏的一层，必须单独记住。

入口模块：

```text
src/financial_expense_analyzer.py
```

输入：

```text
companies/{公司名}_{代码}/annuals/{年份}_年度报告.md
data.db/clean_annual
```

输出：

```text
companies/{公司名}_{代码}/financial_expense.yaml
```

这一步发生在 `clean.py` 之后、`defaults_gen.py` 之前。它不是 DCF 输出，也不是 yaml1 覆盖层。它是从年报“财务费用”附注中拆出来的多年档案。

为什么必须有它：

`clean_annual.fin_exp` 通常只是利润表里的财务费用净额。DCF 预测时，财务费用不能只靠这个净额平推，因为模型需要知道：

- 利息支出是多少。
- 利息收入是多少。
- 其他财务费用是多少。
- 利息支出应当跟平均有息债务走。
- 利息收入应当跟平均现金走。

`financial_expense.yaml` 会保存按年份归档的分析结果，典型结构包含：

```text
periods:
  "2024":
    status: approved
    confidence: high
    components:
      interest_expense: ...
      capitalized_interest: ...
      interest_income: ...
      subsidy: ...
      other: ...
    derived:
      interest_expense_rate: ...
      cash_interest_rate: ...
      other_fin_exp_abs: ...
      interest_expense: ...
      interest_income: ...
    checks:
      total_check:
        ok: true
      basis_check:
        ok: true
```

`defaults_gen.py` 只会采用 latest、`status=approved`、`confidence=high`、检查通过且年份匹配 `base_period` 的记录。采用后，`defaults.yaml` 里的 `income.financial_expense` 会把 source 改成：

```text
annual_report.fin_exp_note
```

这说明财务费用参数来自年报附注证据，而不是单纯机械平推。

如果你修改财务费用相关逻辑，必须检查三处是否一致：

- `src/financial_expense_analyzer.py` 如何生成 `financial_expense.yaml`。
- `src/defaults_gen.py` 如何读取并覆盖 `income.financial_expense`。
- `src/calc.py` 如何用这些参数做循环财务费用计算。

## 第四层：defaults.yaml 是机器平推底座

入口模块：

```text
src/defaults_gen.py
```

输入：

```text
data.db/clean_annual
data.db/meta
companies/{公司名}_{代码}/financial_expense.yaml
```

输出：

```text
companies/{公司名}_{代码}/defaults.yaml
```

`defaults.yaml` 是唯一正式 YAML2。它不是 forecast params，也不是人工假设文件。

它的职责是把最新年度 clean 数据变成完整、可审计、可平推的参数底座。每个参数通常长这样：

```yaml
gpm:
  value: 0.28
  source: 1 - clean_annual.oper_cost / clean_annual.revenue
```

典型内容包括：

- `model.forecast_years`
- `model.revenue_yoy`
- `model.wacc`
- `model.terminal_growth`
- `model.terminal_capex_da_ratio`
- `model.plug`
- `market.total_shares`
- `market.net_debt`
- `income.revenue`
- `income.gpm`
- `income.cost_rates`
- `income.cost_abs`
- `income.financial_expense`
- `balance_sheet.base`
- `balance_sheet.revenue_pct`
- `balance_sheet.cogs_days`
- `balance_sheet.capex_pct`
- `balance_sheet.depr_rate`
- `cashflow.base_nwc`

如果没有 yaml1，系统仍然可以走 defaults-only 路径，把 defaults 广播成逐年数组后跑 DCF。

## 第五层：yaml1*.yaml 是人的判断覆盖层

入口文件：

```text
companies/{公司名}_{代码}/yaml1*.yaml
```

处理模块：

```text
src/yaml1_cleaner.py
```

`yaml1` 不是完整模型，而是稀疏覆盖。它只写人真正想改的东西。

收入拆分和算法模板的详细接口见 `docs/yaml1算法模板契约.md`。这里先讲它在流水线中的位置。

`yaml1` 主要支持两类内容。

第一类是收入分解。收入分解分两层：结构层是 `decomposition_sum` 递归求和，叶子层才是具体算法模板。

```yaml
income.revenue:
  kind: decomposition
  rollup: sum
  segments:
    segment_a:                         # 可以继续嵌一层 decomposition
      revenue_family: factor_product
      base:
        base_year: 2024
        unit_factor_to_million_cny: ...
      factors:
        - key: stores
          label: 门店数
          base: ...
          projection: { kind: yoy, values: [...] }
        - key: sales_per_store
          label: 单店收入
          base: ...
          projection: { kind: abs, values: [...] }
```

当前可执行的收入 leaf 模板只有这些：

- `factor_product`：n 因子连乘。适合门店数 × 单店收入、装机量 × 利用小时 × 电价、生息资产 × 净息差等。
- `growth`：基年收入按 `revenue_yoy` 逐年递推。
- `abs`：用 `revenue_abs` 直接给逐年收入金额。
- `kind: formula` + `formula_ref`：引用 `formulas.nodes` 中已经求值的受限公式节点，适合跨期递推、分段函数、中间变量复用等模板装不下的长尾。
- `vol_price` / `vol_price_margin`：旧样本兼容名，本质是二因子 `factor_product`。

如果 yaml1 含有 `formulas.nodes`，`yaml1_cleaner.py` 会先执行受限 formula/DAG 求值，再递归折叠 decomposition 树，把所有 leaf 收入加总，最后换算成 `model.revenue_yoy`。注意：预测收入增长率锚定的是 `clean_annual` 的基年收入，而不是分部手工加总值。分部加总和 clean 基年收入的残差会进入 backtest。

单位纪律：

`defaults.yaml`、`.modelking/forecast_params.yaml`、`calc.py` 和 `forecast/` 主体金额单位都是百万元人民币。业务 driver 可以保留自己的自然单位，例如吨、元/吨、门店、万元/店、兆瓦、小时、电价，但 revenue leaf 最终必须通过 `unit_factor_to_million_cny` 折算到百万元收入。

典型例子：

```text
收入（百万元）= 销量（吨） * 吨价（元/吨） / 1,000,000
收入（百万元）= 门店数（家） * 单店收入（百万元/家） / 1
收入（百万元）= 生息资产（百万元） * 净息差 / 1
```

如果单位拿不准，不能靠调 `unit_factor_to_million_cny` 让 backtest 看起来通过；必须回到历史原子和来源材料核对。

如果每个 revenue leaf 都带 `knobs.margin`，cleaner 会执行 margin fold：

```text
income.gpm_t = sum(leaf_revenue_t * leaf_margin_t) / sum(leaf_revenue_t)
```

这条路径只折成整体 `income.gpm`，`calc.py` 不看分线 margin。

over-determined 守则：

- 任一 leaf 带 `margin` 时，所有 revenue leaf 都必须带 `margin`。
- 任一 leaf 带 `margin` 时，禁止同时写顶层 `income.gpm` knob。
- 没有任何 leaf 带 `margin` 时，才使用顶层 `income.gpm` 手拍。
- `formula` / DAG 只能通过 `formulas.nodes` + `formula_ref` 进入 cleaner；不得自创 `revenue_family`。`mix_allocation` 仍未实现，遇到父定子占比分配仍需举旗。

第二类是普通 knob：

```yaml
income.gpm:
  kind: knob
  values: [...]
  src: "..."

income.cost_rates.sell_exp:
  kind: knob
  values: [...]
  src: "..."
```

`kind=knob` 的路径必须能映射到 `defaults.yaml` 中的合法路径。否则清洗失败。

普通标准路径也可以使用 `kind: formula`：

```yaml
formulas:
  nodes:
    sell_exp_rate:
      kind: formula
      unit: ratio
      expr: "fixed_sell_exp / revenue + variable_sell_exp_rate"
      inputs: [fixed_sell_exp, revenue, variable_sell_exp_rate]

income.cost_rates.sell_exp:
  kind: formula
  formula_ref: sell_exp_rate
  src: "..."
```

这条路径和 `kind=knob` 一样进入 overlay，区别只是逐年值由 cleaner 的 formula/DAG 求值器算出。formula 输出路径仍必须存在于 `defaults.yaml`，不能拿它创造 calc 不认识的新成本/毛利入口。

`terminal.fade` 用来把显式期扩展到完整预测期：

```yaml
terminal:
  explicit_end: 2031
  fade:
    kind: linear
    to_year: 2036
    fade_paths: [model.revenue_yoy]
    hold_paths: []
  perpetual_growth: 0.025
```

如果某个路径在 `fade_paths` 中，它会从显式期末值线性收敛到永续增长；其他逐年路径默认 hold 最后一年值。

这条契约现在由 `tests/test_yaml1_cleaner.py` 和 `tests/test_yaml1_formula.py` 锁住：最小 fixture 必须能覆盖 `factor_product`、`growth`、`abs`、受限 formula leaf、标准路径 formula overlay、嵌套 decomposition、顶层 `income.gpm`、leaf margin 折 `income.gpm`、`income.financial_expense.other_fin_exp_abs`，并从 `clean_yaml1()` 一路跑到 `calc.build_forecast_statements()`。以后新增模板或改路径时，先补测试再改 skill。

## 第六层：forecast_params.yaml 是编译产物

处理模块：

```text
src/yaml1_cleaner.py
```

输出位置：

```text
companies/{公司名}_{代码}/.modelking/forecast_params.yaml
companies/{公司名}_{代码}/.modelking/yaml1_clean_report.json
```

`forecast_params.yaml` 是逐年标准参数表。它是 `calc.py` 唯一接受的参数输入。

不要在公司目录顶层维护这些文件：

```text
forecast_params.yaml
yaml1_clean_report.json
yaml2_yearly.yaml
```

它们必须属于 `.modelking/`，因为它们是内部编译产物，不是人工维护界面。

`yaml1_clean_report.json` 很重要，里面记录：

- yaml1 路径
- defaults 路径
- clean_annual 来源
- base year
- explicit horizon
- revenue fold 结果
- unit factors
- yearly_paths
- backtest 状态
- warnings
- errors

如果清洗报告里有 errors，预测不能继续。

结果判据：

- `errors` 非空：停止，不生成正式预测。
- `backtest.status` 不是 `passed`：不能当正式结果，必须先解释或修正。
- `warnings` 非空：不一定阻断，但必须逐条读；它们通常是口径差、增长诊断、历史残差或需要复核的边界。
- `fold.revenue_yoy`、`fold.segment_base_revenue`、`fold.gpm_values` 是排查收入折叠和 margin fold 的第一现场。
- `forecast_build.json.review_flags` 或 `dcf_summary.json.review_flags` 非空：三表和估值可以生成，但需要人工复核 flag 指向的年份和科目。

## 第七层：calc.py 生成预测三表与 DCF

入口模块：

```text
src/calc.py
```

输入：

```text
.modelking/forecast_params.yaml
```

输出数据对象：

```text
income_statement
balance_sheet
cash_flow
dcf
summary
```

`calc.py` 的原则是“会计优先”。它按年循环，每年顺序是：

```text
收入
-> 利润表
-> 资产负债表
-> 现金流量表
-> FCFF
```

但是财务费用和 BS plug 有循环依赖：

```text
平均债务/现金 -> 财务费用 -> 净利润 -> 留存收益 -> BS plug -> 现金/债务 -> 平均债务/现金
```

所以 `solve_forecast_year()` 会迭代：

```text
build_income_statement
-> build_balance_sheet
-> financial_expense_from_balances
-> repeat until converged
```

利润表核心逻辑：

```text
revenue_t = revenue_{t-1} * (1 + revenue_yoy_t)
oper_cost = revenue * (1 - gpm)
cost_rates.* = revenue * rate
cost_abs.* = absolute value
fin_exp = interest_expense - interest_income + other_fin_exp
operate_profit = total_revenue - total_cogs + operating_adjustments
total_profit = operate_profit + non_oper_income - non_oper_exp
n_income = total_profit - income_tax
n_income_attr_p = n_income - minority_gain
```

资产负债表核心逻辑：

```text
revenue_pct fields = revenue * pct
cogs_days fields = oper_cost * days / 365
capex = revenue * capex_pct
depreciation = previous fixed assets * depr_rate
fix_assets = previous fix_assets + capex - depreciation
undistr_porfit = previous undistr_porfit + n_income_attr_p - dividends
minority_int = previous minority_int + minority_gain
plug cash or st_borr to balance BS
```

现金流量表不是单独预测，而是从 IS 和 BS 推导：

```text
D&A = depreciation + amort_intang_assets + lt_amort_deferred_exp + use_right_asset_dep
delta_nwc = current operating_working_capital - previous operating_working_capital
CFO = n_income + D&A - delta_nwc
CFI = -capex
cash_change = ending money_cap - beginning money_cap
CFF = cash_change - CFO - CFI
```

每年 hard check：

```text
BS: total_assets == total_liab_hldr_eqy
CF: beginning cash + cash increase + qa plug == ending cash
```

FCFF：

```text
EBIT = operate_profit + fin_exp
NOPAT = EBIT * (1 - tax_rate)
FCFF = NOPAT + D&A - capex - delta_nwc
```

DCF：

```text
PV_FCFF = sum(FCFF_t / (1 + WACC)^t)
terminal_fcff = last_nopat + last_D&A * (1 - terminal_capex_da_ratio)
terminal_value = terminal_fcff * (1 + terminal_growth) / (WACC - terminal_growth)
enterprise_value = PV_FCFF + PV_terminal_value
equity_value = enterprise_value - net_debt
per_share_value = equity_value / total_shares
```

WACC、terminal growth、terminal capex/D&A ratio 属于 DCF 层，可以在不重建三表的情况下重新估值。

## 第八层：forecast.py 是正式运行入口

入口模块：

```text
src/forecast.py
```

正式命令：

```bash
py -m src.forecast --ticker 002946.SZ
```

`forecast.py` 做完整编排：

```text
1. 定位公司目录
2. 找到 yaml1*.yaml 与 defaults.yaml
3. 调用 clean_yaml1()
4. 写 .modelking/forecast_params.yaml
5. 写 .modelking/yaml1_clean_report.json
6. 调用 build_forecast_statements()
7. 调用 value_from_statements()
8. 写 forecast/ 预测三表与 DCF
9. 写 .modelking/forecast_build.json
10. 拼接 full_is/full_bs/full_cf
11. 写 forecast/run_manifest.json
```

正式输出目录只能叫：

```text
forecast/
```

`calc.py` 的 `write_outputs()` 会先清空旧 `forecast/` 再写新结果。禁止用 `forecast_current/`、`forecast_fixed/`、`forecast_yaml1/` 之类目录承载正式输出。

常见重跑场景：

```text
只改 yaml1*.yaml：
  py -m src.forecast --ticker 002946.SZ

只想检查 yaml1 清洗：
  py -m src.yaml1_cleaner --ticker 002946.SZ

改 yaml1_cleaner.py / yaml1_formula.py / calc.py / forecast.py：
  py -m pytest tests/test_yaml1_formula.py tests/test_yaml1_cleaner.py tests/test_forecast_pipeline.py -q
  py -m src.forecast --ticker 002946.SZ

改 financial_expense_analyzer.py 或 financial_expense.yaml 结构：
  先重建或核对 financial_expense.yaml
  再重建 defaults.yaml
  再跑 py -m src.forecast --ticker 002946.SZ

改 clean.py 或 annual_report_reconciler.py：
  先重跑 clean/override 流程
  再重建 defaults.yaml
  再跑 forecast
```

原则是：越靠上游的层被修改，越要把下游都重跑一遍；不要只改一个中间产物来“看起来修好了”。

## 第九层：历史与预测如何拼成完整三表

拼接函数：

```text
src/forecast.py::_write_full_tables()
```

输入：

```text
data.db/clean_annual
forecast/forecast_is.csv
forecast/forecast_bs.csv
forecast/forecast_cf.csv
```

输出：

```text
forecast/full_is.csv
forecast/full_bs.csv
forecast/full_cf.csv
```

拼接逻辑：

```text
1. 从 clean_annual 读取历史年度宽表。
2. 只保留 period <= base_period 的历史行。
3. 若历史 period 与预测 period 重叠，丢掉历史行。
4. 按预测三表的列名投影历史数据。
5. 将历史行与预测行 concat。
6. 按 period 排序。
7. 写 full_*.csv。
```

这里有两个口径转换：

```text
income.credit_impa_loss -> credit_impa_loss
total_opcost = total_cogs
```

第一个转换是因为 clean 历史表要区分 income 和 cashflow 的同名字段，而预测引擎内部利润表只用 `credit_impa_loss`。

第二个转换是为了让历史和预测里的 `total_opcost` 含义一致，避免历史 TuShare 口径和预测定义不一致。

注意：`full_is.csv`、`full_bs.csv`、`full_cf.csv` 是展示和分析用的完整时间序列，不是 DCF 输入。DCF 已经在预测三表、FCFF 和 terminal value 上完成。不要把 full 三表再喂回 `calc.py`，也不要用 full 三表反向覆盖 `.modelking/forecast_params.yaml`。

## 常见误区

误区一：把 `defaults.yaml` 当成最终预测参数。

不对。`defaults.yaml` 是 YAML2 底座。真正给 `calc.py` 的是 `.modelking/forecast_params.yaml`。

误区二：让 `calc.py` 直接读 yaml1。

不对。`calc.py` 永远不看 yaml1。yaml1 必须先经过 `yaml1_cleaner.py`。

误区三：忘记 `financial_expense.yaml`。

这是高风险误区。财务费用预测依赖利息支出率、现金利率和其他财务费用拆分。`financial_expense.yaml` 是这些参数的年报附注证据来源，位置在公司目录顶层。

误区四：年度 clean 失败时手工 plug。

不对。年度 hard check 失败通常表示 TuShare 漏明细，应该走 `annual_report_reconciler.py` 找年报证据并生成 approved override。季度才允许 QA plug。

误区五：在公司目录顶层写中间产物。

不对。`.modelking/` 是内部产物目录；`forecast/` 是正式结果目录；公司目录顶层只放人工/证据输入和主数据库。

误区六：单位不清时靠 unit_factor 调平。

不对。`unit_factor_to_million_cny` 是单位换算，不是配平旋钮。收入 leaf 的历史原子必须先忠实，单位换算只能表达真实单位关系。

误区七：把 full 三表当成模型输入。

不对。full 三表只是历史和预测的拼接结果，方便看长序列；预测和估值的输入是 `.modelking/forecast_params.yaml`。

## 修改流水线时的同步规则

凡是修改下列内容，都必须同步更新本文档：

- `data_fetcher.py` 的原始数据结构或字段覆盖规则。
- `clean.py` 的 clean 表结构、校验规则、QA plug、override 应用逻辑。
- `financial_expense_analyzer.py` 的输出结构。
- `financial_expense.yaml` 的位置、结构、采用条件或字段含义。
- `defaults_gen.py` 的参数生成逻辑。
- `yaml1_cleaner.py` 的 yaml1 语义、terminal fade、backtest 或 forecast params 输出。
- `calc.py` 的三表推演、财务费用循环、FCFF 或 DCF 逻辑。
- `forecast.py` 的正式输出目录、`.modelking/` 产物、完整三表拼接逻辑。
- `workbench.py` 或前端对这条流水线的调用方式。

如果代码和本文档冲突，新接手的大模型会被误导。改流水线不改文档，等同于留下隐性 bug。
