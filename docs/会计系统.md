# MKA 会计系统(field_registry)—— 改会计科目必读

> 本文件是 `src/field_registry.yaml` 的权威使用手册。改字段分类/排序/标签、加字段、排查会计归属问题,先看这里。
> ARCHITECTURE.md 只留指针,不重复。CLAUDE.md 顶部「会计问题入口」指向本文件。

## 一句话定位

`src/field_registry.yaml` 是三表 325 个 TuShare 官方字段的**会计元数据唯一真源**。`clean.py` 的校验分类、`workbench.py` 的前端排序与标签、`docs/数据格式参考.md` 全部从它派生。**改一处,三处同步**,不再有并行声明漂移。

## YAML 结构

每个 statement(income / balancesheet / cashflow)一个块:

```yaml
statements:
  income:                       # 顶层 key = income / balancesheet / cashflow
    name: 利润表
    unit: 百万元
    category_order:             # 展示桶顺序(不含 subtotal——subtotal 是内联小计,非展示桶)
      [revenue_item, cost_item, operating_adjustment, ...]
    category_labels:            # 全 category → 中文标签(含 subtotal)
      revenue_item: 收入项
      subtotal: 小计/合计
      ...
    fields:                     # ← 有序列表,顺序 = 会计准则序 = 前端展示序
      - {field: revenue, label: 营业收入, category: revenue_item}
      - {field: oper_cost, label: 减:营业成本, category: cost_item}
      - {field: total_cogs, label: 营业总成本, category: subtotal}      # 小计就插在它该出现的位置
      - {field: invest_income, label: 投资收益, category: operating_adjustment,
         resolve_children: [ass_invest_income, amodcost_fin_assets]}
      - {field: ass_invest_income, label: '其中:对联营和合营企业的投资收益',
         category: sub_item, resolve_parent: invest_income}
      - {field: assets_impair_loss, label: '减:资产减值损失',
         category: operating_adjustment, sign: questionable}
      - {field: total_assets, label: 资产总计, category: subtotal, role: total}  # BS 总计粗体
```

**两个 category 声明的关系:**
- `category_order`:展示桶的**顺序**(前端按桶分组渲染时的桶序)。不含 `subtotal`——subtotal 字段是内联小计,在 `fields` 列表里就位,不是独立桶。
- `category_labels`:**全部** category → 中文标签的映射(含 subtotal,以及任何字段用到但不在展示桶里的 category)。前端每行的"分类说明"列从这里取。

两者分离,因为"桶的展示顺序"和"所有 category 的标签"是不同的集合(标签 ⊇ 桶序)。

## 字段七维

每条 field 最多带这些维度:

| 维度 | 必填 | 说明 |
|------|------|------|
| `field` | 是 | TuShare 官方字段名(禁用内部别名) |
| `label` | 是 | 中文展示标签,直接带"减:/其中:"前缀 |
| `category` | 是 | bucket 归类(revenue_item / cost_item / subtotal / current_asset / ...) |
| `resolve_children` | 否 | 父项 → 子明细列表(IS/CF 合并科目拆分,如 fin_exp → [利息费用, 利息收入]) |
| `resolve_parent` | 否 | 子项 → 父项(反向索引,文档/对账用) |
| `sign` | 否 | `questionable` = 符号已带会计含义(三个减值科目:资产/信用/其他减值损失) |
| `role` | 否 | `total` = BS 三个总计(资产总计/负债合计/负债及股东权益总计)粗体;缺省按 `category==subtotal` 判小计 |
| `combo_of` | 否 | BS 合并科目 → 拆分项列表(`category: combo` 专用,如 accounts_receiv_bill → [notes_receiv, accounts_receiv]) |

## 谁消费它

| 消费方 | 从 registry 取 | 取代了改版前的 |
|--------|---------------|---------------|
| `clean.py` | `IS/BS/CF_FIELD_CATEGORIES`·`IS/BS_SUB_RESOLVE`·`COMBO_RESOLVE`·`SIGN_QUESTIONABLE_IS_FIELDS`(via `from .field_registry import`) | 手维护的分类字典(5 处并行声明) |

下游模块通过 `clean.X` 重导出访问(如 `from src.clean import BS_FIELD_CATEGORIES`、`clean.IS_FIELD_CATEGORIES`),迁移后名字不变,无需改下游。

## 怎么改(常见操作)

1. **改某字段分类**(如把 X 从 cost_item 移到 operating_adjustment):
   编辑 `field_registry.yaml` 改该字段的 `category` → 跑 `python -m scripts.gen_field_reference` 重生文档 → 跑 `python -m pytest tests/test_field_registry.py`。clean.py 的 bucket 求和自动跟进(它读 registry)。

2. **改某字段展示顺序**:在 `fields` 列表里挪那一行。顺序就是会计序,前端跟着变。

3. **改中文标签**:改该字段 `label`。"减:"前缀直接写进 label(不再有 LABEL_OVERRIDE 补丁)。

4. **加新字段**(罕见,通常是 TuShare 新增官方字段):在对应 statement 的 `fields` 加一条,`category` 必须已在 `category_labels` 里;若是全新 category,同时在 `category_order`(若是展示桶)和 `category_labels` 登记。

5. **加合并科目(combo)**:字段 `category: combo` + `combo_of: [拆分项]`,拆分项必须同 bucket。clean.py 的 `COMBO_RESOLVE` 自动重建。

改完统一跑:
```bash
python -m scripts.gen_field_reference     # 重生 docs/数据格式参考.md
python -m pytest tests/test_field_registry.py   # 守一致性
python -m src.clean --ticker 600519.SH --mode annual   # 抽一家验配平未破
```

## 不要做的事

- ❌ **不要在 `clean.py` 里改分类**——`IS/BS/CF_FIELD_CATEGORIES` 等已是从 registry import 的别名,文件里根本没有字典定义可改。
- ❌ **不要在 `workbench.py` 里加排序/标签**——`STATEMENT_META` 已瘦身到只剩 key/name/title/unit,排序标签全在 registry。
- ❌ **不要手改 `docs/数据格式参考.md`**——它是生成物,跑 `gen_field_reference` 重生。
- ❌ **不要把公司特判焊进 registry**——违反项目第一原则(通用性高于一切)。registry 是字段级、公司无关的。公司列报口径差异(如比亚迪预计负债列流动)走 `annual_report_overrides.json` 的 `clean_category` 重分类,不进 registry。

## 边界(B1:元数据同源,公式留代码)

- `check_is/bs/cf` 里的 subtotal 公式(`total_cogs = sum(cost_item)`、`total_assets = 流动+非流动` 等)**仍写在代码里,不数据驱动**。registry 只统元数据,不统校验公式。check 函数已稳定(美的/紫金/茅台/比亚迪/万科/三一全过),数据驱动重写是另一个独立重构,风险不值。
- `knowledge/known_tushare_defects.json` **独立保留,未并入 registry**——它是 reconciler 的 LLM 检索提示(触发条件 + 字段),不是字段元数据。
- 6 个 `qa_*_plug` 审计字段**不在 registry**——它们是 clean 阶段的配平脚手架,不是会计科目,留在 `clean.QA_FIELDS`。

## 长期守卫

`tests/test_field_registry.py` 锁 9 条不变量,任何手改 YAML 引入悬空引用/缺标签/字段数漂移都会被测试挡住:

1. 三表字段数 = 86 / 150 / 89(TuShare 官方数值字段数)
2. 每个字段有 label,且 category 在 category_labels 里
3. resolve_children 引用的父子字段都真实存在
4. combo_of 拆分项存在且同 bucket
5. total_fields ⊆ subtotal 字段
6. SIGN_QUESTIONABLE ⊆ IS 字段(且恰为三个减值科目)
7. credit_impa_loss 漂移已修(IS=operating_adjustment + 减:前缀;CF=supplementary)
8. field_order 覆盖该报表全部字段,无遗漏无重复
9. subtotal 不在 category_order(它是内联小计,非展示桶)

## 改版背景(2026-06-22)

改版前,同一字段的会计身份散在 5 处并行声明:`clean.py` 分类字典、`workbench.STATEMENT_META` 的 field_order/category_order/subtotal_after、`LABEL_OVERRIDE`、`数据格式参考.md`,各自维护、已漂移(典型:`credit_impa_loss`/`assets_impair_loss`/`oth_impair_loss_assets` 在 clean.py 是 operating_adjustment,在 stale 文档里是 cost_item,代码还在用 LABEL_OVERRIDE 迁就文档)。本改版把它们合并进 `field_registry.yaml` 单一真源,漂移由构造消除。设计稿见 `docs/plans/2026-06-22-field-registry-design.md`。
