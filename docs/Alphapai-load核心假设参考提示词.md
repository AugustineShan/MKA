# Alphapai-load 核心假设参考稿提示词

把下面整段复制给 Alphapai 网页端。替换 `{公司}`、`{ticker}`、`{关注点}` 即可。

---

# 任务：把 Alphapai 金融数据库理解装载成核心假设参考.md

请基于 Alphapai 自有金融数据库和可检索资料，为 `{公司}`（`{ticker}`）生成一份 **`核心假设参考.md`**。

这不是研报摘要、不是估值报告、不是投资评级。你的任务是把数据库里的公司理解、业务线索、最新进展、预测候选和分歧，整理成一份后续建模 Agent 可以裁决的“人话核心假设参考稿”。

## 0. 产物定位

严格遵守：

- 模式：`alphapai-load`
- 状态：`reference`
- 用途：供后续核心假设裁决，不可直接编译，不可直接作为正式模型。
- 禁止输出 YAML1、JSON schema、DCF、目标价、买卖建议或正式估值结论。
- 你可以给候选值和建议旋钮，但必须标 `reference / 待裁决`。

链路是：

```text
Alphapai 金融数据库理解 -> 核心假设参考.md -> 后续裁决 -> 正式核心假设 -> 编译/预测
```

不是：

```text
Alphapai -> 直接 YAML1 / 直接 DCF / 直接正式核心假设
```

## 0.1 交互方式：先会议 memo，后完整参考稿

不要第一轮直接输出完整 `核心假设参考.md`。第一轮必须先像分析师开会一样，输出一页 **会议 memo**，讲清你对公司的理解、预测候选和待裁决问题，然后停止等待用户确认。

会议 memo 必须包含：

```text
我读完 Alphapai 材料后，先给结论：
1. 这家公司当前核心 thesis 我读成...
2. 利润表主链路最该裁决的是...
3. 我建议的业务骨架是...

时间轴候选：
| 项 | 候选 | 理由 |
|---|---|---|
| 历史事实至 | ... | ... |
| 显式期 | ... | ... |
| 衰减期 | ... | ... |
| 永续增长 | ... | ... |

关键预测候选：
| 区块 | 我的理解 | 候选旋钮 | 主要证据/分歧 |
|---|---|---|---|

待后续裁决：
- ...

这个理解方向可以吗？确认后我再输出完整 `核心假设参考.md`。
```

第一轮不要倾倒完整 markdown 模板、全部资料摘录、所有 source range 或完整 `knobs` 块。用户确认后，第二轮再按第 3-6 节输出完整 `核心假设参考.md`，并在文末写完整 `knobs` 机器自报清单。

## 1. 检索范围

请主动检索 Alphapai 数据库中与 `{公司}` 有关的高权重材料，优先级如下：

1. 最新年报、季报、业绩预告、经营数据公告、重大事项公告。
2. 公司官网、交易所问询回复、投资者关系活动记录、业绩会纪要。
3. 高质量研报、深度报告、行业报告、一致预期和盈利预测。
4. 产业链、价格、销量、产能、渠道、门店、用户、订单、招投标等经营线索。
5. 历史业务变化、竞争格局、管理层表述、政策和行业周期。

如用户给出 `{关注点}`，优先围绕该关注点检索，但不能牺牲利润表主链路完整性。

输出中必须列出资料清单：材料名称、日期、类型、用途、可信度、时效性。

可信度分层：

- A：公司公告、财报、交易所文件。
- B：业绩会、投资者关系、管理层纪要。
- C：高质量研报、行业报告、一致预期。
- D：弱来源、市场传闻、你的推断。

资料冲突时必须写“来源与裁决”：候选 A、候选 B、暂采用哪个、为什么、未采用方放哪里。不能静默丢掉冲突信息。

## 2. 写作纪律

按“事实、推断、建议”三层写：

- 事实：来自公告、财报、纪要、研报或数据库的明确数据和表述。
- 推断：你基于多源资料归纳出的业务判断。
- 建议：可供后续裁决的候选旋钮或参数化方式。

关键数字必须带年份和来源。拿不到就写“未检索到 / 待核”，不要编。

不要倒算派生利润来反推旋钮。收入、毛利、费用、税率等要尽量写成业务驱动或明确比率/金额候选。

默认范围是 **利润表 + 业务层盈利模型**：

- 可以处理：收入、业务线拆分、毛利/成本、销售费用率、管理费用率、研发费用率、税金及附加、below-OP 项目、所得税率、少数股东。
- 默认不主动预测：`financial expense`、`EBIT`、`DA`、`CAPEX`、`CWC`、`shares`、`WACC` 等 BS/CF/DCF 驱动。
- 财务费用例外：只有资料明确给出利润表外生“其他财务费用”时，才可作为 `other_fin_exp_abs` 候选；由现金、债务、利率、BS 推导的 financial expense 不写成预测旋钮。
- BS/CF 硬边界：所有库存、应收、合同负债、资本开支、折旧、营运资本、WACC、股本等线索一律放入收纳区，标注“可能的 BS/CF thesis，待人工判断是否走 /da 或专门流程”。你不得自行判断它们是核心 thesis，不得给逐年预测候选，不得写进 `knobs`。

## 3. 输出抬头

用户确认会议 memo 后，请只输出一份 Markdown，标题为：

```markdown
# {公司}-{日期}-核心假设参考.md
```

抬头必须包含：

```text
模式: alphapai-load
状态: reference
公司: {公司}
ticker: {ticker}
资料截至: {你检索到的最新资料日期}
历史事实至: {最新已披露年度/季度}
建议显式预测期: [{YYYY}, {YYYY}, ...]
建议衰减期至: {YYYY / 待裁决}
永续增长候选: {x% / 待裁决}
用途: 供后续裁决，不可直接编译
```

时间轴四数是参考候选，不是拍板结果。你必须给出：

1. 历史事实至哪一年/季度。
2. 建议显式期从哪年到哪年。
3. 衰减期建议到哪年或写“待裁决”。
4. 永续增长候选或写“待裁决”。

显式期不能机械默认 5 年；必须覆盖资料中已经披露或高置信的拐点年份，例如产能投产、合同到期、税收优惠到期、补贴退坡、价格周期拐点、门店/用户增长拐点。无法判断时写“待裁决”，不要硬平推。

## 4. 正文结构

用户确认后输出完整参考稿时，请严格按以下顺序输出。

### 4.1 一页总览

总览只是导读，不能替代后面 4.4-4.9 的结构化候选。即使总览已经说清楚，也必须继续按后文模板逐段输出。

用 5-10 条 bullet 写清楚：

- 公司怎么赚钱。
- 核心业务线是什么。
- 最近 1-2 年最大变化是什么。
- 未来 3-5 年最关键收入驱动是什么。
- 毛利率/费用率核心变量是什么。
- 数据库材料里的最大分歧是什么。
- 哪些判断最适合交给后续裁决。
- 时间轴四数候选是什么，以及覆盖了哪些已知拐点。

### 4.2 资料清单

| 材料 | 日期 | 类型 | 用途 | 可信度 | 时效性 |
|---|---:|---|---|---|---|

### 4.3 业务理解

写清主营业务、商业模式、产品/服务、客户、渠道、地区、价格、销量、产能、订单、行业位置、竞争格局、近年结构变化、管理层关键表述。

### 4.4 收入拆分候选

每条业务线按模板写：

```markdown
### {业务线} [上挂: 营业收入; compiler: factor_product/growth/abs/formula; status: reference]
- 这条线是什么:
- 历史事实:
  - headline:
  - 业务原子:
  - 单位:
  - 来源:
- 参数化候选:
  - 推荐 family:
  - 旋钮:
  - 派生:
- 预测候选:
  - {YYYY}: ...
  - {YYYY}: ...
- 三件套:
  - 谁定:
  - 为什么:
  - 来源:
- 来源与裁决:
- 风险/缺口:
```

可用 compiler family：

- `factor_product`：量 x 价、门店 x 单店、用户 x ARPU、产能 x 利用率 x 价格等。
- `growth`：收入增速。
- `abs`：收入绝对值。
- `formula`：只有资料明确存在跨期、滞后或复杂公式，且普通 family 装不下时才建议；必须标 `待回测验证`。

优先使用最简单可解释的 family。不要为了显得精细而强行 formula。

### 4.5 毛利/成本候选

写历史趋势、产品结构、价格、原材料、产能利用率、汇率、规模效应。说明建议整体毛利率还是分业务线毛利率。

```markdown
### 毛利率 [上挂: 营业成本; compiler: gpm / leaf_margin; status: reference]
- 历史:
- 驱动因素:
- 预测候选:
- 三件套:
- 风险/缺口:
```

分线毛利率可以写在正文作为候选，但末尾 `knobs` 块默认先不要写 `leaf_margin`，除非用户明确说后续系统支持该项。为了兼容后续工具，`knobs` 优先写整体 `gpm` 或收入/费用/税等明确旋钮。

### 4.6 费用率候选

分别写：

```markdown
### 销售费用率 [上挂: 销售费用; compiler: cost_rate; status: reference]
### 管理费用率 [上挂: 管理费用; compiler: cost_rate; status: reference]
### 研发费用率 [上挂: 研发费用; compiler: cost_rate; status: reference]
### 税金及附加率 [上挂: 税金及附加; compiler: cost_rate; status: reference]
```

每项写历史费用率、变化原因、经营杠杆或投入周期、预测候选和证据。

### 4.7 below-OP、税率、少数股东候选

覆盖以下项目；没材料就写“未检索到明确判断，建议后续沿用默认/年报口径”：

- 资产减值损失
- 信用减值损失
- 其他收益
- 投资净收益
- 公允价值变动净收益
- 资产处置收益
- 营业外收入
- 营业外支出
- 有效税率
- 少数股东损益率
- 其他财务费用外生项，仅限明确利润表外生项

绝对值单位用“百万元人民币”。收益为正，损失/支出为负，符号口径必须说明。

### 4.8 BS/CF/DCF 线索收纳

本节只收纳，不预测，不给逐年候选，不进 `knobs`。

凡是库存、应收、应付、合同负债、资本开支、折旧、营运资本、FCFF、WACC、股本、回购、分红、DA/CAPEX 排程等信息，都写在这里：

```markdown
### {线索}
- 资料说了什么:
- 可能影响:
- 为什么不在本稿入模:
- 建议去处: 待人工判断是否走 /da 或专门流程
```

### 4.9 中期与 terminal 候选

先重述时间轴四数候选：历史事实至、显式期、衰减期、永续增长。然后只写显式预测期建议、中期衰减方向、永续增长候选或“待裁决”。不要写 WACC、目标价或评级。

### 4.10 待裁决清单

| 序号 | 事项 | 候选值/方向 | 证据 | 分歧 | 建议处理 |
|---:|---|---|---|---|---|

### 4.11 收纳区

放不进入预测旋钮但有复盘价值的信息：地区、渠道、客户、产品号、副拆分、口径差异、管理层定性表述、研报分歧、行业风险、不适合本层预测的 BS/CF/DCF 信息。

不要只写“见资料”。能誊数就誊数；誊不动就写具体待办。

## 5. knobs 机器自报清单

文末必须放一个 fenced block，语言标签必须是 `knobs`。它是正文预测旋钮的机器可读回声，不是 YAML1。

只写正文里明确给出逐年候选值的“预测输入旋钮”。没有足够逐年数值时，写 `knobs: []` 并说明原因。

### 5.1 顶层结构

下面只是格式示例，不得照抄年份和值。实际 `horizon` 必须使用第 3 节给出的建议显式预测期；如果无法形成逐年候选，就写 `knobs: []`。

```knobs
horizon: [YYYY1, YYYY2, YYYY3]
terminal:
  explicit_end: YYYY3
  fade:
    to_year: null
    kind: 待裁决
  perpetual_growth: null
reason: "Alphapai-load reference 稿，terminal 待裁决"
knobs:
  - anchor: "#整体毛利率"
    family: gpm
    unit: pct
    values: [29.2, 29.9, 30.5]
    status: reference
    source: "正文同源: Alphapai 数据库参考"
```

### 5.2 单条 knob 字段

每条必须包含：

- `anchor`：正文小节锚点，必须用引号包住，例如 `"#低温鲜奶"`。
- `sub`：同一小节多个旋钮时必须写，例如 `销量`、`吨价`、`收入`。
- `family`：语义族名。
- `unit`：只允许 `pct`、`ratio`、`abs_mn`。
- `values`：逐年数组，长度必须等于 `horizon`。
- `status`：本稿统一写 `reference`，估算项可写 `estimated·待校准`。
- `source`：简短说明来源。

### 5.3 单位

- `unit: pct`：写百分数显示值。例如 15.4% 写 `15.4`，不要写 `0.154`。
- `unit: ratio`：写小数或倍率原值。例如 30 天写 `30`，2.5% 若作为 ratio 写 `0.025`。
- `unit: abs_mn`：写百万元人民币金额。例如 -70 百万元写 `-70`。

平推或“维持不变”也必须展开逐年：

```yaml
values: [15.4, 15.4, 15.4, 15.4, 15.4]
```

不能写：

```yaml
values: "全程 15.4%"
```

### 5.4 推荐 family

`family` 在这里是语义标签，不是 YAML1 path，也不是让你扩展模型能力的许可。为降低误写，只允许以下保守 family。其他情况写正文或收纳区，不进 `knobs`。

| family | 适用 | unit |
|---|---|---|
| `factor_yoy` | 销量、价格、门店数、ARPU 等因子增速 | `pct` |
| `growth` | 收入增速 | `pct` |
| `abs` | 收入或 below-OP 科目绝对值 | `abs_mn` |
| `gpm` | 整体毛利率 | `pct` |
| `cost_rate` | 销售/管理/研发/税金及附加费用率 | `pct` |
| `tax_rate` | 有效税率 | `pct` |
| `minor_rate` | 少数股东损益率 | `pct` |
| `other_fin_exp_abs` | 其他财务费用外生项 | `abs_mn` |

不要在 `knobs` 块里写：

- 历史值。
- 派生预测值，例如收入金额、毛利额、净利润、净利率、EBIT。
- YAML1 path、defaults path、base、history、完整拆分树。
- WACC、股本、目标价。
- BS/CF/DCF 线索，包括 capex、折旧、营运资本、库存、应收、合同负债、FCFF。
- `leaf_margin` 或复杂 formula input，除非用户明确说后续系统支持。
- terminal fade 期逐年序列。

### 5.5 knobs 示例

下面只展示字段形态。年份和值不得照抄；实际年份必须来自抬头的建议显式预测期，`values` 长度必须等于 `horizon`。

```knobs
horizon: [YYYY1, YYYY2, YYYY3]
terminal:
  explicit_end: YYYY3
  fade:
    to_year: null
    kind: 待裁决
  perpetual_growth: null
reason: "Alphapai-load reference 稿，terminal 待裁决"
knobs:
  - {anchor: "#低温业务", sub: 销量, family: factor_yoy, unit: pct, values: [7, 6, 6], status: reference, source: "正文同源"}
  - {anchor: "#低温业务", sub: 吨价, family: factor_yoy, unit: pct, values: [1, 1, 1], status: reference, source: "正文同源"}
  - {anchor: "#整体毛利率", family: gpm, unit: pct, values: [29.2, 29.9, 30.5], status: reference, source: "正文同源"}
  - {anchor: "#销售费用率", family: cost_rate, unit: pct, values: [15.4, 15.1, 14.8], status: reference, source: "正文同源"}
```

## 6. 最终自检

用户确认后，输出完整 `核心假设参考.md` 前请检查：

- 只输出 Markdown，没有 YAML1、DCF、目标价、评级。
- 抬头明确 `状态: reference`，且不可直接编译。
- 资料清单和时效性完整。
- 事实、推断、建议分开。
- 关键数字有年份和来源。
- 顺序是：总览 -> 资料 -> 业务理解 -> 收入 -> 毛利 -> 费用 -> below-OP/税 -> BS/CF/DCF线索收纳 -> 中期 -> 待裁决 -> 收纳区 -> knobs。
- 没有把派生利润、EBIT、WACC、股本、目标价写成预测旋钮。
- `knobs` 值与正文预测候选一致。
- 示例年份和值没有被照抄。
- `unit: pct` 使用百分数显示值，不是小数。
- 所有 `values` 长度等于 `horizon`。

第一轮只输出会议 memo 并等待确认；不要直接输出最终 `核心假设参考.md`。用户确认后，第二轮直接输出完整 `核心假设参考.md`，不要再解释你将如何做。
