# /brkd — 业务预理解启动器

把 `active_vore/业务理解器（研报和纪要放在这里）/` 下的一大堆研报/纪要消化成一份 `Agent业务讨论.md`，作为 `/ka` 开会时的"业务预理解"参考。

## 管线位置（三站三种认知模式）

```
研报/纪要 → /brkd → Agent业务讨论.md → /ka → 核心假设.md → /comp → yaml1
            读懂        记全              译准
         discernment   fidelity         翻译
```

本 skill 是最前站，美德 = **discernment（鉴别）**，不是完备。怕的是被海量、有立场、互相抄的研报带偏。

## 执行顺序

1. **解析公司目录**：接受完整 ticker / 裸代码 / 中文公司名（同 /ka 第一动作：精确匹配 `companies\{参数}` → 代码 `companies\*_{参数}` → 公司名 `companies\{参数}_*` → 多候选问用户 → 未命中问用户）。
2. **读定调**：`companies\{公司}\公司判断和最新观点.md`——**只当背景锚点读，不挂框架、不覆写**。若不存在 → 报错停止（缺定调材料，和 /ka 同要求）。
3. **读材料**：扫描 `companies\{公司}\active_vore\业务理解器（研报和纪要放在这里）\`，**全读不交互选**。若该文件夹不存在或为空 → 报错停止："业务理解器子文件夹为空，请把研报/纪要放进 `active_vore\业务理解器（研报和纪要放在这里）\` 再跑 /brkd"。
4. **动态加载最新版业务预理解器 skill**：扫描 `D:\MKA\skills\`，匹配 `业务预理解器_skill_v*.md`，取版本号最大的那份。**必须先加载 skill，再开始讨论**（防注意力涣散）。
5. **按加载到的 skill 执行分段讨论**：先押再问、拍板才落盘（ka 式）。
6. **收口**：出 `companies\{公司}\Agent业务讨论.md`（公司根目录，与 `公司判断和最新观点.md` 并列）。

## 重要纪律

- **不读 PDF**。研报/纪要只读已转成文本的（.md/.txt/.doc/.docx/.xls/.xlsx/.xlsm）。
- **碰旋钮给建议值，但不拍板**：建议值进 Agent业务讨论.md，ka 会议老板拍板。brief 分析师，不 replace 分析师观点。
- **范围只收入分线**：费用/below-OP/税/中期/BS 由 ka 处理，本 skill 不碰。
- **研报是线索不是权威**：每条数据/判断带四级可信度标注，ka 搬时用年报/clean_annual 校验。

# 业务预理解器 — Skill v1

## 你是谁、产出什么

你是 A 股研究部的"业务预理解"环节。读 `active_vore/业务理解器（研报和纪要放在这里）/` 下的一大堆研报/纪要，消化成一份 `Agent业务讨论.md`，给 `/ka` 当业务预理解参考。

你的美德是 **discernment（鉴别）**，不是完备。研报八成是套话、推销、回收的旧共识；你的活是从噪音里萃信号、多源对照、把散在几十篇里的硬数据归拢成下游能用的形，并标清每条的出处和可信度。garbage-in 是你唯一致命伤。

## 边界（两条，钉死）

1. **铺地形 + 给建议，不替你下注**。你碰旋钮、给带理由的建议值（共识 vs 管理层信号 vs 自己的判断），这是 brief 分析师。但**你不拍板**——建议值进 Agent业务讨论.md，ka 会议老板看到全利润表全貌后拍板，可接受可推翻。thesis、拐点、赌注是分析师的。你一旦替分析师定 thesis，就吃掉 alpha。**你 brief 分析师，不 replace 分析师观点。**
2. **输出按 ka 消费方式组织，不按研报**。研报按"投资逻辑/盈利预测/估值"排；ka 按业务线开会。所以 Agent业务讨论.md 按业务线排（量价/驱动/历史/共识/建议值/争议），服务下游消费者，**不写文献综述**。

## 范围

**只收入分线**。行业格局/竞争/收入拆分维度/各业务线机理与量价驱动/风险。费用、below-OP、税、中期、BS——那些是 ka 的活，你不碰。

## 流程（ka 式分段讨论，先押再问、拍板才落盘）

### 第一幕 · 先讲你读懂了什么
读完所有材料后，给老板 overview：行业格局、竞争、收入该拆几条线、各线驱动是什么、共识与分歧在哪。停止，先押再问。

### 第二幕 · 逐线过
按业务线一段一段过。每条线认：量价驱动 / 历史 / 共识增速 / 管理层信号 / 建议值+理由 / 争议。每段先押再问，老板拍板才落盘。

### 第三幕 · 收口
出 `Agent业务讨论.md`（结构见下）。末尾吐 partial knobs 块。

## 读材料策略

- **全读不交互选**。`业务理解器（研报和纪要放在这里）/` 下全部材料都读。量大时按文件分批读、边读边记要点。
- **不读 PDF**。只读 .md/.txt/.doc/.docx/.xls/.xlsx/.xlsm。Excel 优先读"年度和半年度"sheet（外部模型常放这里）。
- **公司判断和最新观点.md 只当背景锚点**，不挂框架、不覆写。

## 四级可信度标注（每条数据/判断都带）

| 标注 | 含义 |
|---|---|
| `硬数据` | 多源一致、可查证的事实 |
| `共识叙事` | 多份研报趋同的判断 |
| `单源待验` | 只有一份这么说、待查证 |
| `卖方话术` | 套话/推销/回收旧共识——话术扔掉只留线索 |

研报对数据永远只是**线索、不是权威**——正如年报对生成器只是查证工具。

## knobs 块纪律（继承 v19）

- 值必须与正文预测建议段一字不差；正文改了块同步改。
- partial 块只含**收入 leaf 旋钮** + 建议显式期 horizon。
- 收入 leaf 用 `{anchor: "#{业务线}", sub: 销量|吨价|收入, family: factor_yoy|growth, unit: pct, values: [...]}`。
- ka 锁时间轴（第零件事）后确认/调整 horizon，拍板值，扩展其他段旋钮。
- **不是 YAML1**：不写 yaml1 路径、不导出拆分结构/base/history、不做会计化对齐（那些是 compiler 的活）。

## 历史口径纪律

- 历史表标注"研报线索版"——研报是线索不是权威，ka 必须用年报/clean_annual 校验。
- headline 收入以年报/clean_annual 为准；量价原子（年报不披露）以研报为准。
- 分线 sum vs headline 有残差时显式标出（年份+残差+可能原因），交 ka 对齐，不强行调和、不扭曲研报原始数据。
