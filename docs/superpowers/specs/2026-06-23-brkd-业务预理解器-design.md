# /brkd 业务预理解器 — 设计文档

> 日期: 2026-06-23
> 状态: 设计已与用户对齐，待实现

## 1. 背景与动机

MKA 建模管线当前两站：

```
核心假设.md ──/comp──▶ yaml1 ──forecast──▶ DCF
            (ka 生成器)      (编译器)
```

ka（`核心假设生成修改器_skill_v19.md`）一肩挑两件事：① 业务理解（读研报/纪要→懂业务→定收入拆分维度+各线驱动）+ ② 建模格式（完整利润表、搬历史、旋钮、knobs 块、compiler 族标记、三件套）。两件事是两种心智活动，挤在一个 skill 里导致 ka 过重。

更关键：**读研报形成业务理解**是整条管线唯一还纯靠人力的入口，也是研究部接新公司最耗时的前端。研报量大、有立场、互相抄，agent 至今没有规模化吸收这种不可信外部语料的能力。

本设计新增管线最前站——**业务预理解器**，把"业务理解"从 ka 拆出来独立成站。

## 2. 定位：第三种认知模式

```
大量研报/纪要 ──/brkd──▶ Agent业务讨论.md ──/ka──▶ 核心假设.md ──/comp──▶ yaml1
              业务预理解        生成器              编译器
              读懂              记全                译准
              discernment       fidelity            翻译
```

三站三种心智，互不串味：

| 站 | 美德 | 怕什么 | 干什么 |
|---|---|---|---|
| 业务预理解器（新） | discernment 鉴别 | 被海量、有立场、互相抄的研报带偏 | 从噪音里萃信号、多源对照、数据萃取、标清出处 |
| 生成器 ka | fidelity 忠实 | 漏、格式错 | 忠实穷尽地记一个已知模型 |
| 编译器 comp | 翻译准 | 译错 | 机器化翻译 |

业务预理解器是全系统**第一次让 agent 大规模吸收不可信外部语料**。它的纪律重心不在"格式对"，在"**别被带偏 + 标清出处**"。garbage-in 是它唯一致命伤。

## 3. 命令与文件

| 项 | 值 |
|---|---|
| 启动命令 | `/brkd <公司名或代码>`（breakdown） |
| 启动器 skill | `.claude/skills/brkd/SKILL.md` |
| 执行 skill | `skills/业务预理解器_skill_vN.md`（动态加载最新版，同 comp/ka 模式） |
| 产物 | `companies/{公司}/Agent业务讨论.md`（公司根目录，与 `公司判断和最新观点.md` 并列） |

触发方式：**独立 skill**，手动先跑 `/brkd` 产出 `Agent业务讨论.md`，再跑 `/ka`。不做 ka 内置前置——三站各管一段，互不串味。

## 4. 目录改动 + init/ka 改动

`active_vore/` 从扁平改成两个职责子文件夹：

```
companies/{公司}/active_vore/
├── 核心假设生成（模型放在这里）/     ← ka 改读这里
└── 业务理解器（研报和纪要放在这里）/ ← /brkd 读这里
```

### init.py 改动

`init.py` 创建公司目录时，自动在 `active_vore/` 下建这两个子文件夹（已存在则跳过）。**只保证新 init 出来的目录结构正确**，存量公司扁平文件迁移不管（用户自行归置）。

### ka 改动

- **第四动作"读取最新活跃素材"**：从扫 `active_vore/` 改为扫 `active_vore/核心假设生成（模型放在这里）/`，取最新文件。
- **第一幕"先讲你读懂了什么"前**：先读 `Agent业务讨论.md`（若存在）作为业务预理解参考。ka 收入段开会时：历史和骨架现成（来自 Agent业务讨论.md），旋钮建议值现成，老板拍板接受/调整即可。

## 5. 业务预理解器工作流程（ka 式分段讨论）

1. **解析公司目录**（同 ka 第一动作：精确/代码/公司名匹配）。
2. **读定调**：`公司判断和最新观点.md`——只当**背景锚点**读，不挂框架、不覆写（thesis 是 ka 那层的事）。
3. **读材料**：扫 `active_vore/业务理解器（研报和纪要放在这里）/` 全量研报/纪要及其他材料，**全读不交互选**（符合"重输入"承诺）；量大时按文件分批读、边读边记要点。
4. **第一幕·给 overview**：把读懂的业务摆出来——行业格局、竞争、收入该拆几条线、各线驱动是什么、共识与分歧在哪。停止，先押再问。
5. **逐线讨论**：按业务线一段一段过，每条线认量价/驱动/历史/共识/建议值/争议。老板拍板才落盘。
6. **收口**：出 `Agent业务讨论.md`（含 partial knobs 块）。

讨论形态：**ka 式分段讨论**——先押再问、拍板才落盘。判断是老板的，skill 只 brief。

## 6. Agent业务讨论.md 结构（按 ka 消费方式组织，不按研报）

**关键纪律**：研报按"投资逻辑/盈利预测/估值"排，ka 按业务线开会。产物按业务线排，服务下游消费者，**不写成文献综述**。

```markdown
# Agent业务讨论 · {公司} {代码}
> 材料: 列出读了哪几份研报/纪要（来源清单）
> 日期: YYYYMMDD
> 性质: 业务预理解参考，非权威数据源（权威数字 ka 从年报/clean_annual 取）
> 范围: 只收入分线（费用/below-OP/税/中期/BS 由 ka 处理）

## 行业格局与竞争（共识 / 分歧 标注）

## 收入拆分建议：N 条线（每条线是什么、为什么这么拆）

### {业务线A} [建议: 量×价族; compiler: factor_product]
- 建议骨架: 旋钮=销量yoy/吨价yoy；派生=收入(销量×吨价/100)、占比；上挂: 收入→营业收入
- 机理: 这条线怎么赚钱、增长引擎（ka 写三件套"为什么"的素材）
- 争议: 多空分歧 [可信度]

历史（研报线索版·ka 须用年报/clean_annual 校验 headline）:
| 指标\年 | 2020 | ... | 2024 | 来源·可信度 |
| 销量(万吨) | ... | | | [研报A·硬数据] |
| 吨价(元/吨) | ... | | | [研报A·硬数据] |
| 收入(M) | ... | | | [研报B·共识叙事] |
> 分线 sum vs headline 营业收入: 2023 差 +0.9% [口径残差·待 ka 对齐]

预测建议（业务预理解·ka 会议拍板）:
- 研报共识: 销量yoy +8%、吨价yoy +0.3%  [共识叙事·3份趋同]
- 管理层信号: 纪要称"24h鲜奶产能受限,扩产到2026"  [单源待验]
- 建议值: 销量yoy +6%（共识下调·产能瓶颈）、吨价yoy +0.3%（维持）
- 理由: 共识+8%未计产能瓶颈；管理层纪要显示2026前难放量→下调

### {业务线B} ...

## 风险与看点

## 机器自报清单（partial knobs 块·仅收入 leaf·ka 扩展）
```knobs
horizon: [2025, 2026, 2027, 2028]   # 建议显式期，ka 锁时间轴时确认
knobs:
  # 收入 leaf 因子（仅本 skill 拥有的收入旋钮；费用/below-OP/税/衰减期由 ka 补）
  - {anchor: "#{业务线A}", sub: 销量, family: factor_yoy, unit: pct, values: [6,6,6,6]}
  - {anchor: "#{业务线A}", sub: 吨价, family: factor_yoy, unit: pct, values: [0.3,0.3,0.3,0.3]}
  - {anchor: "#{业务线B}", sub: 收入, family: growth,     unit: pct, values: [...]}
```
```

### 四级可信度标注（核心纪律，每条数据/判断都带）

| 标注 | 含义 |
|---|---|
| `硬数据` | 多源一致、可查证的事实 |
| `共识叙事` | 多份研报趋同的判断 |
| `单源待验` | 只有一份这么说、待查证 |
| `卖方话术` | 套话/推销/回收旧共识——话术扔掉只留线索 |

研报对数据永远只是**线索、不是权威**——正如年报对生成器只是查证工具。

## 7. 预处理契约（与 ka 格式衔接）

Agent业务讨论.md 的业务线段落，长成 ka `核心假设.md` 收入段的"半成品"——骨架、历史、旋钮建议值、knobs 块备好，三件套终稿和拍板留给 ka。

| 段落 | 业务预理解器 | ka 接手 |
|---|---|---|
| 拆几条线 / 族 / compiler 标记 | ✅ 建议 | 确认/调整 |
| 骨架（旋钮/派生/上挂） | ✅ 建议 | 确认/调整 |
| 量价历史序列（带来源·可信度） | ✅ 从研报归拢 | 搬进核心假设.md；用年报校验 headline，量价原子（年报不披露）以研报为准 |
| 口径残差 flag | ✅ 标出 | 对齐处理（ka 本来就要做加总校验） |
| 机理 + 争议 | ✅ 当素材 | 写进三件套"为什么" |
| 研报共识增速 + 管理层信号 | ✅ | — |
| **建议旋钮值 + 理由** | ✅（碰旋钮） | 老板拍板：接受/调整 |
| **partial knobs 块（收入 leaf）** | ✅ 预写 | 拍板后改值；补费用/below-OP/税/衰减期/少数股东旋钮 |
| 三件套终稿 | ❌ | ✅ |
| 费用/below-OP/税/中期/BS | ❌ | ✅ |

两条格式纪律：
1. 历史表标注"研报线索版"——研报是线索不是权威，ka 必须用年报/clean_annual 校验。把"别被带偏"焊进格式。
2. 每条数据带 `来源·可信度`——四级标注逐条标，ka 搬时能取舍。

knobs 块纪律（继承 v19）：值必须与正文预测建议段一字不差；partial 块只含收入 leaf 旋钮 + 建议显式期 horizon；ka 锁时间轴（第零件事）后确认/调整 horizon，拍板值，扩展其他段。

## 8. 两条全局边界

1. **铺地形 + 给建议，但不替你下注**。skill 碰旋钮、给带理由的建议值（共识 vs 管理层信号 vs 自己的 discernment），这是 brief 分析师——给一个有防御力的起点，不是空地形。但 **skill 不拍板**：建议值进 Agent业务讨论.md，ka 开会时老板看到全利润表全貌后拍板，可接受可推翻。真正的 bet 仍在 ka"和你一起"那层。skill 一旦替分析师定 thesis/拐点，就吃掉 alpha。**它 brief 分析师，不 replace 分析师观点。**
2. **输出按 ka 消费方式组织，不按研报**。按业务线排（量价/驱动/历史/共识/建议值/争议），不按研报的"投资逻辑/盈利预测/估值"排。服务下游消费者，不写文献综述。
