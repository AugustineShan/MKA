## 一、愿景:三个东西

这个项目本质上在做三个东西,重要性排序如下:

```
2. 核心文件      —— 每只票的结构化认知载体(项目的中心)
1. 理解器        —— 生成和维护核心文件的对话式理解器(Omnivore 族)
3. 底座与消费者  —— 可信数据底座 + 把核心文件编译成 DCF 的引擎,及未来更多消费者(MKA)
```

**核心文件是中心。** 一只票的核心文件 = 分析师对这家公司全部结构化认知的 single source of truth:怎么拆、靠什么驱动、每个判断为什么这么给、来源是什么、哪里还悬着。**它的当前载体是一份 markdown:`核心假设.md`**(人类可读、带出处、可复盘的盈利模型底稿)。YAML1(drivers)**不是核心文件本身,而是核心文件被 compiler 编译出来、喂给 DCF 的 driver 形态**——因为 DCF 是第一个明确的消费场景,但不是最后一份,也不是唯一的消费者。

**理解器(Omnivore 族)** 的使命是陪分析师把散落、半成型的判断催化成结构化、可计算、可追溯的形态;认知未完成时诚实保存中间态(`核心假设参考.md`),认知完成时严格交付结构(`核心假设.md`)。当前主力成员是**核心假设生成器 v17**(原 Omnivore 升级)。它**只产 markdown,不产 yaml**——翻译成 YAML1 是下游 compiler 的活(防火墙:避免理解器越界、自己幻觉出 yaml)。

**底座与消费者(MKA)** 提供:经过严格配平校验的 clean 财报数据(已建成)、把判断编译进 DCF 的整条计算链(**已建成并通电**)。这条链不是"calc 直接吃 YAML1",而是分三棒:**compiler** 把 `核心假设.md` 翻成 `yaml1`(理解层的 raw_tushare:忠实、非标、引擎不能直接吃);**`yaml1_cleaner.py`**(理解层的 clean.py)把 yaml1 ⊕ yaml2 折叠/展开/resolve/回测成标准参数;**`calc.py`**(纯 DCF 算账核,**永远看不到 yaml1**)只吃清洗后的标准参数。calc.py 是核心文件的第一个消费者;报告生成、复盘、未来的叙事分析都是后续消费者。

一句话:**帮分析师写好核心文件(核心假设.md),然后消费核心文件。**

```

                          │
              [compiler·LLM·纯翻译]✅(=yaml1compiler_v4 (2).md)─> yaml1   (理解层的 raw_tushare)

```

## 二、设计原则(全部经实战检验)

1. **契约所有权唯一。** 每种数据格式有且只有一个"拥有者"(协议文档或代码),其他一切文档只链接不复制。格式变更只发生在拥有者处。

2. **需求写行为,不写字段。** 验收标准是"对真实公司表现出什么行为",不是"包含哪些字段"。

3. **Sparse overlay 分层。** YAML2 = 完整、配平、无判断的默认底座(机器从 clean 财报生成);YAML1 = 不完整但有观点的判断覆盖层(人通过对话生成、compiler 翻译)。`yaml1_cleaner.py` 合并两者。YAML1 永不膨胀成第二份 YAML2。

4. **对眼神三态,禁止静默填值。** 每个进入 YAML1 的预测值,必须处于三种合法判断来源状态之一(分析师直接给 / 提议后确认 / 明确认领回落默认)。值存在但分析师从未过目,是头号禁忌。

5. **引擎管辖权。** 财务费用由引擎根据现金/负债余额循环倒算,其下游(利润总额/净利/归母/EPS)同属引擎管辖。YAML1 对引擎管辖科目不承载、不校验、不强制锚定;素材中这些科目的断链(#VALUE!/#REF!)是设计预期,不是缺陷。

6. **认知载体 ≠ 计算参数表。** 不驱动计算的信息(副拆分、孤立情报、口径说明)同样是认知,必须有标准化保留区。计算引擎安全跳过它们,但未来消费者能读到。"不驱动计算"不等于"不值得保留"。**(v2:已落地为 YAML1 的一等公民 `stash` kind——见原则 11。)**

7. **完整性 = 双层定义。** 机器层:引擎能在 YAML1+YAML2 合并后无歧义跑通(硬性 checklist);分析师层:所有"该有观点"的 driver 都经过分析师意识(固定最小集 + 公司特异性增补)。完整性不等于填满,sparse 是设计意图。

8. **真实公司验收。** 每个阶段的完成标准是在真实 ticker 上端到端跑通并通过硬校验,不是单元测试绿了。clean 数据任何一条年度硬校验不过即停止;失败的运行不改判成功。

9. **LLM 是翻译,Python 是执行。** LLM 负责理解非标素材、对话、归因、把自然语言算法翻成可执行式(编译时一次);一切确定性计算、校验、配平、求值由 Python 完成(运行时,LLM 不进计算环)。LLM 永不直接修改数据,只产出供 Python 校验后应用的结构化提议。

10. **Excel 只在边界。** 券商模型进来(输入)、给基金经理的报表出去(输出),系统内部不存在 Excel。

11. **"绝不丢信息"是逐层接缝铁律(v2 新增,推广原则 4/6)。** 整条链每一道接缝——素材→`核心假设.md`、`核心假设.md`→yaml1——都守同一条:**忠实搬运;捕获不了就举旗(标待办);绝不静默降级成"看着像捕获了"的占位。** 头号失败是"静默占位"(如收纳区只留单元格引用、不誊数:既不是数据、又不喊自己残缺)。上游丢了的,下游再诚实也补不回(compiler 只能搬它拿到的 .md)。所有副拆分一视同仁誊全,读不动就标待办、不退而留引用。

12. **修复在源头焊死(v2 新增)。** 符号、`unit_factor`、forex 路径这类系统性问题,在它该被解决的那一层一次性解决,不靠每条线 / 每家公司手工绕。

13. **符号靠利润表配平等式动态验证,不假设、不手洗(v2 新增)。** 减值类符号曾横跨 clean_annual/yaml2/calc.py 静默注水;根治:减值从 `total_cogs` 剥离、按真符号并入营业利润,clean.py 的 IS 校验从"假设符号"升级成"验证符号"。TuShare 符号随 2019 报表格式变更逐年翻 → 定**预测期靠 yaml1 现代口径、历史期靠动态验证**。

### 任务 D:Commit 版本管理

**目标**:核心文件 + 计算结果的快照、diff、revert、历史列表——"投资判断的版本控制",支撑场景 4.7 复盘和 4.4 偏差分析。

**验收行为**:任意时点可快照当前核心文件全家与计算结果;任意两个快照可 diff 出 driver 级变更明细(含理由);可 revert 到任意历史版本并重算复现当时结果。

**定位**:防君子不防破解——文件夹裸露可读,机器只守门(避免静默更新),不用加锁/二进制阻止。**不做**:前端界面(CLI/API 层先行)。

### 任务 E:季度机制

**目标**:年度 master / 季度 slave、distributed/pinned/actual 状态、Q4 永远倒挤作为一致性检验器、实际数据覆写后的联动。

**前置纪律**:季度数据格式**不沿用旧产品文档的 YAML3 schema**。设计时先和分析师重新对一遍需求(协议级对话),确定格式拥有者后再开发。需求语义(master-slave、Q4 倒挤、三种状态)稳定,表达形式不预设。

**验收行为**:季度实际数据到位后自动覆写、Q4 自动重算;Q4 隐含值异常时显式告警;pin 某季度后其余季度正确联动。

**Q4 倒挤约束**:Q4 复用与历史回测同一套约束-反推-诊断设施;具体实现留到任务 E 开工时再建,本次不预建。

### 任务 F:工作台前端

**目标**:核心文件的编辑器 + 计算结果查看器 + commit 界面 + B 类 stash 展示。原地改值、写理由、实时重算、提交/恢复。

**定位约束**:工作台的 AI 是轻量的(环境提示、输入校验),不做判断、不调参、不建议值。重判断的对话全部发生在理解器(网页端 Claude),工作台只是 IDE。

**不做**:跨公司聚合视图、股价/上行空间计算、多用户。

### 任务 G:周边理解器与生成器

- **Driver Update**(轻量 modify):文本边际信息 → 结构化变更提议 → Python 校验后应用。低成本模型承担,协议与生成器 modify 模式共享。
- **Annual Update**:年报后偏差复盘 + 机械滚动 + 新年预测讨论(场景 4.4)。
- **Report Gen**:读核心文件全家组装研报,数字与观点同源(场景 4.6)。原则:不编造观点,只组装分析师自己的判断。
- **Excel 导出**:计算结果 → 给基金经理的 Excel(输出边界)。

每项启动时单独立任务卡,此处只锁需求归属。

### 任务 H:算法模式 / 开放公式层(v2 新增,延后)

**目标**:让系统能消费"不是简单加减乘除"的驱动逻辑(如茅台收入←产能←基酒滞后链)——自然语言把算法讲清 → compiler 翻成可执行式 → 引擎求值 → 回测复现历史才算数。

**纪律(贯穿)**:**默认关、按需开、绝不第一轮开、举手共探、回测当闸。** 这是系统最强也最易出错的能力,简单族公司用不到;触发判据是**种类(不是简单加减乘除),不是深度**(纯加总再深也只是 decomposition 归并;真关系才上公式层)。

**分阶段(默认关所以可后置)**:
1. 现在:简单引擎已通,`formula` kind 的口子已在 yaml1 留好(四种 kind 多态);受限求值器**已建**(`src/yaml1_formula.py`,安全 AST + DAG 按年份滞后序求值 + 回测台,实验性·仅合成 fixture 验证)。
2. 第一个复杂公司逼出来,再建三件:yaml1 公式节点落地(表达式/输入声明/历史种子/回测参照/上挂)、compiler 算法分支(NL→可执行式 + 必带回测)、`yaml1_cleaner.py`/引擎公式求值层(**安全沙箱:受限表达式,不是任意代码执行** + DAG 拓扑序 + 年份滞后序求值 + 回测台)。

**不做**:不为想象中的复杂性提前建求值器;不预先列死"允许哪些运算"(那是变相 hardcode)。

## 七、开发纪律

1. **每次开发完成后同步更新 ARCHITECTURE.md**(MKA 侧既有纪律,延续)。
2. **向 Agent 下任务只给意图 + 约束 + 验收闭环**,实现层(命名、数据结构、模块划分)由 Agent 先提方案、人拍板后写码。
3. **失败如实上报**:校验不过的运行不改判成功;override/补数只走审计链;raw 数据永不修改。
4. **每个任务以真实公司端到端验收**,并对已验证公司做回归。
5. **协议变更优先于代码变更**:实战暴露协议缺陷时,先修协议文档、再改代码,保持契约拥有者的权威。
6. **修复在源头焊死(原则 12)**:系统性问题一次性解决在它该解决的那层,不让下游每家各踩一遍。

## 八、不做什么(稳定,沿袭需求文档)

- 不做实时行情、交易执行、多因子、高频
- 不做自动预测——假设必须经分析师之手(直接给/确认提议/认领回落)
- 不做 AI 自主调参——AI 提议须经确认,决定权永远在分析师
- 不做跨公司聚合视图、股价与上行空间计算(输出给上层系统)
- 不做多用户——个人工具
- 系统内部不操作 Excel——Excel 只在 I/O 边界
- 不替代人的判断——模型是工具,不是决策者
