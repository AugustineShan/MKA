# 技能简要分类

> 给 Agent 的快速分流卡片。这里帮助你先判断该用哪个 skill；真正执行前仍要读取对应 `.claude/skills/*/SKILL.md` 和它加载的最新版动态 runbook。
>
> **STOP：先判斜杠路由，再选择任何外部技能。** `/ka 百润股份`、`/ka 002568`、`/comp 002946`、`/init 新乳业` 是 MKA 任务路由；即使公司名或股票代码像证券问题，也不是行情查询、不是交易建议、不是通用 A 股分析、不是 shell。
>
> **首次冷启动**先读 `docs/快速上手.md`（端到端主线），本文件用于任务级的 skill 分流。
> **规则迷路**先读 `docs/MKA规则导航图.md`。它是契约索引，不替代具体 skill 或契约。

## 一句话总览

```text
/init -> clean 历史事实
/audit -> 财务健康度雷达（历史数据 flags + evidence pack）
Alphapai 下载 md -> /collect 归位 Skills素材包（KA / PJBG，幂等 + 保留最新）
Alphapai业务拆分抓取器 -> 业务拆分历史 factpack
raw 投研材料 -> markdown staging
/brkd 或 /load 或 Alphapai-load -> 候选理解 markdown
/ka -> official 核心假设.md
/comp -> 信息保全后的 yaml1 + forecast
/adj / /frontend-edit / /annual-update -> 已有正式稿的安全修改
/da -> 重资产 DA/CAPEX 专项排程
/pjbg -> 评级报告研究结论（标题+短期+中长期+投资建议）+ 注入 docx 模板出评级报告
```

核心原则：

- **斜杠词优先是 MKA 路由**：`/ka 百润股份`、`/comp 002568`、`/init 新乳业` 不是行情查询、不是交易建议、不是 shell 命令。只有用户明确说“查行情/股价/涨停/分时/盘面/资金”时，才调用股票行情分析能力。
- 核心假设.md 是判断源头；`yaml1`、`forecast_params.yaml`、`Agent/forecast/` 都是派生产物。
- 非标投研材料先 markdown 化，再进入候选理解或 `/ka` 裁决；raw PDF/Word/Excel/网页正文不能直接冒充正式判断。
- **人工筛选门**：markdown 存储区是 cache，不是证据入口；看见 markdown 不等于必须吸收。只有人工放入同权重判断材料、BRKD/LOAD/Alphapai 完成产物、KA 目录顶层 markdown 或 `/init` 事实索引的内容，才进入本轮裁决。
- **入口窄，收纳宽**：人工筛选门只限制读取范围；已进入本轮的材料里，有复盘价值但暂不入模的信息宁可进收纳区/stash，不得因为减负乱扔。
- 同权重判断材料 = `公司判断和最新观点.md` + `重要文件/` 顶层材料 + `最高权重材料-放Agent最应对齐的材料/` 顶层材料；文件夹名保留“最高权重”，但不压过分析师手写 thesis。
- **主导方向**：`/brkd`、`/load` 先忠实整理已被人工放入本轮入口的业务拆分+详细历史+业务数据，预测旋钮是次要 draft 候选；`/ka` 拿 brkd/load/alphapai 完成产物和分析师裁决预测，同时保全已筛选材料中与预测有关的历史。预测是末，业务结构与历史是本。
- **/ka 主拆分优先级**：正式收入主拆分优先采用分析师手写 thesis、同权重材料、BRKD/LOAD/Alphapai/reference 中更深入业务的拆分；官方披露拆分默认用于校验、桥表、父级 headline、副拆分/收纳或兜底，不自动压过分析师拆分。
- **财务费用分流纪律**：生息利息项/利息净额/`interest_expense_rate`/`cash_interest_rate` 交 defaults/引擎，不在 `/ka` 手拍、不在 `/comp` 翻成 knob；`other_fin_exp_abs` 是利润表外生·非利息项，`/ka` 必须作为“非息财务费用”与销售/管理/研发/税金及附加同表同权重检测反馈，默认沿用 `Agent/financial_expense.yaml` / defaults，特殊企业或汇兑、手续费、贴息趋势被明示时裁决。只要 official 费用段点名本项（含沿用 defaults），`/comp` 都显式落 `income.financial_expense.other_fin_exp_abs`；只有旧稿完全未提时才回落 defaults 缺席。
- **展示纪律**：核心假设生成链路（`/brkd`、`/load`、Alphapai-load、`/ka`）聊天阶段先会议 memo，展示优先用一张历史/预测合并主表，把来源、处理去向、待裁决点和收纳/风险放进表格列；完整历史原子、source range、`knobs` 和 reference 回执写文件，不在聊天里碎片化倾倒。
- Alphapai业务拆分抓取器只交业务拆分历史 factpack，不写预测。
- `/brkd`、`/load`、Alphapai-load 只交候选理解，不拍正式判断。
- `/ka` 是唯一裁决器；已有正式稿的局部变化不要回到 `/ka` 做 modify。
- `/comp` 只翻译，不研究、不重判；A 类进计算，B 类进 history/stash/display，歧义进待核清单。
- `/comp` 先按 Semantic IR 盘点：源文块识别 -> IR 分类 -> yaml1 落点 -> audit 六段；IR 是翻译账本，不是新运行时 schema。
- 分不清候选晋升、B 类去向、BS/CF 例外或 `/ka`/`/adj`/`/annual-update` 边界时，先看 `docs/MKA规则导航图.md`，再去读对应真源。
- BS/CF/DCF 驱动默认不在理解层主动预测；重资产 DA/CAPEX 走 `/da`。
- `Skills素材包/KA（ALPHAPAI拆出来的东西放在这里）/` 是 `/ka` 人工入口；顶层所有 `*.md` 都会被 `/ka` 读取。命名为 `核心假设参考{来源}_{YYYYMMDD}.md` 的按候选稿裁决，其他 markdown 按信息指引读取。
- 候选稿必须带 `## 待 /ka 裁决清单`；`/ka` 逐项裁成采纳、收纳、缺口或丢弃后，才可能晋升为 official。

## 分层地图

### 1. 数据地基层

**`/init`**（完整文档见 `docs/init技能文档.md`）

- 读：公司名/代码、TuShare、公告、已有 approved override。
- 写：`raw_tushare` 镜像、`clean_annual`、`clean_quarterly`、`core_metrics_overview.*`、`OfficialBreakdowns`（官方营收拆分）、公告 Markdown、财务费用档案、`defaults.yaml`。
- 停：clean hard check 未闭合（exit 3）、raw 被要求手改、关键历史事实缺口。
- 交给：`/brkd`、`/ka`、`/comp`、`/annual-update`、工作台。

**`/audit`**

- 用途：对 coverage 公司跑确定性财务健康度雷达，识别财务造假、盈余操纵或质量恶化的早期信号。
- 读：`Agent/data.db` 的 `clean_annual` / `clean_quarterly`，本地年报 Markdown、`Agent/recon/` evidence；可选读 TuShare 辅助接口（审计意见、质押、股东增减持、大宗交易、管理层）。
- 写：全局 `audit_runs/{run_id}/flags_matrix.yaml` / `risk_ranking.md` / `run_manifest.json`；单公司 `Agent/audit/flags_latest.json` / `evidence_pack_latest.json`。
- 边界：首版只做 Layer 1 flags + evidence pack，不做 LLM 判决；不改 `raw_tushare`、clean 表、核心假设、yaml1 或 forecast；不构成交易建议。

### 2. 候选理解层

**`/brkd`**

- 用途：把研报/纪要/年报和 clean 历史整理成业务与利润表预讨论。
- 主导方向：忠实记录业务拆分+历史+收纳数据（即便不参与计算）优先于预测。
- 产物：`Agent业务讨论.md`（公司根目录），状态是 draft，所有预测建议待 `/ka` 拍板；可选参考稿落 KA 参考稿区 `核心假设参考brkd_{YYYYMMDD}.md`。
- 边界：不锁最终时间轴，不写正式 `核心假设.md`，不写 yaml1，不跑 DCF。

**`/load`**

- 用途：把唯一 Excel 模型按它自己的 vintage 时间轴装载出来。
- 主导方向：保真装载业务拆分+历史+收纳数据（即便不参与计算）优先于搬运预测。
- 产物：KA 参考稿区 `核心假设参考load_{YYYYMMDD}.md` + `Agent/Load/{load_id}/` 沙箱副本。`/load` 止于此 markdown，不编译 yaml1、不跑 DCF。
- 边界：`model_boundary.*` 是时间轴归口；禁读后验材料；不覆盖正式 `Agent/forecast/`。

**`/webload`**

- 用途：为网页端执行 `/load` 打包 allowed materials、禁读清单、边界和执行说明。
- 产物：`WEBCLAUDE/模型装载部分/`。
- 边界：只 prepare + 打包，不理解模型、不写核心假设、不编译。

**`/webka`**

- 用途：为网页端执行 `/ka` 打包规则与材料（4 份规则 + 同权重判断材料 + BRKD/LOAD + KA 目录顶层 markdown/reference/旧稿 + `defaults.yaml` + `core_metrics_overview` / `financial_expense.yaml` / `OfficialBreakdowns`）。
- 产物：`WEBCLAUDE/webka(Claude帮你统摄核心假设）/` 下 3 份合并 md（`readme first.md` / `必读和素材.md` / `速查参考.md`）。
- 边界：纯打包，本地先跑 `ka_prepare` + 强制 §2/§6b 门禁；不在本地裁决、不落盘核心假设；网页端产出的稿子由用户拷回本地走 `/ka` 落盘（`ka_archive` + 写今日新稿 + `/comp`）。

**Alphapai-load prompt**

- 用途：让 Alphapai 用它自己的金融数据库吐出 KA 参考稿区 `核心假设参考alphapai_{YYYYMMDD}.md`。
- 上游：可先用 `docs/Alphapai/Alphapai业务拆分抓取器.md` 产出 `Alphapai业务拆分事实底稿.md`，专门抓用户指定主拆分、桥表和高价值辅助拆分历史。
- 产物：reference 稿，落 KA 参考稿区，只供后续裁决。
- 边界：不是本地启动器；若有业务拆分 factpack，必须优先承接，不得把用户指定主拆分替换成官方更稳健口径；BS/CF/DCF 线索一律收纳，不能自行判断核心 thesis，不能写进 `knobs`。

**Alphapai业务拆分抓取器**

- 用途：让 Alphapai 专门抓 `/init` 标准财务表里没有的非标准业务拆分历史数值；支持你直接给 `部` / `leaf` 表后按表定向取数。
- 产物：`{公司}-{日期}-Alphapai业务拆分事实底稿.md`，状态 `factpack/reference`。
- 边界：只抓历史，不写预测、不写 `knobs`、不写 `horizon`/`terminal`；用户指定主拆分或定向 leaf 表必须锁定为主轴，官方口径只能作为桥表或 sanity check；最近 5 年完整性优先，早期数据有参考价值才抓。

### 3. 正式裁决层

**`/ka`**

- 用途：把同权重判断材料、BRKD、LOAD、KA 目录顶层 markdown、`/init` 事实和分析师 thesis 收口成正式 `核心假设.md`；到 KA 目录读取所有顶层 `*.md`。
- 主导方向：拿 brkd/load/alphapai 产物和分析师裁决预测，同时忠实收集与预测有关的历史防丢数。
- 必须先做：时间轴四数、接缝总账、骨架门、数值门。
- **主拆分选择门（骨架门内，分析师优先）**：主拆分先看分析师和同权重材料指定的业务拆分，再看 BRKD/LOAD/Alphapai/reference 的业务拆分；官方拆分只在缺少更高层级业务拆分或分析师明确选择官方口径时兜底，其他时候作为校验、桥表、父级 headline 或副拆分/收纳。
- **收入配平门（骨架门内，硬性）**：收入 leaf 集合 base 年求和必须 = `clean_annual.revenue`（容差 1 Mn）。leaf 取自年报分产品表时和 = 主营业务收入 < 营业收入，差额 = 其他业务收入（TuShare `oth_b_income` 常系统性留空，见 `known_tushare_defects.json`），必须显式补"其他业务收入" bridge leaf 桥接，禁止重述既有 leaf 凑数。否则 `/comp` 回测闸拦下出不了 DCF。
- **其他财务费用门（费用数值门内）**：先读 defaults 审计项 `income.financial_expense.other_fin_exp_abs` 和 `Agent/financial_expense.yaml`（若需附注构成）。在费用段把它以“非息财务费用”列在销售费用率、管理费用率、研发费用率、税金及附加率之后同表反馈；前几项是费率，`other_fin_exp_abs` 是 abs_mn 金额。默认沿用机器底座；只有特殊企业口径或材料/分析师明示汇兑、手续费、贴息等非利息项趋势变化时，才裁决覆盖。生息利息项和利息净额仍不进 `/ka` knobs。
- 产物：公司根目录 `{公司}-{YYYYMMDD}-核心假设.md`，状态 official，末尾有 official `knobs`。
- 边界：不做旧稿局部 modify；不产 yaml1；默认只裁利润表 + 业务层盈利模型。

### 4. 编译运行层

**`/comp`**

- 用途：把 official `核心假设.md` 忠实翻译成 `yaml1`，通过 fidelity/audit 后跑 forecast。
- 读：official 核心假设、`defaults.yaml`、`数据格式参考.md`、`yaml1算法模板契约.md`、`knobs块契约.md`、`yaml1前端展示契约.md`。
- 写：`Agent/yaml1_*.yaml`、`Agent/.modelking/forecast_params.yaml`、`Agent/forecast/`。
- 边界：不读 PDF，不改原文判断，不用 defaults 反向否定核心假设；信息保全闸不干净时只留 reference/draft yaml1，不跑 official forecast。
- 固定 audit 回执：A 类覆盖、B 类保全、路径待核、语义待核、主动覆盖回读、Forecast 状态；少一段都不算完整审计。
- 财务费用：利息净额、`interest_expense_rate`、`cash_interest_rate` 不翻；若 official 拍了 `other_fin_exp_abs`，只能落 `income.financial_expense.other_fin_exp_abs`。写到顶层 `financial_expense.*` 会被 cleaner 静默丢弃，属于路径失败。

### 5. 正式稿调整层

**`/adj quick`**

- 用途：已有 official 核心假设下，只拨已有 `knobs` 数值。
- 写：归档旧稿、新核心假设、scoped yaml1 patch、新 forecast。
- 边界：白名单不可加宽；不得改结构、slug、family、horizon、fade、历史、来源、stash。

**`/adj incremental`**

- 用途：读 ADJ 增量材料，讨论系统性影响后修改核心假设源文，再走 `/comp`。
- 边界：新增结构或推翻 thesis 基础时，不硬补丁，转 `/ka` 重建。

**`/frontend-edit`**

- 用途：前端 assumption-preview 已试算并拍板后，把变更定点回写到语义源头。
- 写：核心假设正文、`knobs`、对应 yaml1 值、forecast、公司根目录 `*Model-YYMMDD.xlsx`。
- 边界：不读新材料；映射不到 path 立即停；`terminal.explicit_end` 和 `fade.to_year` 属结构改动。

**`/annual-update`**

- 用途：新年报出来后，把旧核心假设滚到最新实际年。
- 流程：读旧稿总账 -> `/init` 刷数据 -> fetcher 填标准线 -> 滚后时间轴确认 -> 估算/重定未来 -> `/comp`。
- 边界：旧稿只读另存；clean 失败停；拿不到的实际只能估算带旗或待补。

### 6. 专项外挂层

**`/da`**

- 用途：重资产公司专用 DA/CAPEX 排程。
- 事实层：`Agent/recon/da_facts_latest.json`。
- 假设层：`Agent/da_schedule.yaml`。
- 边界：LLM 抽事实和参与商议，Python `da_roll` 计算；`da_schedule.base_year` 必须等于 defaults base period。

### 7. 报告产出层

**`/pjbg`**

- 用途：把 `/ka` 核心假设 + `/comp` forecast + PJBG 素材区事实，合成评级报告「研究结论」四件（标题+短期+中长期+投资建议），并注入 docx 模板产出正式评级报告。
- 读：`公司判断和最新观点.md` + 最新 `*核心假设.md`（A/B 层判断）；`Skills素材包/PJBG评级报告素材区/`（C 层事实）；`Agent/forecast/derived_metrics_annual.csv`（D 层数字）；空白模板 `.claude/skills/pjbg/template/评级报告模板.docx`（可 `PJBG_TEMPLATE_PATH` 覆盖）。
- 写：`内部报告/评级报告/{公司}-{YYYYMMDD}-研究结论.md` + `…评级报告.docx`（R01 标题 + R04 研究员[从前端配置 `MKA_RESEARCHER_NAME` 读] + R06 研究结论 + R08 盈利预测全表，按节标题/标签定位注入，字体随模板、加粗随 .md）。
- 边界：方向/落点只从 A/B、数字只从 D、C 只供事实；R08 盈利预测全表从 forecast CSV 渲染（对齐评级报告模板 R08 体例：ROE 在 EPS 前、(百万) 标注、不带 +/x；年份窗同前端）；投资要点/正文部分模板自带「见附件」不内嵌，其余表头字段（含「是否重大推荐」）/声明不碰；判断缺失/素材空/forecast 缺 → 停下问分析师，不自填。模板可替换（按节标题定位，换模板通常不改代码；`scripts/pjbg_make_template.py` 从填好的报告派生空白模板，自动清洗可变值、保留标签、投资要点/正文写「见附件」）。

## 常见误分流

| 误用 | 正确去处 |
|---|---|
| 想让 `/ka` 直接读原始 Excel | 先 `/load` |
| 想让 `/ka` 读一堆研报纪要 | 先 `/brkd` |
| 想让 Alphapai-load 顺手抓高/中/低端多年拆分 | 先跑 Alphapai业务拆分抓取器，再把 factpack 贴给 Alphapai-load |
| 有正式稿后让 `/ka` 改几个数 | `/adj quick` 或 `/frontend-edit` |
| 新年报已覆盖旧预测起点还继续 `/comp` | `/annual-update` |
| 在 `/brkd` 或 Alphapai-load 里写 capex/折旧逐年预测 | 收纳；需要时走 `/da` |
| 手改 `forecast_params.yaml` 或 `Agent/forecast/` | 回到 `核心假设.md` 或数据源 |
| 把 `yaml1` 当判断源头 | `核心假设.md` 才是判断源头 |

## Agent 执行提示

1. 先判断当前任务属于哪一层。
2. 读取对应 `.claude/skills/{skill}/SKILL.md`。
3. 若该启动器要求动态加载最新版 `skills/*_skill_vN.md`，按版本号最大者加载。
4. 写正式产物前检查门禁：时间轴、source-of-truth、clean 状态、knobs 同源、BS/CF 边界。
5. 汇报时说清楚：读了什么、写了什么、停在哪些门禁前或已经通过哪些门禁。
