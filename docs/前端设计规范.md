# ModelKing 前端设计规范

本文件沉淀 ModelKing 工作台（`app/` React + Vite，`src/workbench.py` FastAPI 壳）在迭代中确立的设计偏好与原则。改前端前先读这份，新增组件须符合这里的格式与禁忌。

> 设计基线：Apple HIG / SF Pro 风格——白底、克制约束、无渐变、无装饰图标、数字严谨。但**实际色值以 `app/src/styles.css` 的 CSS 变量为准**（CLAUDE.md 里写的 `#0071E3/#FF3B30` 与实际 CSS 不符，不要照抄）。

---

## 1. 设计总原则

1. **数据是主角，装饰让位**。工作台是买方研究工具，读者要精确读数。任何"好看但干扰读数"的装饰（大字体数字、彩色底块、渐变、图标）一律不要。紧凑 > 巨大。
2. **框线区分 > 颜色底区分**。当需要分隔两类东西（历史/预测、行组、列组），优先用明确的边框线，不要用灰底/白底色块——色块底在密集表格里会糊成一片、且与行级高亮冲突。
3. **通用性高于一切**（继承项目第一原则）。任何渲染逻辑按**数据结构类型**分派，不按公司名/中文名特判。换任何公司只要它如实声明结构就能跑。看到自己在为某家公司 special-case 就停，退一步一般化。
4. **结构化优先于原文**。能用表格/分组/折叠表达的，不要堆 raw 文本；但 raw 原文（YAML、字段名）要保留一个折叠入口给 power user。
5. **零丢失**。展示层不得静默丢弃数据。结构分派必须有兜底分支（`kv` 类型），任何未知形状落到兜底，绝不丢。
6. **会计科目与顺序的唯一真源是 `src/field_registry.yaml`**。三表的字段分类、会计序、中文标签全部声明在此一处;`clean.py` 校验、`workbench._statement_rows` 渲染、`docs/数据格式参考.md`(由 `scripts/gen_field_reference.py` 派生)三处同源。标签/顺序错了**改 registry**,不再有展示层覆盖补丁(`LABEL_OVERRIDE`/`STATEMENT_META.field_order` 已删)。详见 `docs/会计系统.md`。

## 2. 信息架构（Tab）

固定五个顶级 tab，顺序不可调：

| 顺序 | Tab | 定位 | 内容权威 |
|---|---|---|---|
| 1 | Overview | 公司概览 + 运行契约 + 一键重算 | — |
| 2 | 核心假设展示 | **假设工作台**：收入拆分 + 利润路径 + 所有可编辑 knobs + 参考项 + 原始 YAML；只读解释 yaml1，edit 模式做沙盘预览并生成 `/ka` prompt | yaml1 / editable_assumptions |
| 3 | 完整三表 | 历史 + 预测三张表，独立 tab | full_*.csv |
| 4 | DCF | **结果**：指标卡 + 估值桥 + FCFF 构建 + 敏感性 | dcf_summary / dcf_detail |
| 5 | Materials | 源文件索引 | — |

原则：
- **不要单独开 "Core Assumption" tab**。核心假设的展示权威统一在「核心假设展示」tab。
- **收入拆分 / 关键假设在同一个「核心假设展示」tab 内合并展示**，不要再拆成两个内部 tab。DCF tab 不重复展示拆分假设，只展示结果。
- **完整三表与 DCF 并列**，各占一个 tab；估值桥放在 DCF tab 内直接常驻（不再做成三表的第 4 个子页签）。

## 3. 视觉语言

### 3.1 色系（`styles.css` `:root`，唯一允许的色源）

| 变量 | 值 | 用途 |
|---|---|---|
| `--bg` / `--surface` | `#ffffff` | 底色 / 卡片底 |
| `--text` | `#1a1a1a` | 主文字 |
| `--secondary` | `#5f6368` | 次级文字（表头 eyebrow、说明） |
| `--tertiary` | `#8a8a8a` | 最弱文字（计数、辅助） |
| `--border` | `#e5e5e5` | 普通分隔线 |
| `--border-strong` | `#d8d8d8` | 强分隔线 |
| `--blue` | `#003d7a` | **唯一交互色**（链接、active、主动覆盖标记、分界线、关键行标签） |
| `--red` | `#b42318` | 负数、警告、风险 |
| `--mono` | SF Mono / Menlo / Consolas | 所有数字 |

铁律：
- **`--blue` 只用于交互/强调**（按钮、active 态、主动覆盖蓝点、历史|预测分界线、关键行科目名）。不要拿蓝色当普通文字色。
- **负数一律 `--red`**，无例外。
- **数字一律 `--mono` + 右对齐**。
- **禁止渐变、禁止装饰图标、禁止阴影堆叠**。卡片用 1px `--border` + 圆角，不用浮夸阴影。
- 不引入新色系。需要新状态色时，先看能否复用 `--blue`/`--red`/`--secondary`。

### 3.2 字体与层级

- 系统字体栈：`-apple-system, "SF Pro Display", BlinkMacSystemFont, "Helvetica Neue"`。
- 表头 eyebrow：11px all-caps，`--secondary`，letter-spacing 0.05em。
- h1 32px / h2 22px / h3 适当递减，`letter-spacing -0.3~-0.5px`。
- 表格内数字默认 13px 左右；**输入框/卡片大数字不要超过 20px**（敏感性数值曾用 40px 挤占主表，已禁）。

### 3.3 间距与卡片

- 卡片 padding 14–24px，section 之间 gap 24px。
- 表格 cell padding 紧凑（6–10px），行高让数字密度可读但不拥挤。

## 4. 表格设计（最高风险区，最严格）

完整三表是工作台核心，规则最细：

### 4.1 行序 = 严格会计准则序

- 后端 `field_registry.statement_meta_for_table(table).field_order` 给出会计序,`_statement_rows` 直接迭代该列表(小计内联在其会计位),**全部 325 字段都有显式位,无字母序回退**。
- **绝不按字段名字母序展示**。`数据格式参考.md` 现已由 registry 派生、按会计序排列(不再是字母序);但渲染权威仍是 registry,不是文档。
- IS 顺序:营业收入→营业总收入→减:营业成本→税金及附加→销售→管理→研发→财务→**营业总成本**→资产减值→信用减值→其他收益→投资收益→公允价值→资产处置→汇兑→营业利润→营业外收支→利润总额→所得税→净利润→归母→少数。(资产减值/信用减值为 `operating_adjustment`,在营业总成本**之后**、营业利润之前,为配平建模,非列报 bug。)
- BS:流动资产→非流动资产→流动负债→非流动负债→权益,各类内部按准则序。
- 小计/合计行(subtotal/total)在 `field_order` 里内联在其会计位(`category: subtotal`,`role: total` 标三个 BS 总计),加粗高亮;不再用 `subtotal_after` 单独挂载。

### 4.2 标签一致性

- 减项统一带 `减:` 前缀,**直接写在 `field_registry.yaml` 的 `label` 里**(如 `rd_exp`→`减:研发费用`、`credit_impa_loss`→`减:信用减值损失`)。不再有 `LABEL_OVERRIDE` 展示层补丁——改标签就改 registry。
- 损失类科目(资产减值/信用减值)以负号填列,正数=转回;`减:` 前缀让读者一眼知道是减项。

### 4.3 年份表头行

- **纯黑底（`#000000`）白字**，加粗（600），14px。
- 预测年份带 **E 后缀**（`2025E`），历史年份裸数字（`2024`）。
- 历史/预测的区分**只靠 E 后缀 + 分界线**，表头本身不染色区分。

### 4.4 历史|预测分界

- **用框线，不用底色**。正文格子统一白底，去掉"历史灰底/预测白底"的旧设计。
- 分界线：一条 **2px `--blue` 竖线**，贯穿表头与正文，**只出现在第一列预测期**（CSS `.history-year + .forecast-year`），不污染每列预测期。
- 行级高亮（小计行 `#f7f7fa`、关键行 `#f5f5f7` + `--blue` 科目名）保留，但历史与预测在该行内同色——行高亮是按行语义，不是按历史/预测。

### 4.5 列

- **不展示"字段"列**（raw field 名是给开发/debug 的，不给读者）。
- 关键行（IS 的 revenue/oper_cost/operate_profit/total_profit/n_income/n_income_attr_p）加粗 + 浅底 + 科目名 `--blue`。
- 负数 `--red`，空值留白。

### 4.6 范围开关（完整三表特有）

表头右侧分段开关，两档：

| 模式 | 历史 | 预测 | 默认 |
|---|---|---|---|
| 近5年 + 预测 | base 期及之前最近 **5** 年 | **仅显式预测期**（到 `terminal.explicit_end`），fade 期隐藏 | ✅ |
| 完整历史 | 全部历史 | 全部预测（含 fade） | |

- "近5年"= `slice(-5)`，不是 3。
- **显式预测期 vs fade 期**：分界 = `yaml1_assumptions_view.terminal.explicit_end`。显式 = base+1…explicit_end；fade = explicit_end+1…to_year。近5年模式只保留显式预测期，丢掉 fade。
- 无 yaml1（无 `terminal.explicit_end`）→ fallback 展示全部预测期，不崩。
- 另有 `Show zero rows` 复选框，默认关（零值行隐藏，只留小计/关键行）。

## 5. 通用结构渲染（stash 参考项）

「核心假设展示」tab 的参考项（stash）是 per-company 差异最大的部分，必须通用：

- **按 JSON 结构类型分派**，6 类 + 兜底，绝不按公司名/中文名：

| 结构形状 | type | 渲染 | 默认 |
|---|---|---|---|
| 字符串列表 | `list` | 要点列表 | 折叠 |
| `{item:{年:number}}` 年键为纯数字 | `series_table` | 表(行=item,列=年) | 展开 |
| `{item:{非年键:number}}` | `attr_table` | 表(行=item,列=属性) | 展开 |
| 值为 string 的 dict | `text_dict` | 定义列表(名:原文) | 折叠 |
| 值为 number 的 dict | `scalar_table` | 小表(key:value) | 折叠 |
| 其它 | `kv` | raw key-value 兜底 | 折叠 |

- 数据表类（series_table/attr_table）默认展开，文本类默认折叠——读者先看数字，需要时再展开定性。
- 每块带 note/caveat/unit，`⚠` 前缀标 caveat。
- 嵌套子块走 `extras[]` 递归（如财务费用历史 = 序列表 + 2024 锚小表）。
- 通用性自检：renderer 里不能出现任何公司名/业务线名特判。

### 5.1 stash 标签中文化（展示层，复用 FIELD_LABELS）

stash series 的原始 key 常是 TuShare 字段码（`assets_impair_loss`/`sell_exp`）或 knob 码（`gpm`/`effective_tax_rate`），**绝不直接把英文码当行/列标签给读者**。后端 `_humanize_label(token)` 统一解析（workbench.py，纯展示层，不碰 `数据格式参考.md` 契约）：

1. token ∈ `FIELD_LABELS`（= `field_registry.LABELS`,三表 325 字段合并,含"减:"前缀）→ 中文（`sell_exp`→减:销售费用、`assets_impair_loss`→减:资产减值损失）。
2. token ∈ `STASH_CODE_LABELS`（非 TuShare 码补表：`gpm`→整体毛利率、`effective_tax_rate`→有效税率、`minority_ratio`→少数股东损益占比、`fin_exp_total`→财务费用合计、`revenue_yoy`→营收增速 …）→ 中文。
3. `YYYY_<code>` → `{年} {中文}`（`2024_gpm`→`2024 整体毛利率`、`2024_ton_cost`→`2024 吨成本`）。
4. 兜底原样返回（零丢失，未知码不崩）。

`_stash_block` 给每个 series_table/attr_table 挂 `col_labels: {rawKey: 中文}`，行 label 走 `_humanize_label`。terminal `fade_paths`/`hold_paths`（knob 路径）走 `_humanize_path` 取末段中文化（`income.gpm`→整体毛利率）。**分析师手写的 note 正文 / 溯源附注标题（如 `gpm_整体手拍`）不强制改写**——那是分析师的速记，强行中文化会损义；只改"字段码当表格标签"这一类。

### 5.2 一区一表一轴（核心假设展示的合并结构）

核心假设展示页**不再拆「收入拆分 / 关键假设」内部 tab**。收入拆分、利润路径、收入 drivers、利润表 knobs 必须合并在同一张主表里。编辑态采用季度展示页同款交互：能映射到 `editable_assumptions` 的数字单元格直接呈现为不同外观的可点按钮，点击后就在原格内变成 `input + OK`，不跳转、不抽屉、不把用户丢进另一张 knobs 表。没有数据的格子留空（空字符串），不另起表、不另起轴。

| 区域 | 一张统一表的内容 | 时间轴 |
|---|---|---|
| ① 收入拆分 + 关键假设 | 总收入(营业收入+同比) 下方必须补归母净利润、净利润、净利率、净利润同比；再接主拆分·业务线、副拆分、收入拆分 drivers、利润表 knobs。edit 模式下所有可映射单元格都可原地输入 | 历史+预测并集（如 2014–2028E） |
| ② 参考项 | 历史观测（5 块）+ 核对项（2 块）合并进一表；非年份块（分线 attr / 口径 / 溯源 / 定性）折叠在表下 | 历史并集（如 2014–2024） |

实现：`UnifiedYearTable`（`table-layout: fixed`，首列项目名固定宽，年份列等宽）+ 组标题行（`.group-header-row`，整行贯通浅底）。构建器按数据源装 `AxisGroup[]`：

- `buildRevenueGroups`：总收入历史从完整 IS 表 `revenue` 补全（`fullStatementSheets`），base+预测来自 `revenueView`；edit preview 存在时，预测期营业收入与利润行优先读 `preview.statement_sheets`；副拆分 = stash name 含「拆分」的 block（约定，上移紧跟主拆分）。
- `buildEditableAxisGroups`：把后端 `editable_assumptions` 中尚未出现在 assumptions view 的 knobs（尤其收入拆分 drivers / leaf margin / 其他覆盖）转成同一张主表里的行。比率以百分号展示，输入时再还原成小数。旧的全部 knobs 清单只能作为折叠高级清单，不作为主编辑入口。
- `buildReferenceGroups`：仅并入值键全为年份的 `series_table`（`blockIsYearSeries`）；attr_table（如分线 `2024_gpm`）与非年份块退回 `refRest` 折叠。

**行格式**（`AxisRow.format`，避免比率被当整数截断成 0）：`int`=整数金额 · `num2`=2 位小数（参考项通用，单位在组头）· `decimal`=小数比率×100+%（旋钮）· `signedDecimal`=带符号同比% · `volume`=1 位小数（万吨）。**单位中文化**：`humanizeUnit` 把 `million_cny`→百万元、`100mn_cny(存疑)`→亿元·存疑、`pct`→%。

通用性：三区构建器只按数据结构/命名约定分派，零公司特判；任何公司只要 yaml1 结构如实就自动装入对应区，未知形状落 `refRest` 折叠零丢失。

## 6. 假设旋钮展示

- 旋钮来源统一为 `GET /api/companies/{id}` 返回的 `editable_assumptions`，后端负责从 yaml1 自动枚举，前端不得按公司名、业务线名或路径白名单写死。
- 必须兼容所有 knobs 更改：标准路径 knobs、收入拆分 factor projection、leaf margin、terminal 参数都用同一种 cell patch 协议（JSON pointer + year/value）提交。
- 每 knob 一行，列 = 预测年或字段。比率型显示百分号，绝对值型显示数字；edit 模式下输入框不得撑高表格、不得改变年份轴宽度。
- **主动覆盖/查证用蓝点**（`--blue` 圆点 + title）标记，src 含「主动覆盖/查证/弃模型/normalized」即标记。缺失旋钮 = 落 yaml2 平推，不显示。
- edit 模式只调用 `assumption-preview` 做内存重算，预览结果覆盖收入拆分上方的营业收入/净利润路径和 DCF 摘要；不直接写 `核心假设.md` 或 `yaml1`。主编辑入口必须是表内单元格，不要把用户丢进单独的 knobs 大表。
- 用户满意后只生成 `/ka` prompt，让核心假设生成修改器更新 `核心假设.md`，再由 `/comp` 重新编译 yaml1。
- 三段式参数作为可编辑 terminal 行展示：显式期末 / 衰减至 / 衰减方式 / 永续增速 / terminal CAPEX÷D&A。溯源附注从 stash 的「溯源附注」text_dict 取，折叠。

## 7. DCF 展示

DCF tab = 结果，顺序：

1. **指标卡**（4 张）：每股 / EV / 权益价值 / 终值 PV。
2. **估值桥**（直接常驻，不做子页签）：
   - `PV(FCFF) + 终值PV = EV − 净负债 = 权益价值 ÷ 总股本 = 每股`
   - 终值占比 ≥ 60% → 红色警示「估值高度依赖永续」。
   - FCFF 逐年构建表 = NOPAT + DA − Capex − ΔNWC，带折现因子与 PV。
3. **敏感性面板**：WACC / Terminal growth / Terminal CAPEX÷DA 三卡，**紧凑**（数值 ≤ 20px），拖动实时重算。不挤占估值桥。

---

## 8. 交互模式

- **折叠分组**用原生 `<details>`，数据表默认 open、文本类默认 closed。
- **分段开关**（range toggle）：两按钮一组，active 白底 `--blue` 字 + inset border，非 active `--secondary`。
- **复选框**（zero rows）：`accent-color: var(--blue)`。
- 加载/运行态用 `.activity` 文字提示，不要全屏遮罩。
- 错误用 `.error-banner`（`--red` 边），不弹 modal。

---

## 9. 禁忌清单

- ❌ 字段名字母序当会计序展示。
- ❌ 历史/预测用灰底/白底区分（用框线）。
- ❌ 年份表头灰底（用纯黑底白字）。
- ❌ 预测年份不带 E。
- ❌ 展示 raw "字段" 列给读者。
- ❌ 敏感性数值 40px 巨大字体（≤ 20px）。
- ❌ 估值桥做成隐藏子页签（直接常驻 DCF tab）。
- ❌ 单独开 Core Assumption tab。
- ❌ 在「核心假设展示」tab 里再拆「收入拆分 / 关键假设」两个内部 tab。
- ❌ 总收入区只展示营业收入和同比，不展示净利润/归母净利润/净利率路径。
- ❌ 前端直接写回 `核心假设.md` 或 `yaml1*.yaml`（只能沙盘预览 + 生成 `/ka` prompt）。
- ❌ 只支持固定几类 knobs，漏掉收入 drivers、leaf margin 或 terminal 参数。
- ❌ stash 渲染按公司名特判 / 丢块。
